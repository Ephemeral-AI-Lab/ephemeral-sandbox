# Worker Tooling Discipline

These rules are mandatory for all coordination workers operating in a sandbox environment. They ensure reliable, verifiable work output.

---

## Rules

**RULE 0 ŌĆö Shared-workspace hygiene first:** At the start of a task in a shared sandbox, do a quick hygiene sweep of the files and directories you are likely to touch. If you see `.orig` or `.rej` artifacts near your target files, or if the first targeted test run fails during collection with `SyntaxError` or `IndentationError`, stop broader diagnosis and repair that malformed file first. Do not keep investigating behavioral failures on top of a syntactically broken workspace. If the blocking malformed file is clearly outside your owned scope, or lives in a benchmark/test/auxiliary path that is not part of your owned production surface, report it immediately as a workspace blocker instead of silently working around it. Do not spend the attempt editing unrelated contaminated files just to keep a broader regression command running. If `edit_file` is rejected because other syntax errors elsewhere in the same file make a partial edit unsafe, stop repeating the same one-line fix. Switch to `codeact` or a single batched repair that fixes all reported syntax blockers in that file together, then return to normal targeted edits.

**RULE 1 ŌĆö Read before editing:** Before modifying any file, call `read_file` or `read_file_lines` on the target path. Never blindly overwrite or append to a file you have not read in this session.

**RULE 1A ŌĆö Pass concrete path strings, not inferred placeholders:** Every `read_file`, `read_file_lines`, `edit_file`, `read_symbol`, `lsp_*`, or pytest target must use a concrete file path string. Do not call file tools with an omitted `file_path`, `filepath`, a structured object, a remembered placeholder, or a guessed changelog path. If you do not yet know the exact path, discover it first with `list_files`, `read_file_lines`, `read_symbol`, or a narrow shell listing, then retry the file tool with that exact path string. If a tool reports `file_path must be a string` or `filepath must be a string`, do not immediately retry the same tool with another missing argument payload; switch to path discovery first and only retry once you have a verified concrete path string.

**RULE 2 ŌĆö Write through the runtime tools:** Create and edit files ONLY through `create_file` or `edit_file`. NEVER use `run_shell_command` with echo/cat/sed to write files. Any scratch file you create must live under the workspace/repo path you were given. Do NOT call `create_file`, `edit_file`, or `delete_file` on `/tmp`, the home directory, or any other path outside the workspace. For throwaway experiments that do not need to persist in the repo, prefer `run_code` or a read-only `run_shell_command`.

**RULE 2A ŌĆö Do not replay patches blindly:** Use `run_shell_command` to inspect commit metadata or diff hunks, but translate those hunks into explicit `edit_file` changes after reading the target file. NEVER use shell patching flows (`patch`, `git apply`, heredoc diffs, or similar) that can leave partial applications, `.orig` files, or `.rej` files in the workspace.

**RULE 2B ŌĆö Reject partial patch application:** If you ever observe `.orig` or `.rej` artifacts, or if an edit tool reports "search text not found" after you were following an upstream diff, stop treating that diff as directly applicable. Re-read the live file, repair any malformed code immediately, remove the artifact files, and continue with a fresh targeted edit. Do not leave partial patch artifacts behind for sibling workers.

**RULE 2C ŌĆö `delete_file` takes one concrete path string per call:** When removing `.orig`, `.rej`, or other owned artifacts, call `delete_file` once per exact file path string. Do not pass arrays, structured payloads, shell globs, or remembered placeholders into `delete_file`. If you need to remove multiple files, issue multiple `delete_file` calls with explicit string paths.

**RULE 2D ŌĆö `create_file` is for deliberate new repo files with real contents:** Do not call `create_file` with an empty body, placeholder body, or as a scratch pad while you are still thinking through a fix. If you need a throwaway experiment, use `run_code` or a read-only `run_shell_command` instead of creating a workspace file. If `create_file` is rejected because the content is empty or malformed, stop retrying the same create operation. First decide whether the file should exist at all, then retry once with concrete contents and a repo-owned destination path.

**RULE 3 ŌĆö Verify after every create/edit:** After creating or editing a `.py`, `.ts`, `.tsx`, `.js`, or `.jsx` file, you MUST (a) call `read_symbol` on the primary function/class to confirm it, and (b) call `lsp_diagnostics(filepath='<exact verified path>')` on that same file to catch errors. Never omit `filepath`, pass `None`, pass an empty string, or use a guessed placeholder. Do NOT skip verification or substitute `read_file`.

**RULE 3AA ŌĆö Remaining owned focus-test failures are blockers:** If your direct verification still shows failures in a changed test file, a `ci_plan.touches_paths` test file, or a named FAIL_TO_PASS focus file in your owned scope, do NOT call that result "pre-existing" by default. Treat it as unresolved until you trace whether the mismatch belongs in the test expectation, a compat/helper guard, or the owned production path. Only report an external blocker after you verify the failing contract is intentionally unsupported by the provided environment and outside your owned manifest/scope.

**RULE 3AB ŌĆö Declared out-of-scope edits invalidate the attempt:** If you changed or intend to report a tracked file outside your owned scope or `ci_plan.touches_paths`, stop and treat that as an invalid attempt, not a successful completion. Do not claim victory after editing an unrelated file and then describing the remaining failures there as "pre-existing" or "upstream." Either revert back to your owned scope before finishing, or explicitly report that you crossed scope and need replanning.

**RULE 3ABB ŌĆö Diagnose across scope if needed, but do not fix across scope:** You may read adjacent files to confirm whether a failure is rooted outside your owned path cluster. If targeted diagnosis shows the minimal production fix lives outside your owned path cluster or declared `ci_plan.touches_paths`, stop and report that the task is mis-scoped and needs replanning. Do not turn a narrow leaf into an implicit cross-lane patch by editing neighboring production files just because the failing test led you there.

**RULE 3ABA ŌĆö Protected non-exported paths are not valid fix surface:** If the workspace contract marks a path as protected, non-exported, auxiliary, or otherwise excluded from the final patch, do not edit that path just to make the shared sandbox green. Restoring such a path to its known baseline hash is valid hygiene; counting changes to it as task progress is not. If a protected path is dirty or malformed and blocks your owned production work, report it as a workspace blocker or restore it to baseline, but do not finish the task while that protected path remains drifted.

**RULE 3AC ŌĆö Production-scoped retries need a production-scoped fix:** If your owned scope includes non-doc production paths, do not call the task complete just because the named FAIL_TO_PASS or retry-focus tests are currently green in the shared sandbox. If you made no owned production changes, or your edits are limited to tests, docs, or auxiliary files, treat that as workspace cleanup, a masked state, or evidence that the task was mis-scoped. Continue until the fix is reflected in owned production paths, or report that the task needs replanning. **Export attribution:** The coordination export only includes file paths declared in your task's completed artifact. If you report zero changed files, the export will not preserve any sandbox changes in your assigned scope ŌĆö even if the sandbox state looks correct. Passing tests in the shared sandbox do not guarantee your changes survive export.

**RULE 3AD ŌĆö Declare only owned edits in shared-sandbox summaries:** In final summaries and any explicit `Modified:` or `Created:` file lists, name only files you directly edited or intentionally own. Do not run or paraphrase a workspace-wide `git diff`, `git status`, or shared-sandbox change list as your own work. If you must mention sibling activity, label it as sibling context rather than as one of your modified files.

**RULE 3A ŌĆö Syntax before tests:** Before running pytest against a Python test file or module you touched, make sure `lsp_diagnostics` is clean for every modified Python file in your scope. If diagnostics are not clean, fix the file first. Do not use failing collection errors as your primary syntax checker. If a collection error reveals malformed Python in a related file within your scope, repair that syntax first and only then continue with behavioral debugging.

**RULE 3B ŌĆö Test files need direct verification:** If you edit `test_*.py` or `conftest.py`, run pytest directly against the changed file before you finish. Do not rely only on neighboring suites, hand-picked nodes, or a broader repo command that omits the changed test file. If your task description or runtime context names dominant focus files, your final verification must include those named files when they are in your owned scope.

**RULE 3C ŌĆö Direct test-file verification comes before broad suites:** If you changed one or more `test_*.py` files, your first verification after syntax checks must target those exact files. Do not postpone those direct file runs until after broad package suites, doctests, or unrelated regression batches. Use the broad suites only after the changed test files themselves are green, so validation failures are caught at the source instead of after long unrelated test runs.

**RULE 3D ŌĆö Scope skip/xfail changes to the exact failing contract:** Before adding or widening a `skip`, `xfail`, or compatibility guard in tests, reproduce the failure on the exact pytest node ID or smallest parametrized slice that fails. If one parameterization passes and another fails, encode the guard at the `pytest.param(...)` or equivalent narrow case instead of broad-marking the whole test. Re-run both the failing slice and at least one nearby passing slice after the change so you do not convert real passes into `XPASS(strict)` or silently hide unrelated regressions.

**RULE 4 ŌĆö Use LSP on verified paths and real symbols:** Before calling `lsp_goto_definition`, `lsp_hover`, or `lsp_diagnostics` on a new path, confirm the path exists with `list_files`, `read_symbol`, `read_file_lines`, or `read_file`. Call diagnostics as `lsp_diagnostics(filepath='<exact verified path>')`; never call it with `None`, an empty string, or a remembered placeholder. Do NOT invent filenames from changelog or test wording. Run `lsp_goto_definition` at the actual symbol/import site, not line 1 column 1 or an arbitrary file header. If an LSP lookup returns no result on an unverified guess, stop and rediscover the path before retrying.

**RULE 5 ŌĆö Use LSP for cross-file work:** When introducing or changing imports, schemas, or shared contracts, call `lsp_goto_definition` or `lsp_hover` to verify the referenced symbols exist. Call `lsp_find_references` before renaming.

**RULE 6 ŌĆö Prefer the narrowest runtime tool:** Use `run_code` for short Python snippets and `pytest.main(...)` verification so execution does not depend on shell startup/cwd state. Use `run_shell_command` for installs, non-Python CLIs, server commands, or workflows that explicitly need shell semantics. Use `list_files` for directory discovery.

**RULE 6A ŌĆö Discovery must narrow, not loop:** Do not spam `list_files()` on the same directory when the first output already exposed the file tree you need. After one broad listing, switch to a targeted read, symbol lookup, or narrow shell command on the owned path cluster. Repeated root listings without narrowing are wasted tool budget.

**RULE 6AA ŌĆö Inspect tracked files with file/LSP tools, not ad-hoc shell interpreters:** For repository files under your task scope, use `read_file`, `read_file_lines`, `read_symbol`, `grep_search`, and `lsp_diagnostics` for inspection. Do NOT use `run_shell_command` with `python -c`, shell `for` loops, `awk`, `sed`, `perl`, `grep | xargs`, or `py_compile` to scan tracked source files. In coordination sandboxes these shell forms may be blocked or misclassified as unsafe mutation. If you need a syntax check on a Python file, use `lsp_diagnostics`; if you need to find repeated bad fragments, use `grep_search`; if you need to inspect multiple suspicious regions, issue multiple targeted file-tool calls instead of one broad shell script.

**RULE 6AB ŌĆö Stop retry loops after tool throttling or repeated tool rejection:** If you hit tool-call limits, repeated empty-create errors, repeated mutation-blocked shell errors, or the same rejected edit path more than twice in a row, stop retrying that loop. Summarize what you already learned, narrow to one decisive verification step, or report that the current leaf is stale or mis-scoped instead of thrashing the same tool family. Do not keep alternating between speculative `edit_file`, `create_file`, `codeact`, and shell attempts without new evidence.

**RULE 6AC ŌĆö Use sibling context to prevent mirrored duplicate edits:** When sibling summaries, recent sibling changes, or shared-sandbox context already identify the concrete fix file or changed line cluster, treat that as the current fix surface. Do not invent a second "similar" edit in another file just because the high-level symptom sounds related. Before editing a second file, reproduce a distinct failing behavior anchored to that file or identify a separate call site that still reads stale state after the sibling fix. If you cannot show a distinct remaining failure, stop and report that the sibling fix likely already covered the issue. **This rule prevents duplicate edits to OTHER files, not edits to YOUR owned files.** If your task assigns you to modify specific production files, you must still make those changes even if a sibling touched nearby code. Sibling deference applies to scope expansion, not scope abdication.

**RULE 6B ŌĆö Do not mutate the environment to compensate for a wrong interpreter or missing activation:** In coordination sandboxes, treat the provided environment as part of the task contract. Do NOT run `pip install`, `conda install`, `uv add`, `poetry add`, or similar environment-mutation commands unless the task explicitly requires a manifest update in your owned scope. If an import fails, first verify that you are using the intended interpreter/environment prefix for the benchmark or workspace. Retry the same read-only check inside the correct activated environment before concluding a dependency is missing. If the import still fails and you do not own the dependency manifest, report an environment blocker instead of trying ad-hoc installs.

**RULE 7 ŌĆö Keep dependency manifests honest:** If your task introduces a new third-party import, package, CLI, or runtime requirement, update the owning dependency manifest and lockfile in the same task before verification. Do not rely on ad-hoc installs alone. Examples: if you add `EmailStr`, declare `email-validator`; if you add a frontend library, update `package.json` and the lockfile.

**RULE 8 ŌĆö Python commands:** When the workspace contract provides an environment activation prefix (for example conda), prefer `python` after activating that environment so the interpreter matches installed packages. Otherwise use `python3` for ad-hoc Python commands in the sandbox.

**RULE 8A ŌĆö Stop after representative verification:** Once the named retry target or owned focus test is green, and you have one broader representative regression slice for the same change cluster also green, stop and summarize unless a new mismatch appears. That is the verification completion point. Do NOT keep re-running the same already-green retry node, do NOT stack multiple overlapping broad suites just for extra reassurance, and do NOT try to manually cover every file or changelog bullet in your scope after that point. After the retry target is green, you may issue at most one follow-up verification command. That follow-up command must stay narrow: prefer the closest nearby parametrized family, helper-specific test group, or one focused file-level slice. Do NOT expand that single follow-up into a whole large module, package, or cross-feature sweep when a smaller representative subset exists. Choose the smallest broader command that still exercises the same fix cluster and is likely to finish quickly. Do not queue multiple broader pytest commands in the same response, and do not parallelize them. A broader follow-up that reports `0 selected`, `0 collected`, `no tests ran`, or `ERROR: not found` is not valid representative verification; correct the selector once or report the blocker, but do not treat that command as success. If the first broader follow-up instead fails on an unrelated contaminated file outside your owned scope, stop and report the workspace blocker rather than editing that unrelated file. If a long successful pytest command is truncated by the tool output cap, rely on the successful exit status and visible tail unless the tail shows a real failure.

**RULE 8B ŌĆö Fix verification must match the described scenario, not a nearby proxy:** Before declaring a production fix complete, run one minimal reproduction or direct test that matches the actual task contract: the named API/method, the relevant model or input shape, and the expected observable output or failure mode. Do not substitute a nearby inferred case just because it exercises the same file or a similar code path. If your reproduction omits the key wrapper, field shape, mode, or output assertion named by the task, treat it as incomplete evidence. If a later direct test on the described scenario still fails or produces a materially different shape, do not keep the earlier fix marked as complete; treat that as unresolved verification and report that the owned fix is still incomplete or needs replanning.

---

## Completion Check

Once your owned code or tests satisfy the scoped acceptance check, stop expanding the investigation unless you have a concrete unresolved discrepancy. Do not keep spending tool budget on extra confirmation after your modified files are clean, your direct pytest target is green, one representative broader regression slice is green, and no new blocker has appeared. At that point, additional overlapping broad suites are noncompliant unless a fresh failure or conflicting signal appears. After the retry target is green, only one broader follow-up verification command is allowed.

Your task is NOT complete until you list which symbols you verified with `read_symbol`, which files you checked with `lsp_diagnostics`, which changed test files you exercised directly with pytest, confirm that you did not leave `.orig` or `.rej` artifact files in your scope, and state whether you encountered any environment blocker instead of attempting ad-hoc package installation.
