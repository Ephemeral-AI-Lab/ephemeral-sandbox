### Justify naming convention: `host-compatible` vs `isolated`

**Commit:** `10b70de64` — Rename WorkspaceProfile to NetworkProfile and HostCompatible to Shared

**Verdict:** The two profiles differ on exactly one axis — the network boundary.
Both keep a private overlay plus mount/pid/user namespace isolation; only the net
namespace differs. The old names were wrong on two counts:

- `WorkspaceProfile` / `HostCompatible` overclaimed a general "isolation" axis and
  read as "this is the host / localhost", when host-compatible is actually
  isolated in every dimension except network.
- The variants were not even lexically parallel: `host_compatible` framed on
  network sharing, `isolated` framed on general isolation.

**Resolution (shipped in the commit above):**

- Renamed the type `WorkspaceProfile` → `NetworkProfile`, scoping the whole
  selector to the network axis. This fixes the overclaim for free —
  `NetworkProfile::Isolated` now unambiguously means network-isolated.
- Renamed the variant `HostCompatible` → `Shared` (paired with `Isolated`) and
  changed the wire/CLI value `host_compatible` → `shared`, so AI/CLI readers stop
  inferring "the host". `Isolated` unchanged.
- Reworded the `create_workspace_session` / `exec_command` descriptions to state
  the workspace shares the host network namespace while staying isolated in
  mount/pid/user — explicitly "it is not the host".
- Confirmed network (profile) is orthogonal to lifecycle (one-shot vs session);
  the one-shot path simply pins `Shared`. Did not leak one-shot/session into the
  profile vocabulary.
- Removed the transitional `host_compatible` parse alias as unused legacy.

**Verification:** `cargo build` (no warnings), `cargo clippy --all-targets`, and
`cargo test` all clean for the changed crates. Net diff 120 +/127 −.
