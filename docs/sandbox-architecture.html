EphemeralOS Sandbox Architecture

Append-only layer storage, optimistic concurrency control, overlay-isolated command execution, and the daemon/RPC fabric that ties them together. Generated from backend/src/sandbox/.

1. System Overview §

EphemeralOS exists to let many AI coding agents work on the same project at once without corrupting each other’s state. The design has four load-bearing pieces, layered bottom-up:

  1. LayerStack — an append-only, content-addressed log of immutable filesystem deltas. State versions are integers; layers never mutate after publication.
  2. OCC — Optimistic Concurrency Control. Agents prepare changes lock-free, then collide-or-commit at apply time against the live manifest.
  3. Command Execution — leases a snapshot from LayerStack, mounts it via overlayfs, runs a shell command, captures changes, submits them through OCC.
  4. Provider / Daemon / API — the runtime fabric: provider (Docker / Daytona) creates the sandbox, daemon runs inside it, API is the orchestrator-side facade.

2. LayerStack §

2.1 Mental model §

The LayerStack is an append-only, content-addressed delta log for a workspace. Its job is to give every running sandbox a versioned, snapshot-isolated, lease-pinned view of the filesystem without ever copying the whole tree.

  • State is a Manifest: an ordered tuple of LayerRef pointers, with a monotonically increasing version integer.
  • Each layer on disk is one immutable directory of file-writes, symlinks, whiteouts, and opaque-dir markers.
  • A WorkspaceBinding anchors a sandbox’s root path to the layer-stack root; a WorkspaceLease pins a specific manifest version so its layers don’t get garbage-collected while in use.
  • Transactions serialize the publish step. Manifest write is CAS-style: if the active manifest moved while you held the transaction, you get ManifestConflictError and must retry.
  • Squash coalesces non-leased runs of layers into checkpoint layers once depth exceeds a configured ceiling.
Why append-only Multiple concurrent coding agents must be able to read a stable snapshot while others publish. Mutating layers in place would break snapshot isolation. Append-only + manifest CAS + reference-counted leases gives MVCC semantics with overlay-fs compatibility.

2.7 View & snapshot materialization §

To run a command, execution needs a single “flattened” directory it can pass to mount -t overlay. LayerStack.prepare_workspace_snapshot() stack.py:126-162 does this materialization.

  1. Read active manifest under the lock.
  2. Acquire snapshot lease so layers can’t be GC’d during the run.
  3. Create a transient lowerdir at runtime/transient-lowerdirs/<req>-<uuid>/lower.
  4. MergedView.materialize() view.py:195-219 walks layers oldest-first, applying opaque markers, then whiteouts, then writes. With share_inodes=True the materializer hardlinks files instead of copying (cross-device fallback uses shutil.copy2).
  5. Return a PrepareWorkspaceSnapshotResult stack.py:66-73 with lease_id, manifest_version, root_hash, lowerdir path, and timings.
EXDEV note _link_or_copy() in view.py attempts os.link first; on EXDEV (cross-filesystem) or EPERM it falls back to shutil.copy2. By the time the execution layer receives lease.lowerdir, EXDEV is already resolved — execution code never touches cross-device concerns.

2.8 Publisher & squash §

Manifest depth grows linearly with commits. The SquashService squash.py:59-147 coalesces runs of non-pinned layers into single checkpoint layers (B<version>-<uuid>).

BEFORE Manifest v=10, depth=12 AFTER Manifest v=11, depth=4 (L6 pinned by lease) (unpinned runs collapsed) ┌──────────────┐ top ┌──────────────────────────────────┐ top │ L10 │ │ B<v+1>-A │ │ L9 │ │ (was L10..L7 collapsed) │ │ L8 │ ├──────────────────────────────────┤ │ L7 │ ── plan(max=4) ──> │ L6 ●pinned (preserved) │ │ L6 ●pinned │ ├──────────────────────────────────┤ │ L5 │ │ B<v+1>-B │ │ L4 │ │ (was L5..L1 collapsed) │ │ L3 │ └──────────────────────────────────┘ │ L2 │ │ L1 (base) │ Invariants: └──────────────┘ bottom ─ pinned layers preserved ─ depth strictly decreases ─ merged-view semantics unchanged

Plan invariants squash.py:69-101:

  • Pinned (leased) layers are never collapsed.
  • Resulting depth must be strictly less than current.
  • Reduction must clear min_reduction, else skip.
  • Materialization respects opaque-dir markers so merged semantics are preserved.

2.9 Concurrency invariants §

  • Append-only versions: new_version = active.version + 1 publisher.py:107.
  • CAS at publish: active != expected_manifest → ManifestConflictError.
  • Idempotency via digest: .layer-metadata/<id>.digest sidecar. If a retry would produce the same digest as the head layer, publish returns the active manifest unchanged publisher.py:67-80.
  • Lease refcounting prevents GC of layers in use.
  • Lock hierarchy: storage-writer flock (cross-process) > RLock (intra-process) > lease registry.

3. OCC — Optimistic Concurrency Control §

3.1 Mental model §

OCC is database-style multi-version concurrency control adapted to filesystem mutations. The slogan is:

Slogan Prepare in parallel without locks. Collide-or-commit at apply time.

Two agents can prepare_changeset() against the same paths simultaneously. Both compute base hashes from their own snapshot. Both submit to the CommitQueue. The queue’s single worker thread serializes the actual layer publish, and the GatedStager re-checks each file’s hash against the live manifest. Whoever publishes first wins; the loser gets FileStatus.ABORTED_VERSION and is expected to re-read, re-prepare, retry.

This is fundamentally different from pessimistic locking: there is no per-file write lock held between prepare and commit. The only lock is the layer-stack publish lock, held for the brief moment of validation + publish.

3.5 Content hashing §

Hashing is plain SHA-256 over file bytes content_hashing.py:14-15:

def hash_bytes(content: bytes) -> str:
    return hashlib.sha256(content).hexdigest()

For multi-step changes on a single path, the preparer chains hashes changeset_preparation.py:175-202: running_hash starts from the snapshot, each WriteChange attaches it as base_hash, then updates to the precomputed-or-fresh hash of the new content. DeleteChange sets running_hash to None.

3.8 Gitignore handling §

The SnapshotGitignoreOracle gitignore.py:161-209 reads .gitignore files from the active manifest and caches by version. The underlying PathspecGitignoreOracle gitignore.py:44-158 uses the vendored pathspec library.

Directory-exclusion seal Git’s rule: once an ancestor directory is excluded, no deeper !re-include can rescue descendants. EphemeralOS honors this (gitignore.py:89-96).

Ignored paths are not silently dropped — they route DIRECT and can still be staged and published. Routing only chooses the validation policy, not whether to keep the change.

3.9 Stale-base rejection §

The actual OCC check is one function:

// path_staging.py:277-289
def _hash_mismatch(self, state: _PathStageState, base_hash: str | None) -> FileStatus | None:
    """Return ABORTED_VERSION if the gated hash chain disagrees, else None."""
    if not self._profile.check_hash or self._hasher is None:
        return None
    expected = base_hash or None
    current  = self._hasher.hash_current(state.content, exists=state.exists)
    if current != expected:
        return FileStatus.ABORTED_VERSION
    return None

It runs inside GatedStager._apply_change for WriteChange and DeleteChange path_staging.py:200-275, which is itself driven by CommitTransaction.revalidate_and_publish commit_transaction.py:65-141 — and crucially that function reads active_manifest = transaction.snapshot(), the live manifest at commit time, not prepared.snapshot from preparation time. That’s where “optimistic” meets “reject stale base.”

3.10 Maintenance §

AutoSquashMaintenancePolicy maintenance.py:28-106 runs after successful publishes. It checks current manifest depth against AUTO_SQUASH_MAX_DEPTH (default 100, service.py:23); if exceeded, it acquires a non-blocking lock and calls squasher.squash(max_depth). A pending-recheck flag re-triggers squash if another publish arrives during the squash window.

Maintenance runs in an executor thread (service.py:162-171), not on the publish critical path.

4. Command Execution §

4.1 Mental model §

Running pytest tests/ against a workspace that’s being modified by other agents requires four guarantees: (1) read-isolated input view, (2) writes captured rather than smeared into shared storage, (3) atomic publication of those writes, (4) clean tear-down even on crash. The recipe:

  1. Lease a snapshot from LayerStack → get an immutable lowerdir.
  2. Allocate upperdir (writes go here) and workdir (overlay scratch).
  3. Mount overlay (Linux: kernel overlayfs in a unshare child; macOS / no-cap-sysadmin: copy the lowerdir into a merged tree and diff after).
  4. Run the command via subprocess with stdout/stderr piped to ref files.
  5. Capture path changes by walking the upperdir (kernel path) or diffing merged-vs-lower (copy path).
  6. Convert overlay path changes to typed OCC mutations.
  7. Commit through OccClient (or skip if occ_apply=False, as in the read-only snapshot overlay handler).
  8. Release the lease; unmount; rmtree the run dir.

5. Overlay: Lower / Upper / Work / Workspace §

Overlay is the load-bearing primitive that makes parallel coding agents safe. Each agent gets its own upperdir, so writes from agent A don’t leak into agent B’s view. The lowerdir is shared (read-only); upperdir is unique per run; workdir is the kernel’s scratch.

5.1 The mount stack §

workspace_root (e.g. /workspace)
What the command sees. cd /workspace; pytest tests/ resolves files through this mount.
upperdir = OverlayLayout.writes
All writes during the command go here. After unmount, walked by walk_upperdir() to extract OverlayPathChange records.
workdir = OverlayLayout.kernel_scratch
Overlay’s internal scratch space. Must be on the same filesystem as upperdir. Untouched by EphemeralOS application code.
lowerdir = OverlayLayout.base_repo = lease.lowerdir
Materialized via MergedView.materialize() in runtime/transient-lowerdirs/<req>/lower. Read-only; hardlinks files from published layers when on the same FS.
layers/L000005-… → … → layers/B000001-base
The underlying LayerStack — immutable, refcounted by leases, optionally squashed into checkpoints.

Constraints from OverlayLayout.__post_init__ execution/overlay/layout.py:24-48:

  • All four paths (base_repo, writes, kernel_scratch, workspace_root) live strictly under scratch_root.
  • No two paths alias each other.
  • None contains a character from forbidden_overlay_path_chars.

5.4 Capture walk §

The namespace flow funnels into walk_upperdir(), which is a depth-first os.walk over the real overlay upperdir. The invariant is “opaque-dir marker before children” — so consumers can safely apply opaque first, knowing whiteouts inside the dir become redundant.

│   └── new.py              ← write "lib/new.py"

7. API §

The sandbox.api package is the orchestrator-facing facade. Two flavors of entry point:

  • Lifecycle verbs (create_sandbox, start_sandbox, stop_sandbox, delete_sandbox, ensure_sandbox_running, get_health, list_snapshots, list_sandboxes, get_sandbox, set_sandbox_labels, get_signed_preview_url, get_build_logs_url) — defined in api/provider_control.py; thin wrappers over host.lifecycle.

8. Daemon §

The daemon is the in-sandbox JSON-RPC server. It owns the LayerStack, OccService, OccClient, and execution.service inside the sandbox process. It receives envelopes via AF_UNIX or TCP, dispatches into daemon.handler.*, and returns JSON.

8.1 Process boot

python -m sandbox.daemon daemon/__main__.py:1-73 parses --socket / --pid-file / --tcp-host / --tcp-port / --auth-token (with env-var defaults), acquires an exclusive flock on the PID file (so a second instance fails fast), then runs serve(...) on the asyncio event loop.

8.3 Per-request plumbing

  • Path classification: classify_path() daemon/request_context.py:52-99 resolves symlinks, normalizes, rejects .. escapes, decides in-workspace vs out-of-workspace. Single source of truth for path policy.
  • No-follow IO: out-of-workspace writes go through _open_no_follow with O_NOFOLLOW walking, refusing O_ELOOP.
  • OccBackend cache: per-layer_stack_root LRU with 256 entries (daemon/occ_backend.py:45-89); evicts oldest on overflow.
  • Async bridge: run_sync_in_executor() daemon/async_bridge.py:258 seeds a sandbox_io_loop ContextVar pointing at the parent event loop, then dispatches to a 200-worker thread pool. Empirically: asyncio.to_thread gives ~6.4× parallelism; this gives ~45×.
  • Result projection: project_changeset() / project_conflict() turn ChangesetResult into the wire shape.

8.4 Host-side daemon client

host/daemon_client.py (~637 lines) is the orchestrator-side RPC client. The hot path:

  1. call_daemon_api(sandbox_id, op, args, timeout=60) resolves the TCP endpoint via the registered adapter; falls through to thin-client exec if no TCP available.
  2. _dispatch_once_with_retry() tries first _call_daemon_payload; if the daemon isn’t up (exit 97 = _THIN_CLIENT_CONNECT_FAILED), spawns the daemon via launch_daemon.sh, polls api.runtime.ready with backoff (0.25s, 0.5s, 1s, 2s — ~3.5s budget), then retries the call.
  3. TCP path: _call_tcp_daemon opens a fresh socket per call, writes payload + newline, reads until EOF.
  4. Thin-client path: thin_client.py shipped in the bundle is invoked via provider.exec() with the JSON payload on stdin.

The auth token (when configured) is injected into the payload as _eos_daemon_auth_token and stripped+validated server-side.
