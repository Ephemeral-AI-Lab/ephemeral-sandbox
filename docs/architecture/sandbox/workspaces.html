  • Not every task needs the same workspace lifetime or privacy model.
  • The package defines persistent, per-operation, and explicit private-session workspace behavior.
  • Daemon dispatch chooses isolated if open, otherwise ephemeral; main workspace provides binding and base identity.
  • All modes start from LayerStack truth but choose different temporary writable lifetimes.
  • Managed sandbox files live under the final /eos/runtime, /eos/state, and /eos/scratch layout.
Why two non-main modes exist

ephemeral_workspace exists because most tool calls are collaborative: an agent's write should be visible to co-worker agents on their next snapshot. Per-operation overlays plus OCC publish make every accepted write part of shared truth without forcing agents to message each other. This is what eliminates the A2A / P2P coordination layer that degrades other multi-agent coding frameworks — the artifact is the channel.

isolated_workspace exists because some tool calls are experimental: an agent needs a persistent private workbench to try a destructive script, install a package, or attempt a refactor that may fail. Writes inside an isolated handle are captured for audit but never published, and the upperdir is destroyed on exit_isolated_workspace. This is the explicit safety valve for the otherwise-shared blackboard.

Workspace Routing and State

main_workspace

Persistent identity: the bound repo plus LayerStack snapshots, written only through OCC.

ephemeral_workspace

Fresh per-operation overlays for shell, search, and plugin work; simple file APIs can use separate shared direct ops.

isolated_workspace

Opt-in private mode: enter once, run many tools, exit and discard the upperdir.

5.0 Filesystem Layout §

The Rust sandbox now keeps managed files inside three top-level /eos domains. Runtime executables and immutable package versions live under /eos/runtime; durable mutable state lives under /eos/state; transient upload, setup, overlay, command-session, and isolated workspace data lives under /eos/scratch.

5.2 main_workspace §

  • The sandbox needs one persistent repository identity to anchor every projection.
  • There is no dedicated main_workspace package; the persistent identity is the bound repo plus the active LayerStack manifest.
  • Host bootstrap uploads runtime, ensures git, then ensures the workspace base.
  • Main workspace is the source identity; the active LayerStack manifest is the mutable truth.

The main-workspace concept is intentionally not a package: the binding reader lives next to its sole consumer in the daemon, and the underlying helper lives in sandbox/layer_stack/workspace_binding.py. The host lifecycle calls ensure_workspace_base after uploading the runtime and ensuring git backend/src/sandbox/host/bootstrap.py:300-312. Runtime binding fails closed if the binding or manifest is missing/empty — the daemon-side _LayerStackBindingReader adds the manifest-non-empty precondition on top of the base helper backend/src/sandbox/daemon/occ_runtime_services.py:30-55, backend/src/sandbox/layer_stack/workspace_binding.py.

5.3 ephemeral_workspace §

  • Most tool calls should not keep private workspace state after the operation ends.
  • ephemeral_workspace creates fresh overlays for shell/plugin operations and fallback file work; simple file APIs use shared direct LayerStack/OCC ops instead.
  • LayerStack leases lower layers, overlay captures upperdir, and OCC publishes accepted changes.
  • Every call gets a clean snapshot and releases its scratch space when done.

Daemon routing uses an active isolated pipeline only if the caller already has an isolated handle; otherwise it returns get_sandbox_overlay(..., start=False) backend/src/sandbox/daemon/workspace_tool_dispatch.py:30-50. The start=False path is important: it gives the daemon a cached EphemeralPipeline facade without starting a persistent workspace mount backend/src/sandbox/ephemeral_workspace/overlay_registry.py:33-70.

5.3.1 Shared Direct File Ops §

Why direct file ops matter

The shared direct path skips overlay mount, mount-namespace allocation, the unshare child process, and upperdir capture entirely. read_file reads merged LayerStack directly; write_file and edit_file build typed OCC changes from tool args and submit them straight to the commit queue.

The throughput consequence: N agents writing N disjoint files publish in parallel with no CAS contention. The OCC commit queue batches disjoint non-atomic changes into one transaction with one manifest CAS, so N file-API writes cost one publish rather than N serialized publishes. Shell and search cannot use this path because their captured write set is not known until the operation finishes — those go through full overlay capture and serialize per atomic capture.

Practical impact: an agent loop dominated by read/write/edit verbs scales near-linearly with concurrent agents. An agent loop dominated by shell turns hits the OCC publish ceiling earlier and is the main candidate for batching or multi-manifest fan-out work.

5.4 isolated_workspace §

  • Low-confidence or experimental tasks can request a private scratch workspace across many tool calls.
  • isolated_workspace opens one persistent overlay namespace on enter, runs tools inside it, and discards upperdir on exit.
  • Enter leases LayerStack paths; runtime wires namespace/network/cgroup; exit releases the lease and removes scratch.
  • Isolated workspace gives privacy and continuity without publishing changes to shared truth.

The Rust agent-facing lifecycle tools are configured in .eos-agents/tools/enter_isolated_workspace.md and .eos-agents/tools/exit_isolated_workspace.md. Both declare the no_background_sessions hook, which resolves to RequireNoBackgroundSessions before the runtime calls eos-sandbox-api agent-core/crates/eos-tools/src/registry/config.rs, agent-core/crates/eos-tools/src/hooks/require_no_background_sessions.rs, agent-core/crates/eos-runtime/src/isolated_workspace.rs. The legacy Python tools module still records the earlier backend shape: it registered both lifecycle tools with the normal sandbox tool set, then called host lifecycle enter/exit with the current caller and sandbox id backend/src/tools/sandbox/_lib/registry.py:17-31, backend/src/tools/isolated_workspace/enter_isolated_workspace/definition.py:24-47, backend/src/tools/isolated_workspace/exit_isolated_workspace/definition.py:24-46.

The daemon-side pipeline is lazy. Rust enter operations reuse the active singleton when one exists; otherwise the first enter initializes a daemon-local IsolatedSession with the request's layer_stack_root, the canonical /eos/scratch/isolated scratch root, and the JSONL audit sink. The audit sink is daemon-owned rather than runtime-child-owned, so direct RPC tests can inspect lifecycle events even when they do not emit normal scenario logs sandbox/crates/eos-daemon/src/services/isolated_workspace/mod.rs, sandbox/crates/eos-isolated-workspace/src/session.rs, sandbox/crates/eos-isolated-workspace/src/audit.rs.

The Rust daemon now has the same public lifecycle op names behind sandbox/crates/eos-daemon/src/services/isolated_workspace/mod.rs. When enabled, it opens an caller-keyed eos_isolated_workspace::IsolatedSession, emits JSONL lifecycle/tool events, spawns eosd ns-holder, opens inherited namespace FDs from /proc/<pid>/ns, mounts the isolated overlay through eosd ns-runner --mount-overlay, and lets Linux workspace/search/command ops use the active handle without calling OCC. sandbox/crates/eos-daemon/src/workspace_ops.rs routes read_file, write_file, edit_file, glob, and grep; sandbox/crates/eos-daemon/src/services/command_session/mod.rs routes exec_command into the isolated policy adapter while eos-command-session owns command-session start/write/cancel/collect/count lifecycle. sandbox/crates/eos-runner/src/setns.rs implements runner-side setns entry and setns overlay mounting, sandbox/crates/eos-isolated-workspace/src/network.rs creates/moves/attaches the bridge/veth pair and installs static nftables NAT/filter rules through netlink on Linux, and sandbox/crates/eos-ns-holder/src/lib.rs performs best-effort rtnetlink loopback-up/default-IPv6-route hardening plus namespace-side veth link/address/default-route programming. Local amd64 Docker/dask proof passed in an image without ip or nft: enter, bridge/veth, namespace default route, finite command, command-session stdin/progress/natural-exit/timeout/cancel notifications, status/list closure, shared no-publish checks, and two isolated agents both binding TCP port 3000 all passed. Enter and exit run daemon command-session cleanup as a bypass backstop if the model-facing RequireNoBackgroundSessions hook is skipped; that cleanup is not a command-session-only guard. Exit returns and audits cleanup inspection fields for handle/agent map counts, lease release, active leases, holder/cgroup/scratch state, mountinfo references when available, and host-veth removal; the latest live Docker rerun asserts those fields under the real isolated kernel path.
