10.1 Direct LayerStack Read §

[read_file request]
        |
        v
[daemon workspace router]
        |
        +--> [workspace binding exists]
        |       |
        |       v
        |   [LayerStack projection lookup]
        |       |
        |       v
        |   [read content blob / base file]
        |       |
        |       v
        |   [return file contents]
        |
        +--> [no binding] -> [fallback provider file read path]
  1. The daemon router receives a single-path read_file request and confirms no isolated handle is active for the caller.
  2. The router converts the sandbox path to a workspace-relative path through the current WorkspaceBinding.
  3. LayerStack reads the active manifest and asks MergedView for the file content.
  4. MergedView searches newest-first through layer indexes, applying whiteout/opaque semantics during lookup.
  5. The router returns content with direct fast-path timings and zero overlay resource metrics.

Relevant code: router fast path sandbox/crates/eos-daemon/src/dispatch/dispatcher.rs, path binding sandbox/crates/eos-daemon/src/dispatch/dispatcher.rs, direct resource metrics sandbox/crates/eos-daemon/src/dispatch/dispatcher.rs, and merged-view reads sandbox/crates/eos-layerstack/src/stack/projection.rs.

10.2 Ephemeral Shell or Search Operation §

[shell command request]
        |
        v
[daemon workspace router]
        |
        v
[ephemeral overlay pipeline]
        |
        v
[run command in overlay workspace]
        |
        +--> [stdout/stderr/status only]
        |       -> [return tool result]
        |
        +--> [filesystem writes captured]
                |
                v
        [overlay diff -> OCC changeset -> publish]
  1. The router selects the ephemeral pipeline because the caller has no open isolated handle.
  2. The pipeline optionally performs shell pre-mount maintenance when manifest depth is high.
  3. Overlay lifecycle leases the active LayerStack snapshot and allocates one run dir with upper/work.
  4. The namespace runner launches a fresh user/mount namespace and sends the operation payload to the namespace entrypoint.
  5. The entrypoint mounts overlayfs onto the workspace root and runs the primitive command inside that view.
  6. For read-only intent, the parent releases the overlay handle without publishing.
  7. For write intent, the parent captures upperdir changes, converts them to OCC changes, publishes if accepted, emits workspace events, and destroys the run dir.

Relevant code: pipeline run loop sandbox/crates/eos-daemon/src/services/overlay/mod.rs, shell pre-mount maintenance sandbox/crates/eos-daemon/src/services/overlay/mod.rs, namespace entrypoint sandbox/crates/eos-runner/src/fresh_ns/child.rs, and publish helper sandbox/crates/eos-ephemeral-workspace/src/finalize.rs.

10.3 OCC Publish from API Write/Edit §

[write_file / edit_file]
        |
        v
[workspace router detects direct API write]
        |
        v
[prepare OCC changeset]
        |  path safety, base hash, content delta
        v
[commit queue]
        |
        v
[transaction revalidation]
        |
        +--> [valid] ----> [LayerStack publish -> projection refresh]
        |
        +--> [conflict] -> [structured conflict result]
  1. The router converts write_file or edit_file arguments into typed OCC changes.
  2. OCC prepare normalizes the path, checks gitignore state from the snapshot, and attaches base hashes for gated paths.
  3. The prepared changeset enters the commit queue, which keeps publish work serialized and only batches safe disjoint requests.
  4. The transaction reads the current active manifest, revalidates the path group, writes accepted content into OCC staging, and builds layer deltas.
  5. The publisher appends one immutable layer and writes the new manifest if the expected manifest still matches.
  6. The daemon projects OCC statuses into public tool result conflicts and published path metadata.

Relevant code: direct write/edit router sandbox/crates/eos-daemon/src/dispatch/dispatcher.rs, preparer sandbox/crates/eos-occ/src/service.rs, commit queue sandbox/crates/eos-occ/src/commit_queue.rs, transaction sandbox/crates/eos-occ/src/service.rs, and result projection sandbox/crates/eos-daemon/src/services/overlay/mod.rs.

10.4 Isolated Workspace Tool Lifecycle §

[enter_isolated_workspace]
        |
        v
[allocate snapshot lease + scratch directory]
        |
        v
[create isolated namespace / pipeline]
        |
        v
[foreground tools route by active caller_id handle]
        |
        v
[exit_isolated_workspace]
        |
        v
[teardown namespace, release lease, remove scratch]
        |
        v
[changes discarded]
  1. The Rust public tools are configured in .eos-agents/tools; both lifecycle tools declare no_background_sessions and call the daemon isolated API through executor-captured SandboxToolService after the prehook passes.
  2. RequireNoBackgroundSessions is the gate before private scratch exists. It checks outstanding workflows, in-flight subagents, and daemon-visible command sessions for the same agent; any nonzero lane returns ephemeral_jobs_in_flight instead of racing an isolated namespace transition.
  3. Daemon RPC registers api.isolated_workspace.enter, exit, status, list_open, and gated test_reset. The enter/exit handlers catch IsolatedWorkspaceError and return structured handler-level error payloads.
  4. The first daemon enter lazily constructs the resident IsolatedPipeline: it binds the LayerStack client from the requested root, uses the overlay writable root for scratch, installs the JSONL audit sink, publishes the singleton, then runs pipeline initialization.
  5. Pipeline enter prepares one LayerStack snapshot lease, allocates scratch upper/work, builds the handle, wires the namespace holder, namespace fds, veth, overlay mount, DNS, and cgroup, then persists the handle map and emits an enter audit event.
  6. The overlay mount reuses the shared kernel mount path: namespace_runtime.mount_overlay() passes user and mount namespace fds to the setns helper, and the helper enters those namespaces before calling sandbox.overlay.kernel_mount.validate_mount_inputs() and mount_overlay().
  7. Subsequent foreground tool calls reuse the open handle. Python run_tool_call() wraps the handle as an OverlayHandle; Rust workspace_ops and the daemon command-session policy route by the active caller_id handle, mark workspace=isolated, and record changed paths for audit only.
  8. No isolated write publishes through OCC. Rust model-facing exit first requires zero background sessions, then daemon exit removes the handle, measures discarded upperdir bytes, kills the holder, tears down veth, closes fds, releases the lease, removes cgroup/scratch, and emits an exit audit event.

Relevant code: Rust tool config .eos-agents/tools/enter_isolated_workspace.md, .eos-agents/tools/exit_isolated_workspace.md, Rust background-session hook agent-core/crates/eos-tools/src/hooks/require_no_background_sessions.rs, runtime bridge agent-core/crates/eos-runtime/src/isolated_workspace.rs, sandbox API bridge agent-core/crates/eos-sandbox-api/src/tool_api/isolated.rs, Rust isolated file/search routing sandbox/crates/eos-daemon/src/workspace_ops.rs, sandbox/crates/eos-daemon/src/workspace_ops/isolated_workspace.rs, Rust command routing and runtime sandbox/crates/eos-daemon/src/services/command_session/mod.rs, sandbox/crates/eos-daemon/src/services/command_session/ports/isolated.rs, sandbox/crates/eos-command-session/src/manager.rs, Rust daemon lifecycle sandbox/crates/eos-daemon/src/services/isolated_workspace/mod.rs, tools registry agent-core/crates/eos-tools/src/tools/sandbox/mod.rs, host lifecycle agent-core/crates/eos-sandbox-host/src/lifecycle.rs, lazy session construction sandbox/crates/eos-isolated-workspace/src/session.rs, handle lifecycle sandbox/crates/eos-isolated-workspace/src/session/lifecycle.rs, and isolated run/no-publish path sandbox/crates/eos-isolated-workspace/src/session/lifecycle.rs.

10.5 Plugin Write Operation §

[plugin write tool]
        |
        v
[daemon plugin host]
        |
        v
[execute against workspace projection]
        |
        v
[capture plugin output + touched files]
        |
        v
[prepare OCC changeset]
        |
        v
[commit queue -> LayerStack publish -> refreshed projection]
  1. Host dispatch resolves plugin manifest, installs/refreshes the bundle, and ensures daemon runtime for the plugin digest.
  2. The daemon plugin registry dispatches a WRITE_ALLOWED op through the generic workspace overlay wrapper unless the plugin opted out.
  3. The wrapper acquires an operation overlay from the ephemeral pipeline and runs the plugin child in a fresh namespace.
  4. The child sees the workspace root as the overlay projection and writes only into that operation's upperdir.
  5. The daemon parent captures the upperdir, publishes through publish_cycle(), attaches publish metadata, and destroys the overlay.

Relevant code: host dispatch sandbox/crates/eos-daemon/src/services/plugins/dispatch.rs, plugin registry sandbox/crates/eos-daemon/src/services/plugins/service.rs, operation overlay dispatch sandbox/crates/eos-daemon/src/services/plugins/overlay.rs, and workspace publish cycle sandbox/crates/eos-ephemeral-workspace/src/finalize.rs.

10.6 Pyright Latest-Projection Refresh §

[Pyright / LSP request]
        |
        v
[resolve latest workspace projection]
        |
        v
[start or reuse language-server session]
        |
        v
[apply projection refresh to server view]
        |
        +--> [diagnostics / hover / references]
        |       -> [return read-only result]
        |
        +--> [LSP edit action]
                -> [OCC changeset -> publish]
  1. A read-only LSP tool asks the session manager for the current session.
  2. The manager compares the session's manifest key with the active key from projection or overlay.
  3. If different, it acquires or remounts a private overlay handle so Pyright sees the latest LayerStack projection.
  4. The ephemeral pipeline's workspace event bus also sends publish/full-resync events to the session manager.
  5. Pyright receives watched-file notifications, closes deleted open documents, and syncs changed open documents.
  6. LSP write operations, such as applying workspace edits, use a separate operation overlay and publish through OCC.

Relevant code: session reconciliation agent-core/crates/eos-plugin-catalog/lsp/runtime/ppc_service.py, overlay acquisition agent-core/crates/eos-plugin-catalog/lsp/runtime/ppc_service.py, event pump agent-core/crates/eos-plugin-catalog/lsp/runtime/ppc_service.py, Pyright remount/notification agent-core/crates/eos-plugin-catalog/lsp/runtime/ppc_service.py, agent-core/crates/eos-plugin-catalog/lsp/runtime/ppc_service.py, and LSP apply path agent-core/crates/eos-plugin-catalog/lsp/runtime/ppc_service.py.

Behavior suites

  • sandbox/crates/eos-e2e-test/tests integrated public file, shell, OCC, and overlay workflows.
  • integration-test/SPEC.md ephemeral-mode cleanup and conflict workflows.
  • integration-test/SPEC.md background command heartbeat/cancel/isolated-gate workflows.
  • integration-test/SPEC.md direct daemon RPC lifecycle, isolation, GC, networking, stress, and performance workflows.
  • integration-test/SPEC.md plugin/LSP runtime workflows.

Live artifacts

Scenario-style runs write under .sweevo_runs/scenario_logs. Direct isolated-workspace pytest/soak paths do not rely on those scenario logs; they read daemon JSONL audit from EOS_ISOLATED_WORKSPACE_AUDIT_PATH or the default /tmp/sandbox_isolated_workspace_events.jsonl inside the active sandbox.

Isolated audit sequence

The daemon audit sink appends sandbox_isolated_workspace_enter, sandbox_isolated_workspace_tool_call, sandbox_isolated_workspace_exit, and GC/eviction events. Test fixtures truncate and snapshot the in-container JSONL via raw_exec, then derive phase/latency assertions from the copied file.

Evidence anchors: sandbox/crates/eos-isolated-workspace/src/session.rs, sandbox/crates/eos-isolated-workspace/src/session.rs, integration-test/SPEC.md, and integration-test/SPEC.md.
