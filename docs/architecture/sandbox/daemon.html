  • The daemon is the resident in-sandbox control plane: it owns RPC dispatch, cached LayerStack/OCC services, workspace routing, in-flight invocation tracking, and isolated-workspace lifecycle calls.
  • Host code calls one public transport; the host client chooses Docker TCP when available or the AF_UNIX thin-client path through provider exec.
  • Each request carries an op, invocation_id, and JSON object args; the dispatcher validates it, registers it for cancellation/heartbeat, dispatches the handler, attaches runtime timings, and deregisters it.
  • Engine background tasks are host-managed, but sandbox-bound background work is correlated to daemon invocations so cancellation, heartbeat, TTL reaping, and isolated-workspace gating agree.

6.1 Control Plane Shape §

The daemon is not the storage layer and it is not a provider adapter. It is the warm process inside the sandbox that receives one JSON envelope at a time, reuses expensive runtime state, and routes each operation to the right workspace implementation. Host-side code keeps the provider boundary in Provider; daemon-side code keeps workspace truth in LayerStack and OCC.

6.2 Startup and Transport Workflow §

The host has two ways to reach the same daemon contract. Docker can expose a cached TCP endpoint; providers that cannot publish a port use provider.exec to run the bundled thin client inside the sandbox. Both paths send the same JSON envelope and receive the same JSON response.

During the Rust migration, EOS_SANDBOX_RUNTIME=rust swaps the Python launch script and Python thin client for the uploaded /tmp/eos-sandbox-runtime/eosd binary. For Docker TCP, the daemon binds 0.0.0.0 inside the container so Docker's host-loopback port mapping can reach it; the host still exposes only the mapped host endpoint returned by the provider.

Duplicate registration of the same handler is a no-op, while a different handler for an existing op raises immediately, so runtime extension remains explicit instead of silently overriding commands backend/src/sandbox/daemon/rpc/dispatcher.py:37-53. 6.5 Workspace Routing and State §

The key daemon router is dispatch_workspace_tool_call(). It creates a ToolCallRequest with invocation id, agent id, verb, intent, args, and background flag. If the agent has an open isolated handle, the call runs inside isolated_workspace. Otherwise single-file read_file, write_file, and edit_file use direct LayerStack/OCC fast paths when the path can bind cleanly. Everything else goes through the ephemeral_workspace overlay pipeline backend/src/sandbox/daemon/workspace_tool_dispatch.py:30-75.

File reads

Simple reads bind the requested path to the current workspace view and return the latest file content.

User-visible effect: no workspace changes.

File writes and edits

Single-file writes and edits publish directly through OCC when the path maps cleanly to the workspace.

User-visible effect: accepted changes become part of the shared workspace history.

Shell, grep, and glob

Broader operations run in a fresh ephemeral workspace view. Write-capable calls publish only after captured changes pass OCC.

User-visible effect: successful mutations appear in the shared workspace; rejected mutations leave it unchanged.

Isolated workspace

When an agent has entered isolated mode, its tool calls reuse that private workspace instead of publishing through OCC.

User-visible effect: changes stay private and are discarded on isolated exit.

6.6 Workflow Traces §

These traces are intentionally redundant with the component sections. They answer the practical question: "what state changes after this call?" Use them as entry points, then follow the inline links to the deeper sections.

Direct file read

  1. Host sends api.v1.read_file with caller identity and path.
  2. Daemon creates ToolCallRequest and confirms no isolated handle exists.
  3. Router binds the path through workspace binding.
  4. OCC runtime services read active LayerStack text directly.
  5. Result returns workspace="ephemeral", content, existence flag, and resource timings.

Direct write or edit

  1. Host sends api.v1.write_file or api.v1.edit_file.
  2. Router binds a single path and builds typed OCC changes.
  3. OccService.apply_changeset() validates and publishes to the shared LayerStack.
  4. Guarded conflict/result projection becomes the tool response.
  5. No transient overlay namespace is mounted for the fast path.

Shell or search

  1. Router bypasses the direct file fast path.
  2. Ephemeral pipeline acquires a fresh snapshot overlay and namespace.

  5. Lease guard destroys the overlay handle after the response.

6.8 Cancellation and Heartbeats §

Cancellation is intentionally two-stage for sandbox-bound background tasks. The supervisor first asks the daemon to cancel the sandbox invocation, then it updates the local task state and cancels the host asyncio task when the task type permits it backend/src/engine/background/task_supervisor.py:317-339, backend/src/engine/background/task_supervisor.py:432-449. The daemon control handler cancels the registered task and waits briefly for cleanup, reporting whether the task was found and whether cleanup completed backend/src/sandbox/daemon/builtin_operations.py:94-110.

  1. Manager heartbeat groups running sandbox invocation ids by sandbox id every EOS_BACKGROUND_HEARTBEAT_INTERVAL_S seconds.
  2. It calls sandbox_api.heartbeat() for each sandbox, which maps to api.v1.heartbeat through daemon_invocations.heartbeat() backend/src/engine/background/task_supervisor.py:463-499, backend/src/sandbox/api/daemon_invocations.py:34-48.
  3. The daemon registry refreshes last_seen for invocation ids it still tracks and returns a touched count backend/src/sandbox/daemon/rpc/in_flight.py:87-96.
  4. If heartbeats stop, the daemon TTL reaper cancels stale background tasks, marks them ttl_reaped, and increments ttl_reaped_total without deleting the registry entry until normal deregistration backend/src/sandbox/daemon/rpc/in_flight.py:111-149.

The registry only counts entries that are both background and not done for an agent backend/src/sandbox/daemon/rpc/in_flight.py:98-109. That count is the daemon-side half of the isolated-workspace gate described in Isolated Background Gate.

6.9 Isolated Background Gate §

Isolated vs ephemeral background behavior Ephemeral background shell tasks are normal sandbox-bound daemon invocations with heartbeats and TTL cleanup. Isolated enter is blocked while those tasks are still running for the same agent. Isolated exit cancels sandbox-bound background work before tearing down the namespace.

The host-side isolated lifecycle checks both local manager state and daemon-visible in-flight state before entering. A nonzero count returns ephemeral_jobs_in_flight, because an active background task may still write through the default ephemeral/OCC path and should not race an isolated namespace transition backend/src/sandbox/host/isolated_workspace_lifecycle.py:41-77, backend/src/sandbox/host/isolated_workspace_lifecycle.py:244-275. On exit, the host lifecycle cancels per-agent sandbox-bound background tasks, then calls daemon isolated exit and includes the eviction count in the returned phases backend/src/sandbox/host/isolated_workspace_lifecycle.py:98-136, backend/src/sandbox/host/isolated_workspace_lifecycle.py:253-264.
