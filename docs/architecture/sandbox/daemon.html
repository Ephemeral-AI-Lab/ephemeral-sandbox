  • The daemon is the resident in-sandbox control plane: it owns RPC dispatch, cached LayerStack/OCC services, workspace routing, in-flight invocation tracking, and isolated-workspace lifecycle calls.
  • Host code calls one public transport; the host client chooses Docker TCP when available or the AF_UNIX thin-client path through provider exec.
  • Each request carries an op, invocation_id, and JSON object args; the dispatcher validates it, registers it for cancellation/heartbeat, dispatches the handler, attaches runtime timings, and deregisters it.
  • Engine background tasks are host-managed, but sandbox-bound background work is correlated to daemon invocations so cancellation, heartbeat, TTL reaping, and isolated-workspace gating agree.

6.1 Control Plane Shape §

The daemon is not the storage layer and it is not a provider adapter. It is the warm process inside the sandbox that receives one JSON envelope at a time, reuses expensive runtime state, and routes each operation to the right workspace implementation. Host-side code keeps the provider boundary in Provider; daemon-side code keeps workspace truth in LayerStack and OCC.

6.2 Startup and Transport Workflow §

The host has two ways to reach the same daemon contract. Docker can expose a cached TCP endpoint; providers that cannot publish a port use provider.exec to run the bundled thin client inside the sandbox. Both paths send the same JSON envelope and receive the same JSON response.

The Rust host uploads /eos/runtime/daemon/eosd as the resident daemon binary and uses it for both foreground daemon startup and AF_UNIX client calls. For Docker TCP, the daemon binds 0.0.0.0 inside the container so Docker's host-loopback port mapping can reach it; the host still exposes only the mapped host endpoint returned by the provider.

  2. The daemon server accepts one newline-delimited JSON envelope, applies auth and size/read-time limits, then delegates valid requests to the dispatcher sandbox/crates/eos-daemon/src/transport/server.rs.
  3. The thin client connects to runtime.sock, writes the envelope plus newline, shuts down the write side, reads response chunks, and returns structured exit codes for connect and I/O failure agent-core/crates/eos-sandbox-host/src/daemon_client/shell.rs.
  4. If the first send reports connect failure, or an empty response that is safe to retry, the host launches the daemon through the selected runtime launcher agent-core/crates/eos-sandbox-host/src/daemon_client.rs.
  5. The host runs eosd daemon --spawn, which checks for a live non-zombie PID, validates the environment signature, serializes respawn with flock when available, removes stale socket/PID state, preserves the PID/socket/log contract, and starts the Rust AF_UNIX/TCP server sandbox/crates/eosd/src/main.rs, sandbox/crates/eos-daemon/src/transport/server.rs.
  6. After spawn, the host probes api.runtime.ready. Readiness checks the control plane, data-plane services, and mutation gate before the original envelope is retried with bounded connect backoff agent-core/crates/eos-sandbox-host/src/daemon_client.rs, sandbox/crates/eos-daemon/src/ops.

Invariant The daemon may reuse caches, but it must not hide publish semantics. Direct file fast paths still go through OCC; shell and search operations still run through the workspace pipeline; isolated writes remain private scratch until explicit exit.

6.4 Operation Surface §

The operation table is registered at import time. Duplicate registration of the same handler is a no-op, while a different handler for an existing op raises immediately, so runtime extension remains explicit instead of silently overriding commands sandbox/crates/eos-daemon/src/dispatch/dispatcher.rs. Built-in registrations cover isolated workspace lifecycle, LayerStack base/snapshot/lease control, file/shell/command tools, command-session controls, plugin ensure/status, and in-flight cancellation/heartbeat/count operations sandbox/crates/eos-daemon/src/dispatch/dispatcher.rs. In Rust mode, sandbox/crates/eos-daemon/src/dispatch/dispatcher.rs owns the op table, sandbox/crates/eos-daemon/src/ops/command_sessions.rs forwards command ops, sandbox/crates/eos-daemon/src/services/command_session/mod.rs chooses workspace policy and maps JSON, and eos-command-session owns the runtime registry/process lifecycle. Rust no longer registers raw daemon search ops.

6.5 Workspace Routing and State §

The key daemon router is dispatch_workspace_tool_call(). It creates a ToolCallRequest with invocation id, caller id, verb, intent, args, and background flag. If the agent has an open isolated handle, the call runs inside isolated_workspace. Otherwise single-file read_file, write_file, and edit_file use direct LayerStack/OCC fast paths when the path can bind cleanly. Everything else goes through the ephemeral_workspace overlay pipeline sandbox/crates/eos-daemon/src/dispatch/dispatcher.rs.

File reads

Simple reads bind the requested path to the current workspace view and return the latest file content.

User-visible effect: no workspace changes.

File writes and edits

Single-file writes and edits publish directly through OCC when the path maps cleanly to the workspace.

User-visible effect: accepted changes become part of the shared workspace history.

Shell

Broader operations run in a fresh ephemeral workspace view. Write-capable calls publish only after captured changes pass OCC.

User-visible effect: successful mutations appear in the shared workspace; rejected mutations leave it unchanged.

Isolated workspace

When an agent has entered isolated mode, its tool calls reuse that private workspace instead of publishing through OCC.

User-visible effect: changes stay private and are discarded on isolated exit.

A workspace-base reset intentionally stops the peer overlay and drops OCC services before rebuilding the base sandbox/crates/eos-daemon/src/ops.

Routing invariant read_file, write_file, and edit_file are the only daemon tool verbs eligible for the direct LayerStack/OCC fast path. A path-binding failure does not fail the request; it falls back to the same overlay pipeline as shell/search so behavior stays correct for nonstandard paths sandbox/crates/eos-daemon/src/dispatch/dispatcher.rs.

6.6 Workflow Traces §

These traces are intentionally redundant with the component sections. They answer the practical question: "what state changes after this call?" Use them as entry points, then follow the inline links to the deeper sections.

Direct file read

  1. Host sends api.v1.read_file with caller identity and path.
  2. Daemon creates ToolCallRequest and confirms no isolated handle exists.
  3. Router binds the path through workspace binding.
  4. OCC runtime services read active LayerStack text directly.
  5. Result returns workspace="ephemeral", content, existence flag, and resource timings.

sandbox/crates/eos-daemon/src/dispatch/dispatcher.rs

Direct write or edit

  1. Host sends api.v1.write_file or api.v1.edit_file.
  2. Router binds a single path and builds typed OCC changes.
  3. OccService.apply_changeset() validates and publishes to the shared LayerStack.
  4. Guarded conflict/result projection becomes the tool response.
  5. No transient overlay namespace is mounted for the fast path.

sandbox/crates/eos-daemon/src/dispatch/dispatcher.rs

Shell or search

  1. Router bypasses the direct file fast path.
  2. Ephemeral pipeline acquires a fresh snapshot overlay and namespace.
  3. Tool runs inside the mounted workspace view.
  4. Write-intent calls capture upperdir changes and publish through OCC.
  5. Lease guard destroys the overlay handle after the response.

6.7 Background Task Lifecycle §

Background execution is split across typed control planes. The engine owns subagent records, delegated workflow records, command-session completion notifications, progress snapshots, and task status. eos-command-session owns long-running command-session lifecycle, registry, process groups, output cursors, stdin, cancel, sweep, and completion parking; the daemon owns workspace policy selection, concrete daemon ports, isolated active-session callbacks, and cancellable in-flight sandbox work. The public model surface no longer exposes generic background check, wait, or cancel tools.

6.8 Cancellation and Heartbeats §

Cancellation is intentionally two-stage for sandbox-bound background work. Engine session managers first ask the daemon to cancel the sandbox-owned work, then update local session state; command-session teardown is owned by CommandSessionManager agent-core/crates/eos-engine/src/background/session_managers/command/manager.rs. The daemon control handler cancels the registered task and waits briefly for cleanup, reporting whether the task was found and whether cleanup completed sandbox/crates/eos-daemon/src/ops.

  1. Manager heartbeat groups running sandbox invocation ids by sandbox id every EOS_BACKGROUND_HEARTBEAT_INTERVAL_S seconds.
  2. Sandbox control calls map sandbox_api.heartbeat() to api.v1.heartbeat through daemon_invocations.heartbeat() agent-core/crates/eos-sandbox-api/src/tool_api/control.rs.
  3. The daemon registry refreshes last_seen for invocation ids it still tracks and returns a touched count sandbox/crates/eos-daemon/src/runtime/invocation_registry.rs.
  4. If heartbeats stop, the daemon TTL reaper cancels stale background tasks, marks them ttl_reaped, and increments ttl_reaped_total without deleting the registry entry until normal deregistration sandbox/crates/eos-daemon/src/runtime/invocation_registry.rs.

The invocation registry only counts entries that are both background and not done for an agent sandbox/crates/eos-daemon/src/runtime/invocation_registry.rs. The Rust isolated-workspace agent gate now combines persisted delegated workflow state, in-flight subagent state, and the eos-command-session runtime count exposed by the daemon as api.v1.command_session_count; command sessions are the persistent command lane.

6.9 Isolated Background Gate §

Isolated vs ephemeral background behavior Background sessions are outstanding workflows, in-flight subagents, and eos-command-session runtime sessions visible through the daemon adapter. Rust agent-facing isolated enter and exit are blocked while those sessions are still open for the same agent.

The current Rust agent path runs RequireNoBackgroundSessions before both lifecycle RPCs. It checks outstanding workflows, in-flight subagents, and daemon-visible command sessions; any open lane returns a hook_failure tagged ephemeral_jobs_in_flight. On exit, the model-facing path refuses while workflows or command sessions remain, then calls daemon isolated exit only after the guard passes agent-core/crates/eos-tools/src/hooks/require_no_background_sessions.rs. The older Python host lifecycle checked local manager state plus daemon command-session state before enter and cancelled per-caller sandbox-bound background work before exit agent-core/crates/eos-sandbox-host/src.
