Sandbox Tool Dispatch Path #

This page follows one sandbox-backed tool call from an agent-facing wrapper through sandbox.api, the host daemon client, provider exec or daemon TCP, the in-sandbox daemon, and the workspace execution path. Current verdict: sandbox providers own containers and primitive exec; sandbox runtime owns workspace semantics, direct file fast paths, overlay publish, isolated scratch, and daemon in-flight control. For LLM model providers under backend/src/providers, use Model Provider Runtime.

  • Provider bootstrap picks Docker or Daytona once per process, then the registry supplies the default adapter and per-sandbox bindings.
  • raw_exec calls provider exec directly; guarded workspace operations call the resident daemon through call_daemon_api().
  • Daemon transport prefers a Docker TCP endpoint when exposed and falls back to provider exec plus the AF_UNIX thin client.
  • Daemon workspace routing tries active isolated workspace first, then direct LayerStack/OCC for single-file verbs, then ephemeral overlay execution.

Provider Selection #

bootstrap_sandbox_provider() resolves EOS_SANDBOX_PROVIDER first, then central sandbox.default_provider. Valid values are docker and daytona. A process-level sentinel makes the first resolved provider sticky; later calls with the same value are no-ops, and later calls with a different value warn and return.

ProviderAdapter is the common protocol: health, snapshots, create/get/list/start/stop/delete/set-labels, preview and build-log URLs, async exec, and context_preparer(). Provider lifecycle orchestration is outside the provider package: sandbox.host.lifecycle.create_sandbox() uses the default adapter, registers the minted sandbox id, then runs post-create setup; start runs post-start setup; delete forgets plugin state and disposes the adapter binding.

set_default_provider() and get_default_provider() handle create/list/health paths before a sandbox id exists.

Exec Transports #

There are two execution surfaces. sandbox.api.raw_exec() is the unguarded provider primitive: it gets the registered adapter and calls adapter.exec(sandbox_id, command, cwd, timeout). Workspace tools do not use this path directly; they use the daemon transport so workspace binding, OCC, overlay, in-flight cancellation, and typed response parsing all stay inside the sandbox runtime.

call_daemon_api() builds daemon args with layer_stack_root, resolves the adapter, optionally resolves a Docker daemon TCP endpoint, and sends one JSON envelope. If TCP is unavailable or stale, it falls back to provider exec of the bundled thin client, which talks to the daemon's AF_UNIX socket inside the sandbox. The daemon can be relaunched on connect failure, then api.runtime.ready probes control plane, data plane, and mutation gate before retrying the original request.

Fallback pathTCP failure falls back to provider exec plus the AF_UNIX thin client after invalidating the cached endpoint.

Trust boundary Provider exec is a transport primitive. It is not the workspace mutation boundary. Durable workspace writes must publish through daemon workspace routing and OCC unless the caller explicitly uses raw_exec.

Tool To Daemon Path #

Agent-facing sandbox tools are thin wrappers around typed public API calls. tools.sandbox._lib.registry marks each sandbox tool with SANDBOX_CONTEXT, allowing the engine to inject provider context and allowing background launch code to recognize sandbox-bound work. The wrapper resolves repo-relative paths, builds SandboxCaller identity from tool metadata, and calls sandbox.api.

1
Tool wrapper

read_file, write_file, edit_file, grep, glob, and exec_command create request models with caller identity, path/cwd, description, and audit metadata.

2
Public API

sandbox.api.tool builds daemon payload identity fields, selects an api.v1.* op, dispatches with call_sandbox_daemon(), and parses the result.

3
Host daemon client

call_daemon_api() adds layer_stack_root, sends the envelope over TCP or thin client, and converts daemon transport failures into structured dispatch errors.

4
Daemon operation table

dispatcher._register_builtin_operations() registers workspace ops, isolated lifecycle ops, daemon control ops, layer-stack control, runtime readiness, plugin ops, and audit ops.

The daemon still has command-execution operations, but the model-facing public command tool is exec_command. Command sessions use exec_command plus typed command-session controls; engine background dispatch is restricted by engine.background.policy.

Workspace Routing #

The first workspace decision point is dispatch_workspace_tool_call(). It normalizes the request into ToolCallRequest, identifies the agent, checks active isolated workspace state, and then applies the routing order below. This order matters because isolated mode is explicit private scratch, direct file fast paths avoid overlay cost for normal file APIs, and shell/search need a real filesystem view.

Isolated firstActive isolated workspace handles take precedence and skip direct fast paths.
Direct fast pathOutside isolated mode, single-path read, write, and edit requests try the bound LayerStack/OCC route.
Overlay fallbackOutside-binding paths and shell, search, or plugin-style verbs run through EphemeralPipeline; write-capable results publish through OCC.
Direct read
Reads merged text from services.manager.read_text(path) and returns zero overlay resource metrics.
Direct write/edit
Builds typed API write/edit changes and calls services.occ_service.apply_changeset(); published paths project back into guarded mutation results.
Ephemeral overlay
Acquires a latest snapshot lease, runs the tool in a private namespace, captures upperdir changes, and publishes through OCC.
Isolated workspace
Host enter rejects when sandbox-bound background work is still running, daemon enter creates an agent handle, and exit tears down private state after draining background work.

OccService prepares changesets lock-free, infers base hashes against the chosen snapshot, then serializes commit through CommitQueue and CommitTransaction. The transaction revalidates against the live active manifest, so a stale prepared changeset receives normal OCC rejection instead of silently overwriting concurrent work.

Background Control #

Background sandbox control is split by typed surface. launch_background_tool() is subagent-only. exec_command creates an eos-command-session runtime session through the daemon adapter, registers its command_session_id with the engine supervisor, and exposes typed command-session progress, stdin, cancel, and completion delivery.

tracksBackgroundTaskSupervisor stores subagent records, workflow records, and command-session command records keyed by their typed ids.
registersdispatch_envelope_async() registers the current daemon asyncio task by invocation id, agent id, op, and background flag before invoking the handler.
cancelsSubagent cancellation targets the engine task. command-session cancellation calls the typed daemon command-session cancel API through api.v1.command.cancel or isolated-workspace drain.
heartbeatsThe supervisor periodically polls command-session completions and keeps lifecycle state current for isolated-workspace gates and automatic notifications.
gatesenter_isolated_workspace() checks both local background count and daemon command_session_count before opening private scratch.

The daemon exposes api.v1.cancel, api.v1.heartbeat, and api.v1.inflight_count for invocation registry control, while api.v1.command_session_count reads the eos-command-session runtime registry through the daemon adapter. Foreground requests do not block isolated entry through the invocation registry path; live command sessions are counted through the command-session-specific path.

Failure modes

  • No default provider registered: bootstrap did not run before provider-control or raw-exec use.
  • Unknown or changed provider value after bootstrap: first provider wins until process restart.
  • Daemon connect failure: host respawns daemon, checks readiness, and retries; repeated failure surfaces transport error.
  • Outside-workspace file path: direct file fast path returns None and daemon falls back to overlay.
  • Stale background invocation: daemon TTL reaper cancels background tasks that miss heartbeats.
  • Isolated entry with running sandbox background work: host returns ephemeral_jobs_in_flight or command_session_count_unavailable.
