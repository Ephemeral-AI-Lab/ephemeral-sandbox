This page covers the model-provider package used by the agent query loop. The source package is agent-core/crates/eos-llm-client/src. It is separate from agent-core/crates/eos-sandbox-host/src/provider.rs, which owns Docker sandbox lifecycle.

Package Boundary #

  • providers owns the LLM API client contract: request dataclasses, stream event dataclasses, error hierarchy, auth strategies, and client factory.
  • engine.agent.factory.spawn_agent() constructs the provider client through make_api_client() unless a test or caller supplies an external SupportsStreamingMessages client.
  • engine.query.loop consumes only the protocol and event types. It does not know whether the stream came from Anthropic-native API mode, Anthropic coding-plan mode, Codex Responses API mode, or a fake test client.
  • sandbox.provider is a different provider axis: container/sandbox providers. Use Sandbox Tool Dispatch Path for that boundary.

Active Model Resolution #

The active LLM model is DB-backed and seeded from the merged models config section at runtime startup. ModelRegistration stores key, label, class_path, JSON kwargs, active flag, and timestamps. Request And Stream Contract #

MessageRequest is the engine-to-provider boundary. It carries the model id, provider-safe conversation messages, system prompt, max tokens, tool schemas, and optional tool choice. The provider client yields StreamEvent values back to the query loop.

Request input
build_query_run_request() supplies sanitized messages from build_provider_messages(), the run system prompt, registered tool schemas, and prompt-report recording.
Text and thinking
AssistantTextDeltaEvent and ThinkingDeltaEvent map directly to user-visible stream deltas.
Tool deltas
ToolUseDeltaEvent is the early-execution seam. The query loop can start foreground tools before the final assistant message when deferral policy permits it.
Completion
AssistantMessageCompleteEvent is required before final dispatch. It carries the canonical assistant Message, UsageSnapshot, and provider stop reason.

Anthropic Native Client #

AnthropicClient uses the official async Anthropic SDK. Its main architecture role is translating Anthropic content-block streams into the provider-neutral events above.

serializesRequest messages become provider API params through Message.to_api_param(). Tool schemas drop output_schema because the Anthropic API does not accept it in tool definitions.
streamsText and thinking deltas are yielded immediately. Tool-use input JSON is buffered until the Anthropic content block stops, then emitted as ToolUseDeltaEvent.
reassemblesAfter the SDK stream completes, stream.get_final_message() is converted back into the canonical assistant message and usage snapshot.
retriesTransient API failures are retried only before any stream event has been emitted. Once text/tool events have escaped to the engine, failures are translated and raised rather than replaying duplicate deltas.

Coding Plan Clients #

Coding-plan mode is selected by DB class_path, not by the query loop. The package currently contains two plan-mode clients.

Anthropic plan mode

AnthropicPlanClient subclasses AnthropicClient, reads the Claude Code OAuth token from the macOS Keychain entry Claude Code-credentials, and prepends the Anthropic-required Claude Code identity system block.

Codex Responses mode

CodexResponsesClient reads ~/.codex/auth.json, extracts ChatGPT-Account-Id from the JWT, builds a flat Responses-API function-tool envelope, and translates SSE events into provider-neutral stream events.

Mode notice

When make_api_client() resolves a providers.clients.coding_plan.* class, it prints a short [coding-plan-mode] notice before construction so credential failures still show the intended route.

Errors Retry And Auth #

typed failuresAuthenticationFailure, RateLimitFailure, and RequestFailure share EphemeralOSApiError with optional status_code and request_id.
auth strategiesAPI-key mode can use API key or bearer-token auth depending on base_url. Claude coding-plan mode uses a refreshable Keychain OAuth strategy.
refresh-on-401Anthropic and Codex plan-mode clients both retry once after refreshing credentials when an initial 401 occurs.
coding-plan logsPlan-mode failures emit coding_plan_mode_error log entries categorized by auth, rate limit, server, content-filter, model rejection, schema rejection, or Cloudflare challenge paths.
config caveatconfig.sections.providers.RetryConfig defines provider retry defaults, but the current Anthropic client has local retry constants. Treat config and client behavior as separate evidence until unified.

Tests And Failures #

Provider client tests

agent-core/crates/eos-llm-client/tests covers routing, class-path dispatch, Anthropic streaming, Anthropic plan mode, Codex request headers/body, Codex event translation, JWT account extraction, credential refresh, and plan-mode error logs.

Config tests

test_model_store.py and test_model_config_class_path_thread.py cover active model row behavior, environment placeholder resolution, and class-path threading into active kwargs.

Query-loop adjacency

test_provider_history.py covers sanitation of the message history copied into MessageRequest, which is the last engine-owned step before provider streaming.
