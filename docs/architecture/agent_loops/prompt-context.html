This page traces how a TaskCenter main agent gets its instruction prompt, role profile, terminal warnings, context XML, task guidance, and optional skill row. Current verdict: prompt assembly is split intentionally. Runtime prompt construction belongs to the engine, profile parsing belongs to agents.definition, and TaskCenter launch context belongs to AgentEntryComposer plus context_engine.

  • The runtime system prompt is instruction-only: central settings plus fast-mode text, then profile body, then a terminal warning section derived from the actual tool registry.
  • Main profile Markdown frontmatter is the launch contract: tool surface, terminals, notifications, role kind, recipe id, and optional skill path.
  • AgentEntryComposer resolves effective terminals, builds and persists a ContextPacket, renders <context>, adds <Task Guidance>, and appends the skill row.
  • ContextEngine is a packet builder. It does not own lifecycle, retries, iteration closure, or provider execution.

Runtime And Profiles #

build_runtime_system_prompt() only reads settings and returns the base runtime instruction text, with an extra fast-mode section when configured. engine.agent.factory._build_agent_system_prompt() appends the loaded AgentDefinition.system_prompt; it does not inject identity, declared skills, role boundary, or the main-role contract at runtime. Those old runtime-injection claims are stale.

Main profile loading is file-backed. load_agents_tree() parses Markdown frontmatter, defaults name from the filename, requires agent_kind, validates the optional relative skill path, and prepends _main_role_contract.md to every non-private file directly under agents/profile/main/. The resulting AgentDefinition carries the profile body as system_prompt, plus the run surface and context recipe declared in frontmatter.

Runtime base
prompt.runtime_prompt.build_runtime_system_prompt(settings, cwd) returns settings.system_prompt plus fast-mode guidance only.
Profile body
agents.definition.loader._load_agent_files() parses frontmatter and stores the Markdown body as AgentDefinition.system_prompt, with the main-role contract prepended at load time.
Frontmatter contract
Main profiles declare agent_kind, allowed_tools, terminals, notification_triggers, context_recipe, and sometimes skill.
Terminal warning
_finalize_tool_registry_and_prompt() derives terminal tool names from registered tool classes and appends build_termination_condition_prompt().
loadsagents/profile/main/planner.md declares context_recipe: planner, submit_plan_closes_goal, and submit_plan_defers_goal; launch routing later filters that full surface.
loadsexecutor.md and generator_verifier.md both use the generator recipe, while evaluator.md uses the evaluator recipe.
teststest_agent_system_prompt.py protects the current split: runtime prompt plus loaded agent body only, no runtime reinjection of the main-role base.

Composer Rows #

AgentEntryComposer.compose() is the TaskCenter launch seam. It does not build the system prompt row; the engine does that in spawn_agent(). It does build the launch user-message rows from the effective profile and ContextPacket: <context>, optional <Task Guidance>, and optional skill message.

Row 4build_skill_message()Load skill: ..., stripped skill body, and a byte-equal terminal selection block when terminals exist.

EphemeralAttemptAgentLauncher._run_launch() maps those rows into the runner call. When both row 3 and row 4 exist, row 4 becomes the runner prompt and rows 2-3 become initial_messages. When only row 3 exists, row 3 is the runner prompt and row 2 is the initial message. With no row 3, the context is the prompt.

Effective terminals Planner and executor frontmatter stays broad. Launch routing filters it by nested workflow depth: nested planners keep only submit_plan_closes_goal, and nested executors lose submit_execution_handoff. The registered profile is not mutated.

Deps And Scope #

ContextScope is identity only: workflow_id, optional iteration_id, optional attempt_id, and optional task_id. It deliberately carries no store handles. ContextEngineDeps is the frozen dependency bundle with goal, iteration, attempt, and task stores, plus optional context_packet_store.

ContextScope.for_planner()

Requires goal, iteration, and attempt ids. Used by AgentLaunchFactory.for_planner().

ContextScope.for_generator()

Requires goal, iteration, attempt, and task ids. Used for executor and verifier generator tasks.

ContextScope.for_evaluator()

Passes goal, iteration, and attempt ids, but the evaluator recipe itself requires only the attempt id and derives the iteration from it.

RecipeRegistry

Process-global registry keyed by recipe id. Built-ins are discovered by walking recipes/*_RECIPE once at startup.

validatesContextEngine.build() looks up the recipe, asks ContextScope.assert_fields() to enforce required ids, and then calls the recipe builder.
serializesContextPacket stores target role, canonical refs, typed blocks, metadata, and source ids. Renderers and guidance operate on this packet, not on stores.
registersregister_builtin_recipes() imports recipe modules and registers each module-level *_RECIPE.

Role Recipes #

The built-in recipes are role-specific because the state they expose is TaskCenter-shaped. The reusable substrate is the packet, renderer, registry, scope, and summary helpers. The recipe bodies still read goal, iteration, attempt, and task rows through ContextEngineDeps.

RecipeRequired scopeRendered contextImportant omission
plannerworkflow_id, iteration_id, attempt_id<goal>, prior successful iterations, current <iteration_goal>, and failed prior attempts for retry planning.No generator dependency results for a future DAG; the planner sees only lifecycle context.
generatorworkflow_id, attempt_id, task_idStandalone <plan_spec>, flat <dependency id="..."> siblings, and final <assigned_task task_id="...">.No <deferred_goal_for_next_iteration>; that is planner/evaluator lifecycle context, not executor work input.
evaluatorattempt_idThe current attempt as flat top-level blocks: <plan_spec>, per-task <task> outcomes (summary-only), and <evaluation_criteria>.No goal/iteration frame, prior attempts, or <deferred_goal_for_next_iteration>; the evaluator judges only this attempt against its criteria.

recipes.iterations.goal_iteration_blocks() emits the planner's goal and iteration scaffold (the evaluator no longer uses it). Iteration 1 uses the literal body (identical to &lt;goal&gt;) instead of repeating the user request. Prior iterations require closed successful plan_spec and task_summary; missing data raises ContextEngineError rather than producing partial context.

recipes.attempts owns two shapes. failed_attempt_blocks() (planner retry context) emits the nested <attempt status="prior" verdict="fail"> body with <plan_spec>, optional <deferred_goal_for_next_iteration>, <status_summary>, per-task <task> summaries, <evaluation_criteria>, and evaluator verdict children when available. current_attempt_flat_blocks() (evaluator) emits the current attempt as flat top-level <plan_spec> + <task> + <evaluation_criteria> blocks — no <attempt> wrapper and no deferred goal.

Renderer And Guidance #

XmlPromptRenderer is a pure function over ContextPacket. It preserves block text byte-for-byte, renders packet order rather than priority order, wraps blocks with default or metadata-overridden tags, groups consecutive blocks with the same group_id, and applies an approximate token-budget compression policy when packet metadata sets token_budget.

Tag choice
metadata["tag"] overrides the default kind-to-tag map. Grouped blocks use group_tag, group_attrs, child_tag, and child attrs.
Compression
Required and high-priority blocks are preserved. Low-priority blocks truncate first, then medium-priority blocks, using source-aware one-line replacements.
Structural closer guard
The renderer scans emitted tag closers and rejects hostile block text that contains a closer such as </goal>. Attempt bodies with pre_rendered_xml opt out and sanitize their embedded user fields inside the recipe.
Context wrapper guard
AgentEntryComposer._wrap_context() rejects rendered output containing </context> before adding the outer envelope.

render_context_outline() walks the same packet metadata the renderer uses. It does not parse rendered XML. It emits a compact outline for row 3, collapses consecutive siblings with the same descriptor, recurses through <iteration> children, and intentionally does not recurse into <attempt> bodies. Labels come from tag_dictionary.py, where only status and verdict are semantic attributes; identity attributes such as iteration_no, attempt_no, task_id, and id are hidden from the outline.

buildscontext_engine.task_guidance.build_task_guidance() returns exactly two sections: What's in context from the packet outline and What to do from ROLE_DIRECTIVES.
wrapsskill_message._wrap_task_guidance() adds the <Task Guidance> envelope and terminal selection block; build_skill_message() appends the same terminal block to skill row 4.
does-not-ownOperational heuristics are not in the task-guidance prose. They live in the role profile body and skill files.

Retry And Deferred Context #

Retry and continuation are TaskCenter lifecycle concepts. AttemptOrchestrator validates planner submissions, persists plan_spec, evaluation_criteria, and optional deferred_goal_for_next_iteration, then creates generator task rows. The context engine later reads that persisted state; it does not decide whether to retry, continue, or close the goal.

[attempt closes]
        |
        +--> [evaluation failed]
        |       |
        |       v
        |   [retry recipe]
        |       |  prior patch, failure report, allowed next action
        |       v
        |   [new attempt context]
        |
        +--> [iteration deferred / needs continuation]
                |
                v
            [continuation recipe]
                |  durable state + deferred reason + remaining work
                v
            [resumed attempt context]
Retry evidenceFailed attempts are context inputs for the next planner attempt in the same iteration.
Continuation gateA deferring plan is possible only when launch routing leaves submit_plan_defers_goal available.
Lifecycle boundaryThe orchestrator and stage advancement create the next iteration; the context engine only reads the persisted result.
Current contract The planner profile explicitly tells the model that deferred_goal_for_next_iteration is the next iteration's whole scope, not a backlog dump. The generator recipe intentionally withholds that field so executors stay focused on <assigned_task>.
