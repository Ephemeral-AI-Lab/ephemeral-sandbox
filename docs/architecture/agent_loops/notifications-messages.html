The message package owns the transcript and output formats that the agent loop mutates. The notification package owns engine-generated reminders that appear both as stream events and provider-visible user-message blocks.

Message Model #

Messagerole plus content blocks
Assistant blockstext, thinking, tool_use
User blockstext, tool_result, system notification
Internal markeris_terminal stays off provider wire
text and thinkingTextBlock and ThinkingBlock model assistant/user text. Thinking blocks are excluded when a conversation message is converted back to provider parameters.
tool requestsToolUseBlock carries the provider tool id, name, and JSON input. The query loop reads these blocks from the final assistant message to decide dispatch.
tool resultsToolResultBlock is appended in a user message for non-terminal tool results. Its metadata and is_terminal fields are engine-local and are not serialized to providers.
system notificationsSystemNotificationBlock is a first-class local block, but provider serialization flattens it into a text block wrapped in <system-reminder> tags.

Provider History #

The query loop keeps a canonical in-memory transcript, but provider calls receive a sanitized copy. build_query_run_request() calls build_provider_messages(), records prompt reports, decorates background-capable tool schemas, and records the initial prompt rows once for message artifacts.

Notification Runtime #

SystemNotificationService is a run-scoped sink. Runtime code, hooks, and rules call notify_system(); the service stores a provider-visible SystemNotificationBlock and also queues a stream-side SystemNotification event.

Rule timing
_run_query_loop() evaluates notification rules at the top of each provider turn, before the next request is built.
Visibility
Pending reminder blocks are appended as a user message so the next provider request sees them. Stream events are flushed separately so the user interface can render them as system messages.
Ordering
dispatch_rules() evaluates rules in list order. A later rule in the same pass cannot observe a reminder emitted by an earlier rule because the caller drains notifications after the pass.
Default rules
The agent factory attaches one-shot tool-call budget warnings at 75%, 100%, and 125% of tool_call_limit, plus the repeating terminal-call reminder, to each spawned agent so runs receive in-band nudges before the 150% hard ceiling.

Stream Output #

Stream events are the live output surface. run_query() stamps emitted dataclass events with agent_name and run_id when those fields are present, so multi-agent logs can group interleaved work by lane.

assistant deltasThinkingDeltaEvent and AssistantTextDeltaEvent stream partial provider output. AssistantMessageCompleteEvent carries the canonical completed assistant message and usage.
tool eventsToolExecutionStartedEvent, ToolExecutionProgressEvent, ToolExecutionCompletedEvent, and ToolExecutionCancelledEvent describe foreground and deferred tool execution.
background startBackgroundTaskStartedEvent carries the short task alias, tool name, and clean input for background launch display.
printer outputMultiAgentEventPrinter buffers text and thinking per (agent_name, run_id) lane, formats shell structured errors, and prints background start details for subagents.

Message JSONL #

AgentMessageJsonlRecorder writes completed conversation rows to append-only JSONL. It records the system prompt explicitly even though the provider API receives it outside the messages array, then writes initial user rows, assistant completions, and tool-result user rows.

  • Rows contain role, serialized content, and metadata with seq, agent_name, run_id, and optional tool fields.
  • Thinking and text deltas are buffered until AssistantMessageCompleteEvent. If the complete message already contains the relevant block, the buffer is dropped.
  • Tool completions are recorded as user messages with ToolResultBlock content, preserving metadata and is_terminal for local artifact analysis.
  • For TaskCenter's multi-row launch shape, seeded initial messages are written between the system row and the final spawn prompt row, preserving system plus context plus task-guidance or skill rows.

Observability And Persistence #

The live stream is only one output surface. The agent loop also feeds audit events, prompt reports, persisted run rows, and message JSONL artifacts.

engine auditengine.audit.stream.audit_events_from_stream_event() translates tool-started and tool-completed stream events into shared AuditEvent rows with redacted input shape, content digests, output shape, terminal marker, node identity, and TaskCenter/sandbox metadata.
prompt reportsPromptReportRecorder appends sequenced llm_request, assistant, and tool_results events when prompt_report_messages_path is present in tool metadata.
agent run rowsAgentRunTracker creates an agent_runs row for TaskCenter-backed runs when the store is ready, then finalizes it with message history, terminal result, token count, and error text.
message artifactAgentMessageJsonlRecorder remains the conversation-level artifact writer. It is distinct from prompt reports, which are provider-turn diagnostics.

Failure Modes And Tests #

Malformed tool history

Provider-history preparation drops unmatched stale tool-use or tool-result blocks from the provider copy while leaving the canonical transcript intact.

Notification drift

Rule tests cover dispatch order, one-shot rule deduplication, recurring rule delivery, and terminal-call reminders.

Artifact gaps

Recorder tests cover initial row writing, assistant completion recording, tool-result recording, delta buffering, and lane-specific flush behavior.

Terminal transcript pairing

Resource-limit tests cover keeping provider history well-formed when the loop stops after tool overshoot.
