Launch and Alias

Background execution is a query-loop dispatch mode, not a persistent shell session or a tool metadata policy. The engine hard-codes the background-manager surfaces in engine.background.policy: subagent launches are supervisor tasks and command sessions expose typed sessions.

The old generic model-facing background tools are retired. The supervisor remains internal, while typed work exposes typed ids: subagents use subagent_session_id and command sessions use command_session_id.

Subagent launchrun_subagent is routed to the supervisor by engine policy. The supervisor record id and the model-facing control id are both subagent_session_id.

Validation failureInvalid deferred input returns a normal tool-result error and no supervisor record. Unsupported tools produce a rejection completion event and no alias.

runningCreated at launch with the initial progress line and updated while the asyncio task is active.

deliveredSet by the matching delivery path: subagents use typed completion/progress delivery, workflows use typed workflow status/cancel delivery, and command sessions use typed command-session completion collection.
Provider historyreduce_background_task_history() keeps provider input compact by retaining only the latest meaningful background snapshots, preferring terminal snapshots over stale running snapshots.
Race handlingIf a natural completion and cancellation race, the latch keeps the highest-precedence terminal outcome so a real completed result is not overwritten by a late cancel acknowledgement.
Parent exitWhen a parent run submits a terminal while a subagent is still running, the loop calls terminate_for_parent_exit(reason="non_cancellation_tool_request"); the subagent is reported as terminated with a typed notification and background audit cancel reason.

Check Wait Cancel Tools

The generic model-facing background management tools are retired. Typed surfaces operate on the supervisor stored in tool metadata for the current run.

Retired generic toolscheck_background_task_result, wait_background_tasks, and cancel_background_task are no longer synthesized for agents or catalog output.
check_subagent_progressRequires subagent_session_id and returns the latest child-agent message snapshot while running, or the terminal tool output once run_subagent has received a terminal result.
cancel_subagentRequires subagent_session_id and requests the subagent early-stop path.
Command-session controlswrite_stdin operates on command_session_id for both input and polling; daemon cancel/collect/count ops stay internal.
Workflow controlscheck_workflow_status and cancel_workflow operate on the delegated workflow handle returned by delegate_workflow.

Sandbox Invocation Control

Sandbox control is no longer exposed through generic model-facing background tools. The engine launcher only backgrounds subagents; command sessions are created by exec_command through the daemon adapter, which receives a command_session_id from eos-command-session and registers it with the supervisor.

Engine tasksubagent_session_id or command_session_id
Sandbox APIapi.v1.exec_command
Daemon registrytracks command session id, agent id, command, status
Pollwrite_stdin collects output
Cancelapi.v1.command.cancel targets the command session
Exit gateisolated workspace drains active command sessions
Transport contractSandboxTransport.call() still carries invocation metadata for normal requests, but command-session lifecycle controls address the daemon-owned command_session_id directly.
Daemon dispatcherdispatch_envelope_async() registers regular in-flight requests, while command session handlers manage long-running command state after the initial exec_command call returns.
Workspace routeOnce inside the daemon, command-session state does not choose the workspace path. The workspace dispatcher still routes file fast paths through LayerStack/OCC, isolated workspace handles when open, or the overlay pipeline for command/search/plugin-style operations.

Daemon Ring Audit Events

Background supervision emits background_tool.* events into the daemon audit ring. The schema is part of the daemon audit v1 event families, not a separate log file.

Normal lanebackground_tool.started, completed, failed, cancelled, and delivered are normal-lane events with a background_tool payload.
Sample lanebackground_tool.heartbeat is sample-lane audit traffic and includes current uptime for long-running sandbox-bound work.
PayloadBackgroundToolSection carries background_task_id, task kind, tool name, agent id, status, exit code, duration, error kind, cancel reason, and delivery latency when available.
Pull surfaceConsumers inspect the ring through api.audit.pull, api.audit.snapshot, and guarded api.audit.reset_floor operations.

Tests and Failure Modes

Launch and dispatch

test_tool_execution.py covers background launch input cleaning, hook execution, failure propagation, and conversation-message exposure to background prehooks.

Supervisor lifecycle

test_background_tasks.py, test_background_terminal_latch.py, and test_subagent_mock_loop.py cover typed id generation, progress buffering, typed control output, command-session cancellation, terminal status precedence, and parent terminal abandonment of active subagents.

Audit visibility

test_background_task_emitters.py covers started, completed, failed, cancelled, delivered events and verifies the emitter does not introduce a new thread per launch.

Daemon control

In-flight registry tests cover invocation cancel, heartbeat refresh, background-only TTL reaping, and daemon handler routing for api.v1.cancel, api.v1.heartbeat, and api.v1.inflight_count; command-session lifecycle tests cover api.v1.command_session_count.
