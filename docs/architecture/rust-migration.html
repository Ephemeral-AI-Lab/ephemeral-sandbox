The current architecture wiki should treat agent-core/ and sandbox/ as the canonical runtime implementation. The legacy Python implementation remains useful as parity evidence and historical comparison, but the only active Python-owned architecture module is backend/src/test_runner: the test and benchmark harness that validates the Rust agent-core and sandbox paths.

Migration Verdict #

  • Runtime owners are Rust. Request bootstrap, root-agent execution, workflow lifecycle, context packets, engine query loop, tool framework, sandbox host/API, daemon dispatch, command sessions, LayerStack/OCC/overlay, isolated workspaces, plugin PPC, and PTY allocation live in agent-core/crates/* and sandbox/crates/*.
  • Python is comparison material except the runner. Old backend/src/workflow, backend/src/engine, backend/src/tools, and backend/src/sandbox paths explain the source behavior the Rust code ports, but docs should not present them as current owners.
  • Rust mostly follows Python's workflows. The shape is intentionally preserved: root Task first, optional delegate_workflow, Workflow -> Iteration -> Attempt, planner-authored generator/reducer DAG, terminal submissions, OCC publish, explicit isolated enter/exit, and background/session controls.
  • Rust removes Python redundancy. Stringly op dispatch, compatibility-only fields, provider branches, runtime bundle shims, hidden close-time parent mutation, and split evaluator/verifier concepts are collapsed into typed IDs, enums, DTOs, crate ports, Docker-only sandbox host paths, reducer plan tasks, and explicit state transitions.
Doc migration rule When an architecture page still cites backend/src/* for runtime ownership, refresh the smallest affected section to a Rust evidence path. Keep Python references only in a labeled Python parity / historical comparison row, or under Test Runner.

Workflow Parity #

  1. Request starts. Rust runtime provisions the sandbox, inserts a request row, mints a root task, and runs the root agent directly through run_ephemeral_agent. This preserves Python's Task-first request model and removes the old synthetic root workflow claim.
  2. Root may delegate. delegate_workflow is non-terminal. It creates delegated workflow state and returns a handle; the parent remains running and must inspect/cancel/finish explicitly.
  3. Workflow decomposes. WorkflowLifecycle creates iterations, handles retry/deferral, and closes the workflow without mutating the parent task.
  4. Attempt plans and runs. AttemptOrchestrator materializes a planner-authored generator/reducer DAG, then the RUN stage schedules ready tasks to quiescence. Reducers are ordinary plan tasks and the exit gate.
  5. Terminal tools persist evidence. Successful terminal tools are stamped by eos-tools, the engine stops the loop, and workflow/runtime code consumes the structured terminal result as persisted state input.
Root context
Root receives the request prompt, not a workflow ContextEngine packet. Workflow ContextEngine remains for delegated planner/generator/reducer launches only.
Parent close
Delegated workflow closure does not auto-close or mutate the parent task. The parent agent is responsible for its own terminal submission.
Reducer naming
Older verifier/evaluator language is stale. Judgement is a reducer plan task, and attempt stages are PLAN -> RUN -> CLOSED.
Root outcome
Rust root submit/failure records terminal state on the root task and finishes the request, but leaves typed execution outcomes empty because root is not a generator/reducer execution role. Python wrote a root-role outcome object.
Background semantics
Subagent/workflow/command-session controls are typed surfaces. Generic background polling is not the model-facing contract.

Performance And Resource Contracts #

  • Space model: Rust preserves the Python spec that lower layers are shared by lease. Runtime disk cost is O(repo) + O(N * changed_bytes), not O(N * repo).
  • Fast file APIs: read_file, write_file, and edit_file use daemon-owned LayerStack/OCC fast paths when a workspace binding exists; shell/search/plugin work uses overlay capture because the write set is not known up front.
  • Conflict control: OCC keeps direct/gated routing, stale-base rejection, publish serialization, conflict projection, and no silent overwrite. Rust should keep load tests anchored to the live E2E concurrency reports under backend/tests/live_e2e_test/sandbox.
  • Overlay guard: The operational auto-squash target remains AUTO_SQUASH_MAX_DEPTH = 100; docs should not reintroduce a runtime OVL_MAX_STACK_GUARD claim.
  • Shell latency: Rust command sessions should keep the non-login Bash contract; bash -lc is a latency tax and should appear only where a test explicitly requires login-shell behavior.
  • Live runner quota: Current Python runner config sets runner.sandbox_quota = 3 and runner.live_e2e.concurrent_sandbox_runners = 3. Treat that as a harness cap for validation lanes, not a Rust runtime concurrency ceiling.

Remaining Python Surface #

backend/src/test_runner remains Python by design today. It is the harness for deterministic mocked-agent scenarios, real-agent/freeform runs, SWE-EVO benchmark setup/evaluation, live E2E orchestration, run reports, and artifact collection. There is no implemented Rust test_runner crate in this checkout; docs/plans/test_runner_rust_SPEC.md is a forward-looking spec, not current architecture. The Python runner must be documented as a consumer and validator of the Rust runtime, not as the owner of runtime behavior.

runner ownsrun_pipeline, RunConfig, per-test store bundles, scenario lifecycle hooks, mock event sources, SWE-EVO setup/eval, run labels, and report schemas.
runner validatesRust request/root-task flow, delegated workflow handles, engine/tool terminal behavior, Rust sandbox command sessions, plugin PPC, OCC, LayerStack, overlay, and isolated workspace lifecycle.
runner must not ownRequest/Task/Workflow lifecycle policy, engine loop semantics, tool framework policy, daemon routing, provider lifecycle, LayerStack/OCC/overlay internals, or plugin PPC state.
