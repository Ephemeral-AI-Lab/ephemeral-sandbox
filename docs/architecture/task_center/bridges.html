This page maps how TaskCenter crosses into the engine loop, tool framework, sandbox API, and live runner artifacts. Current verdict: TaskCenter owns persisted Workflow -> Iteration -> Attempt state plus task rows; engine owns provider turns and tool execution; sandbox owns workspace execution; the runner records the live artifact tree around those seams.

This boundary map is deliberately not a global orchestrator. TaskCenter starts a run, prepares one sandbox binding, composes launch messages, schedules attempt-scoped agents, and receives terminal submissions through persisted state. The engine loop remains the enforcement layer for terminal tools, and sandbox operations still pass through tools/sandbox, sandbox.api, and daemon workspace routing.

Entry And Sandbox Provisioning #

start_task_center_run() is the entry seam for top-level TaskCenter work. It receives stores, runtime config, prompt, optional sandbox id, event callback, optional runner, optional context packet store, and optional sandbox provisioner. It does not run an agent directly; it constructs the run shell and lets Workflow -> Iteration -> Attempt startup launch the initial planner.

Bootstrap failure
If initial workflow start fails after run creation, _finish_run_if_open() marks the run failed before re-raising.

Launch To Engine Path #

The attempt launcher is the TaskCenter-to-engine adapter. It receives an AgentLaunch record from AgentLaunchFactory, attaches TaskCenter runtime metadata, chooses the launch message shape, and awaits the runner. If no runner is injected, it imports engine.api.run_ephemeral_agent lazily and uses the real engine path.

message shapeIf both task guidance and skill exist, the skill is the spawn prompt and the context plus task guidance are seeded through initial_messages. If only task guidance exists, guidance is the prompt and context is the one seeded initial message. Without task guidance, context is the prompt and no initial messages are seeded.
runner contracttest_runner_kwargs_contract.py freezes the runner keyword set: agent_def, sandbox_id, persist_agent_run, task_id, on_event, extra_tool_metadata, and initial_messages.
exhaustion pathIf the runner returns None, crashes, fails, or completes while the TaskCenter task is still RUNNING, the launcher reports a planner, generator, or evaluator failure through the attempt orchestrator.

Tool And Sandbox Routing #

The engine does not special-case TaskCenter sandbox work. TaskCenter metadata enters the normal ExecutionMetadata object, the tool framework exposes it through ToolExecutionContextService, and sandbox tools build typed sandbox.api requests with caller identity.

[agent sandbox tool call]
        |
        v
[engine tool-call dispatch]
        |
        v
[tools sandbox registry]
        |
        v
[sandbox daemon client]
        |
        v
[workspace_tool_dispatch]
        |
        +--> [direct LayerStack/OCC fast path]
        +--> [ephemeral overlay pipeline]
        +--> [isolated workspace pipeline]
metadataThe spawned agent receives TaskCenter launch metadata; the query loop preserves identifiers and conversation messages in each tool execution context.
registrySandbox tools carry SANDBOX_CONTEXT, letting runtime preparation attach provider context and background dispatch identify sandbox-bound work.
callersandbox_caller_from_tool_context() builds typed caller identity that includes TaskCenter ids and the per-call tool id.
daemon opssandbox.api.tool sends typed file, shell, grep, and glob operations through call_sandbox_daemon().
routingdispatch_workspace_tool_call() tries active isolated handles, direct LayerStack/OCC file paths, then overlay execution for shell/search and fallback paths.
Workspace boundary TaskCenter launches agents outside the sandbox. Workspace reads, writes, shell commands, search, isolated mode, overlay publish, and OCC conflict handling still live behind provider-backed sandbox APIs and daemon workspace routing.

Runner And Live Artifacts #

task_center_runner.core.engine.run_pipeline() is the live/mock/benchmark harness entrypoint around TaskCenter. It provisions the sandbox, opens an audit bus, starts AuditRecorder, injects a runner if the mode has one, starts TaskCenter, waits for the launcher to go idle, releases the sandbox, and writes a performance report task.

Run directory
The default path is audit_dir / run_label / utc_self_id. Reports store the run dir on PipelineReport.run_dir.
Stream audit adapter
stream_bridge() converts ToolExecutionStartedEvent and ToolExecutionCompletedEvent into audit events, preserving tool id, metadata, error state, and is_terminal.
State snapshots
AuditRecorder listens to Workflow, Iteration, Attempt, task, and agent-run ORM events, then writes workflow.json, iteration.json, attempt.json, task.json, run.json, and metrics.json.
Message capture
Primary role task directories get message.jsonl. Initial rows are recorded once from system prompt, seeded initial_messages, and the spawn prompt.
Sandbox events
Sandbox audit events are mirrored to sandbox_events.jsonl, either from daemon event pulling or stream-derived sandbox event projection.
Initial-message regression
test_initial_messages_capture.py asserts captured message.jsonl rows include <context>, <Task Guidance>, skill rows, and terminal-selection catalog metadata for planner/executor/evaluator launches.
  • Real engine paths write initial messages via engine.query.request._record_initial_messages_once() and the global agent-run recorder registry.
  • Mock runner paths call AgentMessageJsonlRecorder.record_initial_messages() directly with launch metadata so live scenario assertions inspect the same row shape.
  • The artifact tree is diagnostic evidence, not state authority. TaskCenter Workflow -> Iteration -> Attempt state remains in the stores and is mirrored to files by listeners.

Boundary Rules And Failure Modes #

No global orchestratorAttemptOrchestrator is per Attempt. Entry bootstrap wires registries and stores, but mutation still flows through Workflow, Iteration, Attempt, task, and terminal-submission state.
No peer-to-peer handoffsubmit_execution_handoff starts a persisted child goal. The parent generator moves to WAITING_WORKFLOW and resumes only when a close report routes back through WorkflowClosureReportRouter and AttemptOrchestrator.apply_workflow_closure_report().
Missing attempt runtimeSubmission context resolvers require ExecutionMetadata.attempt_runtime, a task id, a persisted task row, matching attempt metadata, and a registered orchestrator. Missing pieces return terminal tool errors instead of guessing state.
Runner exhaustionWhen an agent run ends without moving its TaskCenter task off RUNNING, the launcher synthesizes a planner, generator, or evaluator failure submission so the Attempt remains owned by TaskCenter state code.
Terminal refusalThe engine may return no terminal result for text-only refusal, resource limit, or crash. TaskCenter treats that as runner exhaustion unless a terminal submission already changed the task state.
Workspace routing surpriseSandbox calls can route isolated, direct, or overlay depending on active isolated handles, file verb, workspace binding, path shape, and daemon availability. TaskCenter should not assume a file mutation bypasses OCC.

Launcher contract

test_runner_kwargs_contract.py catches kwarg drift between EphemeralAttemptAgentLauncher and runner delegates. Launcher retry/exhaustion tests pin no deleted terminal-retry kwarg, initial-message shapes, and fallback behavior.

Artifact shape

test_initial_messages_capture.py live-checks message.jsonl launch rows and terminal catalog rendering. Correctness and full-case tests assert audit trees and message captures exist under run dirs.
