TaskCenter is the persisted multi-agent control plane for complex coding work. Its most important model is Workflow -> Iteration -> Attempt: a Workflow owns the objective, an Iteration owns one bounded planning slice, and an Attempt owns one planner -> generator DAG -> evaluator try. Engine, tools, and sandbox provide execution surfaces around that state model.

Scope And Verdict #

  • The durable state model is Workflow -> Iteration -> Attempt. A Workflow has ordered Iterations; an Iteration owns sequential retry Attempts; an Attempt owns one planner task, a persisted generator DAG, and one evaluator task.
  • TaskCenter does not add peer-to-peer agent communication. Delegation is persisted as a task-origin child Workflow; the parent generator waits in WAITING_WORKFLOW and later resumes through a closure report.
  • The AttemptOrchestrator is per-Attempt machinery, not a global orchestrator. It persists the planner contract, generator rows, generator outcomes, evaluator outcomes, and Attempt close.
  • The Context Engine builds and renders role-specific packets from stores. It does not decide retry, continuation, goal closure, launch policy, or sandbox behavior.
  • Terminal tools are the agent-facing state boundary. Successful terminal tool calls are stamped is_terminal=true by the tool framework, and their payloads mutate TaskCenter through TaskCenter services.

Stale Claim Map #

  • The older standalone TaskCenter HTML still names GoalHandler, IterationManager, IterationClosureRouter, and IterationFactory. Current source names are WorkflowLifecycle and IterationAttemptCoordinator.
  • The older HTML says child Workflow closure transitions the parent task back to RUNNING. Current code routes the closure report into AttemptOrchestrator.apply_workflow_closure_report(), which marks the parent generator task DONE on success or FAILED on failure.
  • The older HTML says executor terminal selection is no longer depth-gated. Current TerminalToolRouter removes submit_execution_handoff when nested Workflow depth is greater than one.
  • The older HTML mentions entry_executor and obsolete renderer compression behavior. Current built-in recipes are planner, generator, evaluator, and XmlPromptRenderer preserves required/high-priority blocks while truncating low then medium blocks.
