This page traces the TaskCenter context engine from launch identity to model-facing rows. Current verdict: ContextEngine builds typed packets from persisted Workflow -> Iteration -> Attempt state; it does not own policy, terminal submission handling, retries, delegated-workflow closure, or provider execution.

Scope And Verdict #

  • ContextScope carries only identity fields: goal, iteration, attempt, and optional task ids. Store handles live in ContextEngineDeps.
  • ContextEngine.build(recipe_id, scope) looks up a registered ContextRecipe, validates the required scope fields, and returns a ContextPacket.
  • AgentEntryComposer is the launch seam. It resolves an effective role profile, builds the packet, optionally persists it, renders <context>, appends <Task Guidance>, and adds the skill row when the profile declares one.
  • EphemeralAttemptAgentLauncher converts composed rows into the runner call. The context engine does not talk to model providers or mutate TaskCenter state directly.

Scope, Deps, Registry #

ContextScope.for_planner()

Requires goal, iteration, and attempt ids. It represents one attempt planner spawn.

ContextScope.for_generator()

Requires goal, iteration, attempt, and task ids. It is used for executor and verifier generator tasks.

ContextScope.for_evaluator()

Passes goal, iteration, and attempt ids, but the evaluator recipe itself requires only the attempt id and derives the iteration from it.

ContextEngineDeps

Frozen store bundle: goal, iteration, attempt, task, and optional context-packet store.

validates required idsEach ContextRecipe declares required_scope_fields. ContextEngine.build() calls scope.assert_fields(...) before invoking the builder.
keeps packets typedContextPacket is the only object that travels from recipes to renderers. Blocks carry kind, priority, text, source provenance, and renderer metadata.
discovers built-insregister_builtin_recipes() imports every submodule under context_engine/recipes and registers module-level *_RECIPE values.
validates profile referencesvalidate_agent_definitions_resolved() rejects a profile whose context_recipe is not registered, while helper profiles without recipes are allowed.
Boundary ContextScope is not a Workflow, Iteration, or Attempt object. It does not carry retry policy, dependency scheduling, terminal outcomes, model settings, or sandbox handles.

Role Recipes #

The built-in recipe set is role-shaped, not agent-instance-shaped. Planner, generator, and evaluator all read the same persisted TaskCenter state through ContextEngineDeps, but each exposes a different slice to keep the launched agent inside its role contract.

Renderer, Guidance, Skills #

XmlPromptRenderer is a pure function over a packet. It walks blocks in packet order, groups consecutive blocks with the same group_id, wraps text in XML tags, and treats priority as a token-budget compression policy rather than a display-order policy.

renders XMLDefault tags come from block kind, but recipe metadata can override standalone tags, group tags, child tags, attributes, and pre-rendered attempt bodies.
protects structureThe renderer rejects structural tag closers found inside ordinary block text. The attempts recipe opts into pre_rendered_xml and separately rejects closers in user-controlled attempt fields.
compresses lower priority firstWhen packet.metadata["token_budget"] is set, low-priority blocks truncate first, then medium-priority blocks. Required and high-priority blocks are preserved.
summarizes contextrender_context_outline() walks packet metadata instead of parsing rendered XML. tag_dictionary.py labels semantic tags and hides identity attributes from the outline.
ROLE_DIRECTIVES
Maps exact agent names to one imperative line such as Plan for <iteration_goal>., Complete <assigned_task>., or Verify the current attempt against <evaluation_criteria>.
build_task_guidance()
Returns two sections: What's in context: from the packet outline and What to do: from the role directive table.
_wrap_task_guidance()
Adds the <Task Guidance> envelope and one <terminal_tool_selection> block when the effective profile declares terminals.
build_skill_message()
Reads the profile's skill file, strips skill frontmatter, emits Load skill: <skill-folder>, wraps the body in <skill>, and appends the same terminal catalog.

Persistence And Wire Shape #

ContextPacketStore is write-once. It inserts packet ids, target role/id, canonical refs, block JSON, metadata, and source ids, and exposes get(); tests assert there is no update or set method. Task rows carry context_packet_id so later audit, helper, and retry flows can recover the frame that launched the agent.

Current row contract The engine still owns the system prompt. TaskCenter supplies user-message rows: context, task guidance, and optional skill.

Retry And Deferred Context #

Retry and continuation are persisted Workflow -> Iteration -> Attempt state that recipes read back later. AttemptOrchestrator accepts planner submissions, stores plan_spec, evaluation_criteria, generator task ids, and optional deferred_goal_for_next_iteration. The context engine never chooses whether to retry, defer, or close; it exposes the evidence appropriate to the next role.

retry evidenceA failed attempt keeps failed generator/evaluator state, and the next planner in the same iteration receives prior failed attempts ordered under the current iteration.
deferred continuationA successful deferring plan closes the current iteration and starts the next one from deferred_goal_for_next_iteration.
planner frameThe next iteration planner sees prior successful iterations as accepted plan plus summary, while the new <iteration_goal> remains authoritative.
evaluator frameThe evaluator never receives the deferred-goal text; excluding it makes penalizing intentional deferral unrepresentable. The planner still sees it in failed-prior blocks as retry context.
generator frameThe generator recipe omits deferred-goal text so executor and verifier tasks stay scoped to the assigned task and dependency outputs.
Staleness trap Do not describe the context engine as a global prompt blob or Workflow -> Iteration -> Attempt policy layer. Current source keeps policy in orchestrator/stage advancement and packet construction in recipes.
