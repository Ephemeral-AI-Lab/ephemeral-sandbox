TaskCenter terminal tools are agent-facing submission endpoints. They are normal registered tools until a successful call crosses the tool framework, where the result is stamped as terminating and the engine loop stops the agent run.

The current shape is intentionally layered: descriptors tell agents when to choose each terminal, submission tools validate and project role payloads, the framework validates output and stamps is_terminal, and the engine owns terminal exclusivity. The advisor helper uses the same terminal contract for its own child run, but its result returns to the caller instead of mutating the parent attempt directly.

  1. make_submission_tools() registers planner, executor, verifier, evaluator, advisor, and explorer submission tools as one framework tool set.
  2. Each terminal tool is decorated with is_terminal_tool=True and a concrete pydantic input model. Most use TextToolOutput, so non-error output must validate as text before the result can terminate the run.
  3. engine.query.loop derives context.terminal_tools from the active tool registry. If a run has any terminals, streamed tool calls are deferred until the final assistant message is available.
  4. engine.tool_call.dispatch rejects any final batch that mixes a terminal tool with sibling tool calls. The rejection returns one error tool result per tool-use id and executes none of the batch.
  5. execute_tool_once() validates input, runs prehooks, emits start, executes the body, validates output, runs posthooks, and stamps non-error terminal results with is_terminal=True.
  6. The dispatcher projects the first terminating result into terminal_result. The query loop sets QueryExitReason.TOOL_STOP and returns without another model turn.

Descriptor Registry #

TERMINAL_DESCRIPTORS is the shared terminal prose registry. Each TerminalToolDescriptor has a parent-facing selection_guidance field and an advisor-facing advisor_review_focus field; render_terminal_catalog() renders either view for a selected terminal list.

parent guidancetask_center.agent_launch.skill_message renders the selected terminal list into launch guidance and skill rows so the agent sees exactly the terminals allowed for this launch.
advisor guidanceask_advisor renders the same terminal names with focus="advisor_review_focus", so the audit prompt reviews the parent's choices through the same registry.
launch-specific setTerminalToolRouter computes an effective agent definition. The registered profile remains unchanged, while the launched copy carries filtered terminals such as planner close-only or executor no-handoff at deeper workflow depth.

Role Submission Terminals #

The role submission terminals are adapters over TaskCenter state owners. They do not coordinate peer agents directly; they resolve the current task, build typed submissions, and call the Workflow -> Iteration -> Attempt owner for the current attempt or delegated workflow.

Planner

submit_plan_closes_goal and submit_plan_defers_goal validate plan_spec, evaluation_criteria, generator tasks, task specs, dependencies, and optional deferred_goal_for_next_iteration. Accepted payloads call AttemptOrchestrator.apply_plan_submission().

Executor

submit_execution_success and submit_execution_blocker resolve ExecutorSubmissionContext and call apply_generator_submission() with generator_role="executor". submit_execution_handoff starts a child goal through WorkflowStarter.start(WorkflowOrigin.task(...)).

Verifier

submit_verification_success and submit_verification_failure are generator submissions with generator_role="verifier". The success path carries checks; the failure path carries unresolved issues for retry or failure handling.

Evaluator

submit_evaluation_success and submit_evaluation_failure call AttemptOrchestrator.apply_evaluator_submission(). The orchestrator and stage advancer, not the tool body, decide whether the attempt closes, retries, or advances.

Validation And Stop Path #

Terminal tools terminate only after the normal tool validation pipeline succeeds. A pydantic input failure, a prehook failure, a body exception, an invalid output shape, or a posthook failure returns an error ToolResult; error results are deliberately not stamped as terminal.

Input validation
parse_tool_input() strips engine runtime-control fields such as background, validates the pydantic input model, and returns a retryable invalid-input message on validation errors.
Body execution
execute_tool_body() catches tool exceptions, logs the traceback, and returns bounded error text with the exception type.
Output validation
validate_tool_output() validates non-error output against the tool's output model. Invalid JSON or pydantic mismatch becomes an error result with output_validation_error metadata.
Terminal stamping
execute_tool_once() uses dataclasses.replace() to set is_terminal=True only when tool.is_terminal_tool and the final result is not an error.
Runtime metadata
ExecutionMetadata carries TaskCenter ids, attempt_runtime, composer, conversation messages, tool registry, background manager, sandbox invocation ids, and extras such as active_terminals.
Loop stop
dispatch_assistant_tools() returns the first terminating result. engine.query.loop stores it on context.terminal_result, sets QueryExitReason.TOOL_STOP, and stops the run.
Failure mode A terminal tool batched with siblings is rejected before any sibling foreground, background, sandbox, or TaskCenter side effect runs. A terminal tool that returns an invalid output shape is also non-terminal because output validation converts it to an error.

Advisor Before Terminal #

ask_advisor is the read-only validation pattern for terminal outputs before the parent makes its own terminal submission. The parent sends the intended terminal name and payload; the helper receives the parent's original context, original task, filtered transcript, terminal catalog rendered with advisor review focus, and the pending submission JSON.

Helper Results And Notifications #

Helper terminal outputs are terminal inside the helper run, then ordinary tool results for the parent. This lets helpers use the same engine stop contract while keeping parent TaskCenter state under the parent submission tools and orchestrator.

Explorer

submit_exploration_result terminates explorer-style work and returns subagent_role="explorer", findings, and references. It does not resolve attempt context or call TaskCenter mutators.

Handoff-after-edit reminder

make_workflow_request_after_edit_reminder() watches for generator edit tools such as write_file, edit_file, and shell. After edits start, it nudges the executor away from submit_execution_handoff and toward success or blocker terminals.

  • Advisor direct launches call engine.api.run_ephemeral_agent with persist_agent_run=False and inherit the active sandbox id when one exists.
  • Verifier and evaluator inline edits via edit_file / write_file use ordinary tool permissions and sandbox routing; there is no separate edit-capable helper.
  • Helper results are evidence for the parent. The parent still must call its own TaskCenter terminal to mutate parent attempt state.

Tests And Maintenance #

Registration and routing

test_submission_tool_registration.py pins the registered submission names and asserts helper request tools are non-terminal while submission tools are terminal. Terminal router tests pin launch-specific filtering.

Validation and stop behavior

test_tool_execution.py covers terminal result capture, terminal batch rejection, framework stamping, and non-terminal continuation behavior.

Planner and role submissions

Planner submission tests cover duplicate tasks, dependency validation, task spec matching, close and defer variants, and Workflow -> Iteration -> Attempt handoff through the attempt orchestrator.

Helpers and reminders

Helper tests cover advisor direct launch shape, missing terminal-result errors, helper terminal metadata, explorer output metadata, and the handoff soft reminder.
