This page documents the live TaskCenter role contracts loaded from agent profile Markdown. Current verdict: roles are declarative profiles plus launch-time filtering; TaskCenter coordination flows through persisted goal, iteration, attempt, task, terminal submission, and close-report state, not through peer-to-peer agent messages or a global agent orchestrator.

Scope And Verdict #

  • Main agents are planner, executor, verifier, and evaluator. Their role contracts live in backend/src/agents/profile/main/*.md.
  • Every main profile gets _main_role_contract.md prepended at load time, so the shared terminal discipline and XML-context boundaries are part of the actual system prompt.
  • Planner submissions may target only registered executor or verifier profiles with dispatchable_by_planner: true.
  • Delegation is state-backed. submit_execution_handoff starts a child goal through WorkflowStarter, the parent task moves to WAITING_WORKFLOW, and WorkflowClosureReportRouter resumes the parent through its attempt orchestrator.

Profile Loading #

Dispatch And Terminals #

Profiles declare the broad tool surface, but launch and planner-submission paths narrow how those declarations are used. The engine registers the union of allowed_tools and effective terminals, while TaskCenter launch routing can return an effective profile copy with fewer terminal tools for this run.

Runtime tool surface
engine.agent.factory._build_agent_tool_registry() registers sorted(set(agent_def.allowed_tools) | set(agent_def.terminals)) and records the role as agent_def.agent_kind.value in tool metadata.
Planner dispatch gate
_is_generator_capable_agent() requires both dispatchable_by_planner and an executor or verifier agent_kind.
Terminal descriptors
tools/_terminals/registry.py owns selection guidance and advisor-review focus for every terminal. Row 3, row 4, and advisor prompts render from the same catalog.
Launch filtering
TerminalToolRouter preserves the registered profile and returns a copied effective definition only when terminals must be filtered.

Workflow Boundaries #

Plannersubmits one DAG and rubric
Executoredits or delegates assigned task
Verifierchecks generator output
Evaluatorjudges attempt criteria
Terminal submissionstate mutation, not prose
AttemptOrchestratorstage and task transitions
WorkflowStarterchild goal from handoff
Close report routerparent task resume
planner boundaryThe planner writes one attempt plan, task DAG, task specs, and evaluation criteria. It does not execute tasks, close iterations, or decide final goal state.
executor boundaryThe executor completes <assigned_task>, reports a concrete blocker, or starts a delegated workflow through WorkflowStarter when handoff is exposed.
verifier boundaryThe verifier is a generator role with read/write tools; it may apply trivial inline fixes via edit_file / write_file and then submit verification success or failure.
evaluator boundaryThe evaluator runs after generator tasks pass and judges the current Attempt against <evaluation_criteria>. Its terminal result drives retry, failure, or continuation through Workflow -> Iteration -> Attempt code.
handoff boundaryA handoff marks the parent generator task WAITING_WORKFLOW. Child completion returns as a WorkflowClosureReport; the parent orchestrator applies it and advances ready tasks.
coordination boundaryNo role sends peer-to-peer instructions to another role. Durable fields, task summaries, context packets, and terminal payloads are the handoff surfaces.

Helper Profiles #

advisor

Blocking read-only helper. It audits a parent agent's pending terminal submission and must end with submit_advisor_feedback.

explorer

Read-only subagent for focused investigation. It uses read_file, glob, and submit_exploration_result.

Notification triggers

request_workflow_after_edit reminds executors about handoff timing after edits. Verifier and evaluator declare no notification triggers; their inline-edit policy is enforced by profile prose plus the advisor pre-terminal gate.

Helper and subagent profiles may omit context_recipe. validate_agent_definitions_resolved() allows definitions with no recipe, and build_explorer_task_guidance() documents that explorer subagents do not go through the TaskCenter composer path.
