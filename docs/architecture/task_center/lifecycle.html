This is the most important TaskCenter page: the control plane is the persisted Workflow -> Iteration -> Attempt model. A Workflow is the objective, an Iteration is the bounded planning slice for that objective, and an Attempt is one planner -> generator DAG -> evaluator try inside that Iteration. Everything else in TaskCenter exists to feed, advance, retry, continue, or close this state chain.

Workflow -> Iteration -> Attempt Model #

  • Workflow: owns the objective, origin, ordered iteration ids, terminal status, and final closure report projection.
  • Iteration: owns one iteration-local goal string plus retry budget. Deferred continuation creates another Iteration under the same Workflow.
  • Attempt: owns one concrete try inside one Iteration: planner terminal submission, generator DAG rows, generator outcomes, evaluator outcome, and attempt close.
  • Task rows: store the launched planner, generator, and evaluator agents for an Attempt. They are not a fourth control-plane level.

Workflow Start Workflow #

WorkflowStarter.start() is the only prompt-to-Workflow start path. It strips and validates the prompt, prepares entry or task origin, creates the Workflow row, creates Iteration 1 with a registered IterationAttemptCoordinator, creates the unstarted initial Attempt, optionally marks the parent task WAITING_WORKFLOW, and then starts the Attempt orchestrator.

Retry And Deferral #

Retry and deferral are separate Workflow -> Iteration -> Attempt decisions. A failed Attempt can retry inside the same Iteration while budget remains. A passed Attempt can either terminally close the Workflow or carry deferred_goal_for_next_iteration, which closes the current Iteration as succeeded and starts the next Iteration under the same Workflow.

Delegated Child Goals #

Delegation creates another complete Workflow -> Iteration -> Attempt chain. It starts from the executor submission context, not from a direct agent-to-agent channel. submit_execution_handoff resolves the running generator task and calls ExecutorSubmissionContext.start_delegated_workflow(), which invokes WorkflowStarter.start(prompt=goal_handoff, origin=WorkflowOrigin.task(task_id=current_task_id)).

Nested Planning Case #

The nested planning case is a delegated child Workflow whose planner must plan inside a parent generator task. This is still the same model: the parent Attempt pauses one generator task, the child Workflow gets its own Iteration and Attempt, and the child planner creates its own generator DAG.

Depth rule The root Workflow is depth 1. A delegated child Workflow is depth 2. In current launch routing, depth greater than 1 disables further planning deferral and executor handoff for that nested Workflow, so nested planning stays bounded by terminal close/blocker outcomes.

Closure Routing And Invariants #

WorkflowLifecycle.handle_iteration_closed() is the transition point from iteration reports to goal closure or continuation. It creates deferred iterations for SuccessDeferred, closes goals for TerminalSuccess and AttemptPlanFailed, then deregisters the closed iteration coordinator.

entry-origin deliveryWorkflowClosureReportRouter._deliver_entry_origin() finishes the TaskCenter run as done or failed and treats already finished runs as idempotent delivery.
task-origin deliveryThe router looks up the parent task, requires WAITING_WORKFLOW, reconstructs the parent task object from AttemptDeps, and calls apply_workflow_closure_report().
process-local registryWhile a parent task is WAITING_WORKFLOW, its parent attempt cannot be quiescent and its AttemptOrchestrator must still be registered in this process. A missing orchestrator is a hard invariant violation.
idempotent close reportsThe router treats parents already DONE or FAILED as already_delivered, and the orchestrator returns silently if a second direct report finds the task no longer waiting.
state validation_core/invariants.py centralizes checks for open goals, contiguous iteration and attempt sequence numbers, stage correctness, ownership, failure reasons, and valid attempt close states.
  • Workflow -> Iteration -> Attempt policy stays in WorkflowStarter, WorkflowLifecycle, IterationAttemptCoordinator, WorkflowClosureReportRouter, and AttemptOrchestrator.
  • The context engine later reads persisted state. It does not choose retry, deferral, child-goal delivery, or goal closure policy.
  • The per-attempt orchestrator is not a global agent orchestrator; it only advances one attempt's planner, generators, and evaluator.
