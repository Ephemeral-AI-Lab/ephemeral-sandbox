EphemeralOS is a multi-agent coding framework built on one bet: a long-horizon coding task is fundamentally a memory-management problem, so memory belongs in cheap, durable storage — not in an expensive, volatile context window. Many small, disposable agents collaborate on one shared, concurrency-safe workspace instead of messaging each other or branching into isolated copies.

Typed multi-agent roles OCC-gated shared workspace O(1) cost per operation Built-in quality gate Forward-only recovery

The Bet §

Two planes, one principle. The TaskCenter control plane holds durable Goal → Iteration → Attempt state and runs a planner → generator → evaluator harness. The Sandbox substrate is a shared workspace where every write is merged safely and made visible immediately. Agents coordinate by reading shared truth — the decision state and the artifact tree — never by talking to each other.

Why EphemeralOS §

What this architecture does that container-, worktree-, and single-thread approaches structurally cannot:

One shared workspace, not isolated copies

Every write / edit merges into one mutable tree and is visible on the next agent's read — collaboration with no agent-to-agent messaging. See Workspace Modes.

Safe concurrent writes (OCC)

Database-grade optimistic concurrency control: tracked files are stale-base-rejected at commit time, so one agent's edit never silently clobbers another's. See OCC.

O(1) cost per operation

Immutable lower layers are shared by lease; each op adds only its changed bytes. A 2 GB repo with 50 agents costs ~2 GB, not 100 GB. See Space Model.

Memory on disk, not in-context

Durable state is the memory; each agent gets a fresh, role-scoped context packet projected from it — no monotonically growing context, no compaction tax.

Built-in quality gate

The planner authors the rubric up front; a per-task verifier, a per-attempt evaluator, and a pre-terminal advisor gate work before it commits.

Forward-only, self-correcting

No rollback machinery: the system hands the next agent accurate context and trusts it to self-correct, and recovery composes across nested execution lanes.

The shared-workspace advantage

Most multi-agent stacks must choose between cheap-but-isolated and collaborative-but-expensive. OCC-over-LayerStack is the only model that is both — cheap and collaborative.

Multi-Agent Landscape §

Where EphemeralOS sits among multi-agent coding approaches. The comparison is on coordination architecture — how agents share work and writes — not on benchmark performance. The decisive axis is concurrent-write safety on a shared tree: most systems are either single-writer, isolate into copies, or share a working tree and avoid collisions by convention (task partitioning, messaging) rather than by control. The EphemeralOS column is grounded in this codebase; other rows characterize systems from public information as of early 2026 and may lag their latest releases — correct any row you know better.

Safety by control, not convention The systems that do run agents in parallel on a shared tree — Claude Code teams, oh-my-claudecode team, Qoder expert teams — keep writes from colliding by convention: partition the task list so two agents do not touch the same files. That holds until they do. EphemeralOS's OCC stale-base rejection over shared LayerStack layers makes overlapping writes structurally safe, so correctness never depends on agents staying out of each other's way.

How a Task Runs §

A run binds a sandbox workspace and starts a root Goal. Every Goal advances through one or more Iterations; each Iteration runs Attempts; each Attempt is one planner → generator DAG → evaluator pass. The whole lifecycle is persisted, so coordination needs no live orchestrator.

  1. Bootstrap. A run binds the workspace and GoalStarter creates the root Goal, its first Iteration, and its first Attempt.
  2. Plan. The planner reads goal + history context and submits a validated generator DAG (unique ids, known deps, acyclic) plus the evaluation rubric.
  3. Generate. Ready generators launch in parallel waves; each reads the shared workspace, does its task, and publishes writes through OCC.
  4. Verify. A per-task verifier checks each generator's output; a failed or blocked node makes its descendants unreachable while runnable siblings drain.
  5. Evaluate. Once the DAG is done, one evaluator judges the whole Attempt against the planner's rubric.
  6. Resolve. Pass closes the Goal or defers the next phase; fail retries within budget, carrying the failure into the next plan.
  planner ── submits ──▶  GENERATOR DAG   (validated: acyclic, known deps)

        g1 ─┐
            ├─▶ g3 ──▶  evaluator ──▶  PASS ▸ close / defer next phase
        g2 ─┘                          FAIL ▸ retry (budget) with hindsight

  per generator:  read ─▶ work ─▶ OCC publish ─▶ submit terminal
  a failed node blocks its descendants; runnable siblings still drain first

Recovery & Replanning §

There is no checkpoint/rollback machinery and no compaction step. The world only moves forward; when something fails, the system hands the next agent an accurate picture of current reality and trusts it to self-correct. Adaptation is layered, and every loop replans at a verified checkpoint rather than mid-thread.

  within-task      each generator is a full agent loop, adapts inside its task
  mid-attempt      generator hits sub-work ─▶ handoff ─▶ child Goal (own lifecycle)
  cross-attempt    failure ─▶ retry; next planner sees prior failures + verdict
  cross-iteration  success + defer ─▶ next phase, prior phases visible

  forward-only:  no rollback, no compaction. Decomposition keeps each agent's
  context small by construction, so there is nothing to compress.
Why no live mid-flight replan No multi-agent controller can watch N parallel sub-workers stream live within a context window — so every multi-agent system replans at return boundaries. EphemeralOS makes those checkpoints durable, structured, and evidence-rich instead of pretending to monitor live.

Workspace Modes §

One tool surface, two lifetimes. Daemon routing picks the mode from the caller's state, so agents use the same read / write / edit / shell tools either way.

ephemeral_workspace · shared

The default. Each operation gets a fresh overlay or a direct OCC fast path, and accepted writes publish to shared truth — so co-worker agents see them on their next read. This is the collaborative blackboard. See ephemeral_workspace.

isolated_workspace · private

Opt-in private namespace (own user / net / pid / mount) for destructive scripts, package installs, or risky experiments. Writes are captured for audit but never published, and the scratch is discarded on exit — the blast-radius safety valve. See isolated_workspace.

Design Principles §

Every major decision is the same instinct applied at a different layer: prevent the expensive, volatile thing from forming; externalize truth to cheap, durable substrate; let capable agents self-correct from accurate context. One bet, five times:

ProblemConventional answerEphemeralOS answer
Long contextcompaction — compress after it growsdecomposition — recursive handoff so it never accumulates
Failure recoverycheckpoint + rollbackforward-only + self-correct from context
Coordinationcentral orchestrator / messagingshared blackboards (durable state + workspace)
Memoryin-model context windowdurable disk — storage over tokens
Concurrent writesisolate, then merge laterOCC commit-back into shared truth
