EphemeralOS is a multi-agent coding framework built on one bet: a long-horizon coding task is fundamentally a memory-management problem, so memory belongs in cheap, durable storage — not in an expensive, volatile context window. The current runtime implementation is Rust-first: agent-core/ owns the agent control plane and sandbox/ owns the sandbox substrate, while Python test_runner remains the validation and benchmark harness.

Typed multi-agent roles OCC-gated shared workspace O(1) cost per operation Reducer evidence boundary Forward-only retry and replan

The Bet §

Two planes, one principle. The Task / Workflow control plane persists every agent assignment as a Task, runs the root Task directly, and launches delegated Workflow → Iteration → Attempt decomposition only when an agent calls delegate_workflow. The Sandbox substrate is a shared workspace where every write is merged safely and made visible immediately. Agents coordinate by reading shared truth — the decision state and the artifact tree — never by talking to each other.

Why EphemeralOS §

What this architecture does that container-, worktree-, and single-thread approaches structurally cannot:

One shared workspace, not isolated copies

Every write / edit merges into one mutable tree and is visible on the next agent's read — collaboration with no agent-to-agent messaging. See Workspace Modes.

Safe concurrent writes (OCC)

Database-grade optimistic concurrency control: tracked files are stale-base-rejected at commit time, so one agent's edit never silently clobbers another's. See OCC.

O(1) cost per operation

Immutable lower layers are shared by lease; each op adds only its changed bytes. A 2 GB repo with 50 agents costs ~2 GB, not 100 GB. See Space Model.

Memory on disk, not in-context

Durable state is the memory; each agent gets a fresh, role-scoped AgentContext projected from it — no monotonically growing context, no compaction tax.

Built-in quality gate

The planner authors a generator/reducer DAG, reducers decide what evidence is valid, and only successful reducer evidence becomes prior-iteration context for later replans.

Forward-only, self-correcting

No rollback machinery: the system hands the next agent accurate context and trusts it to self-correct, and recovery composes across nested execution lanes.

The shared-workspace advantage

Most multi-agent stacks must choose between cheap-but-isolated and collaborative-but-expensive. OCC-over-LayerStack is the only model that is both — cheap and collaborative.

Multi-Agent Landscape §

Where EphemeralOS sits among multi-agent coding approaches. The comparison is on coordination architecture — how agents share work and writes — not on benchmark performance. The decisive axis is concurrent-write safety on a shared tree: most systems are either single-writer, isolate into copies, or share a working tree and avoid collisions by convention (task partitioning, messaging) rather than by control. Safety by control, not convention The systems that do run agents in parallel on a shared tree — Claude Code teams, oh-my-claudecode team, Qoder expert teams — keep writes from colliding by convention: partition the task list so two agents do not touch the same files. That holds until they do. EphemeralOS's OCC stale-base rejection over shared LayerStack layers makes overlapping writes structurally safe, so correctness never depends on agents staying out of each other's way.

How a Task Runs §

A request binds a sandbox workspace, mints a root Task, and runs the root agent directly. Delegated decomposition starts only when that root agent, or a workflow generator, calls non-terminal delegate_workflow; every created Workflow then advances through one or more Iterations, each Iteration runs Attempts, and each Attempt is one planner-authored plan — a DAG of generator and reducer Tasks whose edges are needs. The stages collapse to PLAN → RUN → CLOSED. The lifecycle is persisted, so coordination needs no live orchestrator.

  1. Bootstrap. Runtime entry binds the workspace, creates the request row, mints the root Task with workflow_id=NULL, and runs the root agent. No Workflow is created for the root request.
  2. Plan. The planner reads goal + history context and submits a validated plan — generator tasks plus at least one reducer (unique ids, known needs, acyclic, generator-only dependency DAG, reducers directly gating one or more generator outputs).
  3. Run. The single RUN stage schedules generator and reducer tasks together to quiescence. Ready tasks launch in parallel waves; each generator reads the shared workspace, does its work, and publishes through OCC; each reducer judges whether its needs outcomes satisfy its prompt. A failed or blocked task makes its descendants unreachable while runnable siblings drain.
  4. Close. The attempt PASSES iff every plan task is DONE, and FAILS (TASK_FAILED) if any task failed or blocked.
  5. Resolve. Pass closes the Workflow or defers the next phase; fail retries within budget, carrying failed generator/reducer evidence inside the same Iteration. Task outcomes are task-bound evidence; aggregate attempt, iteration, and workflow outcomes are explicit generator/reducer projections.

Recovery & Replanning §

There is no checkpoint/rollback machinery and no compaction step. The world only moves forward; when something fails, the system hands the next agent an accurate picture of current reality and trusts it to self-correct. Adaptation is layered, and every loop replans at a verified checkpoint rather than mid-thread.

  within-task      each generator is a full agent loop, adapts inside its task
  mid-attempt      generator hits sub-work ─▶ delegate_workflow ─▶ Workflow handle
  cross-attempt    failure ─▶ retry; next planner sees local failed evidence
  cross-iteration  success + defer ─▶ next phase sees successful reducer evidence

  forward-only:  no rollback, no compaction. Decomposition keeps each agent's
  context small by construction, so there is nothing to compress.
Dynamic workflow stance Workflow decomposition is dynamic through durable boundaries, not hidden script variables: fixed plan DAGs cover known fanout, delegate_workflow covers runtime-discovered branches, same-iteration retry handles failed generator/reducer evidence, and deferred Iterations replan only from reducer-approved evidence.

Workspace Modes §

One tool surface, two lifetimes. Daemon routing picks the mode from the caller's state, so agents use the same read / write / edit / shell tools either way.

ephemeral_workspace · shared

The default. Each operation gets a fresh overlay or a direct OCC fast path, and accepted writes publish to shared truth — so co-worker agents see them on their next read. This is the collaborative blackboard. See ephemeral_workspace.

isolated_workspace · private

Opt-in private namespace (own user / net / pid / mount) for destructive scripts, package installs, or risky experiments. Writes are captured for audit but never published, and the scratch is discarded on exit — the blast-radius safety valve. See isolated_workspace.

Design Principles §

Every major decision is the same instinct applied at a different layer: prevent the expensive, volatile thing from forming; externalize truth to cheap, durable substrate; let capable agents self-correct from accurate context. One bet, five times:

ProblemConventional answerEphemeralOS answer
Long contextcompaction — compress after it growsdecomposition — recursive handoff so it never accumulates
Failure recoverycheckpoint + rollbackforward-only + self-correct from context
Coordinationcentral orchestrator / messagingshared blackboards (durable state + workspace)
Memoryin-model context windowdurable disk — storage over tokens
Concurrent writesisolate, then merge laterOCC commit-back into shared truth

Rust Migration §

  • Current owner split: Rust owns request bootstrap, Task/Workflow state, the engine loop, model-facing tools, sandbox host/API, daemon dispatch, command sessions, LayerStack/OCC/overlay, isolated workspace, and plugin PPC.
  • Python comparison role: legacy backend/src/workflow, backend/src/engine, backend/src/tools, and backend/src/sandbox paths are parity/history references, not current runtime owners.
  • Remaining Python owner: backend/src/test_runner drives mocked scenarios, SWE-EVO, live E2E lanes, audit/report assembly, and migration gates. It validates Rust behavior; it does not define runtime behavior.
