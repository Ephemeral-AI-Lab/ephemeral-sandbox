The mocked-agent squad is deterministic but behaves like a real agent: it inspects the prompt, picks a prepared sequence, calls real sandbox tools, inspects their outputs, and submits through the real terminal tool. The squad uses production main-profile markdown definitions so live e2e coverage exercises the same frontmatter, variants, terminals, and system prompts as production launches.

Scope #

  • The mock runner lives at backend/src/task_center_runner/agent/mock/runner.py. It is constructed by build_scenario_config and never imported directly by scenarios.
  • Agent kind dispatch (planner / executor / verifier / evaluator) is the only branching the runner does on the scenario; everything else flows through real submission tools.
  • Prepared tool scripts and probes let the executor lane exercise complex sandbox surfaces (LayerStack, OCC, overlay, plugin, ephemeral workspace, background shell) without growing the scenario protocol.
  • The runner emits four side-channel MOCK_* events that ScenarioLifecycle decodes into the rich RunReport.

Squad Registry #

registered_mock_agents() is the context manager that swaps the live agent registry to the production main-profile markdown definitions for the duration of one mock run.

run_scenario(...)                            agents/profile/main/
    |                                          planner.md
    | with registered_mock_agents():           executor.md     -> name=executor
    |     1. previous = list_definitions()     generator_verifier.md -> name=verifier
    |     2. unregister all currently          evaluator.md
    |        registered definitions
    |     3. for d in load_agents_dir(         |
    |          _MAIN_PROFILE_DIR):             v
    |            register_definition(d)     register_definition(...)
    |     4. yield (run_pipeline executes)
    |     5. finally:
    |        unregister all current; reregister `previous`
  • _MAIN_PROFILE_DIR is backend/src/agents/profile/main; the launcher resolves planner/executor/verifier/evaluator from those markdown files.
  • Restoration on exit makes the swap reversible — pytest sessions that run multiple scenarios back-to-back leave the registry untouched between tests.
  • The benchmark lane does NOT use this context manager; instead bootstrap_real_agent_runtime loads the same markdown tree once via load_agents_tree.

Per-Agent Dispatch #

The runner's __call__ coroutine has the same signature as the real AttemptAgentRunner so TaskCenter cannot tell them apart. The dispatch fan-out is purely by agent_def.agent_kind.

MockSquadRunner.__call__(config, prompt, agent_def, sandbox_id, extra_tool_metadata, ...)
   |
   | 1. metadata = self._metadata_for(...)
   |        sets sandbox_id, agent_name, repo_root, cwd,
   |        role = agent_def.agent_kind.value
   |
   | 2. publish MOCK_LAUNCH_RECORDED  (LaunchRecord)
   |
   | 3. wire_payload = prompt + "\n" + initial_messages text
   |    publish MOCK_PROMPT_INSPECTED (PromptInspection over checks)
   |
   | 4. record initial_messages on the audit recorder
   |
   | 5. publish PLANNER_INVOKED | EXECUTOR_INVOKED |
   |            VERIFIER_INVOKED | EVALUATOR_INVOKED
   |
   | 6. dispatch on agent_def.agent_kind.value:
   |       "planner"   -> _run_planner(metadata, emit)
   |       "executor"  -> _run_executor(prompt, metadata, emit)
   |       "verifier"  -> _run_verifier(prompt, metadata, emit)
   |       "evaluator" -> _run_evaluator(metadata, emit)
   |       (anything else raises)
   |
   | 7. each handler builds a ScenarioContext, asks the scenario for the
   |    decision, prepends advisor-approval shim, calls the submission
   |    tool through execute_tool_once, publishes the *_SUCCESS or
   |    *_FAILURE event, and returns the ToolResult.
   |
   v
EphemeralRunResult(status="completed", terminal_result=<ToolResult>, agent_name, event_count=1)
_run_plannerReads scenario.planner_response(ctx) (or mutable_state.consume_next_planner_response() when one is queued), gates the terminal with the advisor-approval shim, calls the submission tool, and emits PLANNER_COMPLETES_GOAL_PLAN or PLANNER_DEFERS_GOAL_PLAN with the plan summary.
_run_executorWalks scenario.executor_actions(ctx). fail/fail:reason submits submit_execution_blocker. request_recursive_workflow:<pkg> or request_recursive_matrix:<pkg> submits submit_execution_handoff with the scenario's recursive-handoff goal. Otherwise the action name selects a probe (preflight, sandbox_integrity, final_probe, inspect_user_input, execute_package:<id>, dozens of probe-mode tokens) and ends with submit_execution_success.
_run_verifierRuns the verifier-checkpoint script (a readback probe targeting the requirement ledger, recursive close report, or final reconciliation file). Calls scenario.verifier_response(ctx) and submits success/failure with the parsed checkpoint payload. Composite scenarios use the full_stack checkpoint script variant.
_run_evaluatorPlain submission of scenario.evaluator_response(ctx) with the advisor-approval shim prepended.

Prepared Tool Scripts #

PreparedToolScript is a deterministic sequence of real tool calls; PreparedToolScriptEngine walks the steps, emits a status text-delta, calls each tool, and verifies expect_error contracts.

ToolScriptStepOne concrete tool call: label, tool: BaseTool, args: dict, optional expect_error for negative probes.
PreparedToolScriptname, summary, artifact path, and the tuple of steps.
PreparedToolScriptEngineConstructed with the runner's _call_tool. run(script, metadata, emit) emits an AssistantTextDeltaEvent per step, calls each tool, and bails if an expect_error step unexpectedly succeeds.
ToolScriptResultReturned to the runner: script_name, summary, artifact, and the tuple of ToolResult values.

Built-in scripts live at tool_scripts.py (case-style scenarios) and full_stack_tool_scripts.py (composite adversarial). Examples:

inspect_user_input_script

Persists and verifies the requirement ledger parsed from user input. Creates the package dir, asserts the SWE-EVO workspace at /testbed, writes a ledger JSON, drops a conflict-probe file, runs an intentional missing-anchor edit (negative path), seeds AUTO_SQUASH_MAX_DEPTH + 4 writes to drive the squash path, then reads back the ledger.

execute_package_script(package_id)

Walks one package: ensure dir, write evidence JSON, edit it to flip "edited":false to true, read it back, and shell-check the artifact exists.

recursive_step_script

Persists evidence for one delegated recursive goal step. When the context message contains recursive_reconcile, also writes a close-report JSON used by the parent's verifier checkpoint.

final_reconciliation_script

Writes the per-stage final ledger JSON and a stable final-reconciliation.json summary used by the verifier's final_release checkpoint readback.

verifier_checkpoint_script

Readback probe — picks an artifact path by checkpoint name (inventory → ledger, recursive_return → close report, final_release → final reconciliation), reads it, and shell-checks the file tree.

full_stack_*_script

occ_conflict_matrix_script, overlay_edge_matrix_script, layerstack_squash_lease_script, lsp_refresh_semantics_script, recursive_oversized_matrix_script, full_stack_final_reconciliation_script, full_stack_verifier_checkpoint_script. These drive the composite scenarios (FullStackAdversarial, FullCaseUserInput, FullSystemCapacityMatrix).

Probe Library #

Probes are larger end-to-end checks that the executor lane dispatches when it sees a probe action token. The runner has dispatch tables for the families with mode parameters.

Sandbox integrity probe

Inline in runner.py::_run_sandbox_integrity_probe. Mkdir + shell-created file, write_file direct merge, read-back, edit_file direct merge, batch sandbox_api.edit_file, intentional missing-anchor conflict (negative path), shell append to drive squash, final read-back asserting the trailing line is present.

Auto-squash commit-resume probe

Drives the OCC mutation critical path until layer-stack depth crosses AUTO_SQUASH_MAX_DEPTH, then issues edits, reads, a shell readback, and one intentional missing-anchor conflict. Captures timings into a summary artifact so the paired test asserts on commit_resume_wait_s, auto_squash.total_s, and depth_before > AUTO_SQUASH_MAX_DEPTH.

complex_project_build_*

Three flavors: pure build, build + shell/edit/LSP, build + grep/glob. Each has a smoke variant. Drives a multi-package repo through the public-tool pipeline (shell, edit_file, read_file, optionally LSP go-to-definition / references / hover). Imported lazily inside the runner to keep top-level imports light.

ephemeral_workspace_probe

Modes: all_verbs, concurrent_writes, same_path_conflict, policy, cancellation, o1_disk. Exercises the agent-facing isolated-workspace lifecycle plus the host-side sandbox isolated-workspace machinery.

plugin_workspace_probe

Modes: read_only_lsp_refresh, write_allowed_publish, intent_contract, iws_policy, setup_failure, service_evict. Drives the sandbox plugin pipeline through public-tool requests, asserts intent and policy.

background_shell_probe

11 modes mapped through a dispatch dict: golden, cancel, interleave, exhaustion, partial-write cancel, cancel-during-maintenance, late-cancel-race, mixed-fg-bg same-path conflict, heartbeat loss, exit-iws drain, engine-restart no-lease-leak, many-small-writes do-not-starve-dispatcher. One mode per concurrency invariant.

high_concurrency_probe

Three roles: seed, worker (indexed), reconcile. The scenario emits one seed action, several worker actions in parallel, and a reconcile action so the runner stresses LayerStack/OCC/overlay together.

heavy_io_zoned_probe

Same seed/worker/reconcile shape but with heavy I/O zoned reads/writes. Used by HeavyIoZonedConcurrent.

  • Probe-mode dispatch dicts raise on unknown tokens; the runner never silently no-ops a typo in the scenario.
  • Heavy probes are imported lazily inside the dispatch handler so adding a new probe does not slow scenario import.
  • Every probe ends with a JSON or text summary artifact whose path is returned to the runner as the executor's artifacts list.

Advisor-Approval Shim #

The mock squad bypasses the engine loop, so conversation_messages arrives empty and every gated terminal would trip AdvisorApprovalPreHook. The shim injects the same two-message pair the engine would have produced when an agent calls ask_advisor followed by an advisor approval verdict for the specific terminal being submitted.

runner builds metadata for terminal tool T
   |
   v
_approve_terminal(metadata, T):
   existing = metadata.get("conversation_messages") or []
   gated["conversation_messages"] = (
       build_advisor_approval_messages(tool_name=T.name)
       + existing
   )
   return gated
   |
   v
execute_tool_once(T, args, gated_metadata, emit)
   |
   v
AdvisorApprovalPreHook sees the matching approval message pair
   and lets the terminal submission proceed.

Negative path: pass the WRONG tool name to _approve_terminal and the
hook still fires, confirming the gate is not a no-op.

The shim is the smallest thing that lets every gated terminal test ride the production code path without re-implementing the engine loop.

MOCK_* Side-Channel Events #

Four EventType values exist only for the mock lane. Real-agent and benchmark runs never emit them; subscribers must tolerate absence.

MOCK_LAUNCH_RECORDEDLaunchRecord(task_id, attempt_id, agent_name, role, prompt_preview). Published at the top of every __call__ so the report shows every dispatch.
MOCK_TOOL_CALL_RECORDEDToolCallRecord(task_id, tool_name, is_error, metadata). Published from _record_tool_check after every sandbox tool call inside a probe or script step.
MOCK_PROMPT_INSPECTEDPromptInspection(task_id, agent_name, role, checks, justification). Published once per dispatch; passed ANDs every check.
MOCK_SANDBOX_CHECK_RECORDEDSandboxCheck(name, passed, detail, changed_paths). Published from probes that go through sandbox_api directly (batch edit, conflict detection, intentional conflict).
  • ScenarioLifecycle.on_event decodes each payload into the typed record and appends it to the matching list; run_scenario copies those lists into the legacy RunReport.
  • The RunReport.passed_prompt_inspections and passed_sandbox_checks properties AND the boolean fields across every recorded item — so a single failed probe in a long scenario makes the whole report fail.
