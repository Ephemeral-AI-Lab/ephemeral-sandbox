A Scenario is a pure description of what a mocked planner / executor / verifier / evaluator should submit at each decision point. The mock runner translates those descriptions into real submission tool calls against a real TaskCenter run, and an audit-bus-fed lifecycle accumulates the events into a rich RunReport for assertions.

Scope #

  • Scenarios live under backend/src/task_center_runner/scenarios/. Composite scenarios live at the top level; focused scenarios are grouped by concern under subpackages.
  • SCENARIO_REGISTRY is the catalog (~45 entries across pipeline, sandbox, capacity, planner-validation, and composite lanes).
  • Scenarios never own state. The mock runner runs the scenario, the bus carries events, hooks read MutableMockState, and ScenarioLifecycle accumulates everything for the report.
  • The decision protocol is intentionally narrow: planner / executor / verifier / evaluator responses, optional recursive-handoff goal text, and hooks.

Scenario Protocol #

The protocol is implemented either via ScenarioBase (with overridable defaults) or by satisfying Scenario structurally.

nameDotted-path identifier used by SCENARIO_REGISTRY and as the audit recorder's scenario label. Example: pipeline.initial_workflow, sandbox.background_shell_golden.
expected_event_sequenceOrdered tuple of EventTypes the scenario expects. Drives assert_event_sequence hooks.
planner_response(ctx)Returns a ToolCallSpec(submit_plan_closes_goal | submit_plan_defers_goal, args). args typically contains plan_spec, tasks, evaluation_criteria.
executor_actions(ctx)Returns a sequence of action tokens consumed by the runner's executor dispatcher: "preflight", "fail"/"fail:reason", "request_recursive_workflow:<pkg>", probe action names. Default is empty.
verifier_response(ctx)Returns a ToolCallSpec targeting submit_verification_success or submit_verification_failure.
evaluator_response(ctx)Returns a ToolCallSpec targeting submit_evaluation_success or submit_evaluation_failure.
recursive_handoff_goal(ctx)Optional goal text used when an executor action requests a recursive handoff; defaults to None (the runner synthesizes a placeholder).
hooks()Sequence of Hook instances; registered into the per-run HookSet by build_scenario_config.

ScenarioContext is the live state visible to the scenario at each decision: attempt, iteration, goal, prompt, metadata, audit_recorder, mutable_state, task_id, agent_name, context_message, plus optional requirement_ledger, package_plan, and matrix_plan for composite scenarios.

ScenarioLifecycle Wiring #

One builder, one lifecycle, one shared MutableMockState. build_scenario_config is the single point of truth where the MockSquadRunner factory, HookSet, and lifecycle are wired together.

build_scenario_config(scenario, ...)
   |
   +--> MutableMockState()
   +--> HookSet().register(scenario.hooks()) + extra_hooks
   +--> ScenarioLifecycle(scenario, hook_set, mutable_state)
   +--> runner_factory = (ctx) -> MockSquadRunner(repo_dir, ctx.bus,
   |                              scenario=scenario,
   |                              mutable_state=mutable_state)
   |
   v
RunConfig (mock mode):
   sandbox = AttachExisting(sandbox_id)
   bootstrap = None
   lifecycle = ScenarioLifecycle (subscribed to bus on_event)
   run_label = "scenario_logs/<scenario.name>"


run_pipeline executes; per event:
   AuditEventBus.publish(event)
      |
      +--> ScenarioLifecycle.on_event(event):
              captured_events.append(event)
              mutable_state.seen_events.append(event.type)
              if event.type in MOCK_*:
                  decode payload into LaunchRecord /
                  ToolCallRecord / PromptInspection /
                  SandboxCheck and append
              hook_set.fire(event, "post", mutable_state)
                  -> append HookResults
captured_eventsEvery event the bus published in subscription order. Read by run_scenario to populate RunReport.events.
hook_resultsOne HookResult per hook firing, including assert_event_sequence verdicts.
launches / tool_calls / prompt_inspections / sandbox_checksTyped records decoded from the four MOCK_* event payloads. Used by mock tests to assert on prompt content, sandbox correctness, and tool dispatch.

Scenario Catalog #

The registry has five lanes, each owning a concern. Use the lane to pick where a new test should live; do not add cross-lane scenarios.

Pipeline (state machine)

Drives planner / generator DAG / evaluator control flow with the lightest possible executor action. Failures here mean a regression in task_center/ proper.

Reference scenarios: InitialWorkflow, NestedWorkflow, NestedWorkflowFailure, IterativeDeferral, DependencyDagSerial, DependencyDagParallel, DependencyDagDiamond, DependencyDagMixed, DependencyBlockedDescendants, AttemptRetryPlannerFailure, AttemptRetryGeneratorFailure, AttemptRetryEvaluatorFailure, AttemptBudgetExhausted, GeneratorFailureQuiescence, DeferredParentPlannerTerminalRouting, InitialMessagesCapture.

Sandbox

Exercise LayerStack / OCC / overlay / plugin / ephemeral-workspace / background-shell paths through real submission tools. Failures here mean a regression in sandbox/.

Subgroups: complex project build (ComplexProjectBuild*, ComplexProjectBuildGrepGlob*, ComplexProjectBuildShellEditLsp*), ephemeral workspace (EphemeralWorkspaceAllVerbs, EphemeralWorkspaceConcurrentWrites, EphemeralWorkspaceSamePathConflict, EphemeralWorkspacePolicy, EphemeralWorkspaceCancellation, EphemeralWorkspaceO1Disk), plugin (PluginReadOnlyLspRefresh, PluginWriteAllowedPublish, PluginIntentContract, PluginIwsPolicy, PluginSetupFailure, PluginServiceEvict), background shell (BackgroundShellGolden, BackgroundShellStop, BackgroundShellInterleave, BackgroundShellExhaustion, BackgroundShellPartialWriteCancel, BackgroundShellStopDuringMaintenance, BackgroundShellLateCancelRace, BackgroundMixedFgBgSamePathConflict, BackgroundHeartbeatLossReapsOnlyStaleBg, BackgroundExitIwsDrainsAgentTasks, BackgroundEngineRestartNoLeaseLeak, BackgroundManySmallWritesDoNotStarveDispatcher), concurrency (HighConcurrencyLayerstackOverlayOcc, HeavyIoZonedConcurrent, OccConcurrentConflicts, AutoSquashCommitResume).

Capacity

FullSystemCapacityMatrix — composite scenario that walks a pack catalog (capacity/pack_catalog.py) to stress the full Workflow / Iteration / Attempt graph plus sandbox surface together. Skipped unless runner.live_e2e.capacity_enabled is true.

Planner Validation

Negative-path scenarios for planner output validation: PlannerCycleInDeps, PlannerDuplicateLocalId, PlannerEmptyTasks, PlannerDefersWithoutDeferredGoal, PlannerUnknownAgentName, PlannerUnknownDep. The mock planner returns an intentionally bad plan; the runner asserts TaskCenter rejects it through the expected error path.

Composite

Multi-stage scenarios that combine pipeline and sandbox concerns under one entry prompt: CorrectnessTesting (the default scenario used when MockSquadRunner is constructed without one), FullCaseUserInput (requirement-ledger / package-plan / recursive-goal flow), FullStackAdversarial (occ conflict matrix, overlay edge matrix, layerstack squash lease, LSP refresh semantics, recursive oversized matrix, final reconciliation).

Hooks And MutableMockState #

Hooks are insertion-ordered, fired post-event in registration order. Per-event errors are wrapped into HookResult(failed_reason=...) rather than raised.

HookFrozen dataclass: name, event: EventType, when: "pre" | "post", fn: (event, state) -> HookResult.
HookSetInsertion-ordered registry. fire(event, when, state) yields one HookResult per matching hook.
MutableMockStateCross-hook scratchpad with seen_events, flags, and an injected-failure queue (inject_failure / consume_failure) keyed by (role, attempt_id, checkpoint). replace_next_planner_response swaps the next mock planner submission once.

count_events(event_type, name=)

Increments state.flags[f"count_{name}"] on every match. Use to assert counts of a given EventType across the run.

capture_prompt(role, name=)

Copies the *_INVOKED payload for the given role into state.flags["prompt_<name>"]. Pre-event hook.

assert_event_sequence(expected, name=)

RUN_COMPLETED post-hook that asserts the seen-event list contains the expected EventType subsequence in order. Returns a HookResult(asserted=False, failed_reason=...) on mismatch.

fail_verifier_at(checkpoint=)

VERIFIER_INVOKED post-hook that arms a one-shot verifier failure when the invocation payload carries checkpoint == <checkpoint>; the runner consumes the queued failure on the next verifier dispatch.

assert_guard_after_wave(wave_id=)

VERIFIER_SUCCESS post-hook that asserts a verifier guard for wave_id had at least one executor dependency. Defends against accidentally-leaf guards.

assert_recursive_workflow_closed_before_parent_guard

VERIFIER_SUCCESS post-hook that asserts RECURSIVE_WORKFLOW_COMPLETED precedes the parent recursive_return guard in seen-events order.

  • Two builtins are explicitly deferred and raise on call: fail_evaluator_at(attempt_seq=) and assert_squash_after_n_edits(n=). Both need additional event surfaces.
  • Hooks see state.seen_events as a running list — earlier hook firings have already appended their event type when later hooks see it.

Example: InitialWorkflow #

The smallest pipeline scenario — the canonical template for "single-attempt success" tests.

class InitialWorkflow(ScenarioBase):
    name = "pipeline.initial_workflow"
    expected_event_sequence = (
        PLANNER_INVOKED, PLANNER_COMPLETES_GOAL_PLAN,
        EXECUTOR_INVOKED, EXECUTOR_SUCCESS,
        EVALUATOR_INVOKED, EVALUATOR_SUCCESS,
    )

    planner_response(ctx)   -> ToolCallSpec(submit_plan_closes_goal,
                                            preflight_full_plan())
    executor_actions(ctx)   -> ("preflight",)            # runs shell pwd + git probe
    evaluator_response(ctx) -> ToolCallSpec(submit_evaluation_success,
                                            {"summary": "...",
                                             "passed_criteria":
                                                 list(ctx.attempt.evaluation_criteria)})

Run flow under the mock runner:
  PLANNER_INVOKED  -> planner_response -> submit_plan_closes_goal
  PLANNER_COMPLETES_GOAL_PLAN
  EXECUTOR_INVOKED -> executor_actions ("preflight") -> shell tool probe
                                                     -> submit_execution_success
  EXECUTOR_SUCCESS
  EVALUATOR_INVOKED -> evaluator_response -> submit_evaluation_success
  EVALUATOR_SUCCESS

Branch on ctx.iteration.sequence_no and ctx.attempt.attempt_sequence_no to cover more configurations from the same scenario.
