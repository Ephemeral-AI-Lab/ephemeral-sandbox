This wiki maps the Rust model-facing tool crate to the engine dispatch loop, sandbox workspace routing, Workflow terminal state, helper launches, and background execution behavior. Legacy Python tool folders are parity/history references unless a page explicitly names test_runner.

Scope And Ownership

Tool surface

eos-tools owns the agent-facing tool names, specs, metadata, execution lifecycle, terminal submissions, helper launches, workflow tools, isolated workspace tools, skills, and sandbox wrappers.

Engine integration

eos-engine query and tool-call modules decide when tool calls execute, defer, run in background, or terminate the agent loop.

Sandbox dispatch

Sandbox wrappers call typed eos-sandbox-api requests, then Rust daemon routing chooses LayerStack/OCC, per-call overlay, command session, plugin PPC, or explicit isolated workspace execution.

Workflow boundary

Submission and terminal tools are the persistent handoff channel between agents and Workflow state: root outcomes, planner rows, generator DAG tasks, reducer outcomes, delegated workflows, and helper outputs.

1
Agent registry is built.

Runtime agent definitions select allowed tools and terminals. eos-tools exposes ToolName, specs, metadata, intent, background policy, terminal flag, hooks, and context requirements.

2
Provider request exposes schemas.

The query request converts registered tools to API schemas. Optional non-terminal background tools receive an LLM-facing background input; terminal tools remain exclusive.

3
Streaming and final-message dispatch are coordinated.

Mid-stream execution is allowed only when deferral rules permit it. If terminal tools are present, the query loop defers all tool calls until the full assistant message can be validated.

4
One tool lifecycle runs.

Rust tool execution parses input, runs configured pre-hooks, executes the body, validates output, and stamps successful terminal tools with is_terminal=True. Post-hooks are an accepted deferred capability, not part of the current Rust execution path.

5
Result ownership branches.

Terminal results stop the agent loop. Background launches create bg_N records. Sandbox tools cross into daemon workspace routing. Submission tools mutate Workflow state or return helper metadata.

Critical Routes To Keep Straight

Terminal tools

A successful terminal call becomes is_terminal=True, wins the query-loop stop condition, and must not be batched with sibling tool calls.

Async sessions

run_subagent uses subagent_session_id with typed progress/cancel controls. command sessions use command_session_id. Generic background check/wait/cancel tools are retired from the model surface.

Ephemeral workspace

Without an isolated handle, command/search and non-fast-path operations use a per-call overlay. Write-capable captures publish through OCC when they should affect the shared workspace.

Isolated workspace

enter_isolated_workspace is explicit, persistent for the session, and scratch-only. Writes are audited as changed paths but discarded on exit instead of published through OCC.
