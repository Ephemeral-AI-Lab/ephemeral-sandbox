Terminal Tools

Terminal tools are one-way submissions that close the current agent turn. Their text guidance lives in backend/src/tools/_terminals; their runtime behavior is enforced by the tool framework and engine query loop.

Descriptor Registry #

TERMINAL_DESCRIPTORS maps each terminal tool name to a TerminalToolDescriptor with two prose views: selection_guidance for the parent agent and advisor_review_focus for advisor review. render_terminal_catalog(terminals, focus=...) renders a bullet list for the selected terminal names and returns a fallback bullet for unknown names.

The registry is shared instead of duplicated in profile prose. workflow.context_engine.skill_message appends the same <terminal_tool_selection> block to task guidance and skill messages using render_terminal_catalog(..., focus="selection_guidance"). Advisor tooling can render the same terminal set through the review-focus lens.

The descriptor completeness test scans main agent profile frontmatter and fails if a referenced terminal lacks a descriptor or if descriptor guidance fields are blank. That makes prompt drift visible at unit-test time instead of after an agent launch.

Terminal Exclusivity #

A terminal tool is any registered BaseTool whose is_terminal_tool flag is true. At query-loop startup, engine.query.loop derives context.terminal_tools from the active registry. When that set is non-empty, the stream executor defers all tool execution until the full assistant message is available.

Deferral matters because engine.tool_call.dispatch rejects a batch where a terminal tool appears with any sibling tool call. The rejection returns tool-result errors for the batch and executes none of the tools, preserving the terminal invariant before foreground, background, sandbox, or Workflow side effects can happen.

For valid batches, successful terminal execution is stamped in execute_tool_once() by replacing the final ToolResult with is_terminal=True. Failed terminal calls are not stamped, so the agent can receive the error and retry with corrected input.

Query Loop Stop #

  1. The provider stream completes and the final assistant message is appended to the transcript.
  2. dispatch_assistant_tools() executes deferred tool calls or rejects invalid terminal batches.
  3. _first_terminal_tool_result() finds the first result block with is_terminal=True and returns it as the dispatch terminal result.
  4. engine.query.loop copies that result into context.terminal_result, sets context.exit_reason = QueryExitReason.TOOL_STOP, and returns without appending another tool-result user message.
  5. The loop exits; pending background tasks are cancelled in the finally block if any remain.

This is why terminal tools are suitable for final Workflow submissions: the result becomes structured lifecycle data rather than just another conversational message.

Workflow Terminal Profiles #

Workflow no longer filters terminal frontmatter at launch. Main profiles expose one unified outcome terminal per role, while workflow-depth restrictions live at call time in submission prehooks and in one-shot notification triggers injected after initial launch messages. The registered profile is the effective profile.

Prompt rendering and real tool registration see the same terminal set because the profile itself is the effective definition. The terminal registry provides the shared selection text for those tools.
