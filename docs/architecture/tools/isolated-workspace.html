backend/src/tools/isolated_workspace exposes the explicit lifecycle tools for opening and closing a private sandbox workspace for one agent. The tools are registered with the sandbox tool set, but their behavior is lifecycle-oriented: enter_isolated_workspace pins a layer-stack snapshot and creates a persistent private namespace; exit_isolated_workspace cancels/drains per-agent background work, tears down the namespace and scratch state, and discards changes.

  • Isolated mode is opt-in per agent, not the default shared collaboration path.
  • While open, typed workspace verbs route through the isolated pipeline before the shared layer-stack/OCC fast path.
  • Writes are captured for audit but are not published to OCC.
  • Plugin and LSP operations are blocked for that agent while the isolated workspace is active.

Enter Tool

enter_isolated_workspace accepts an optional layer_stack_root. If the input is empty, the tool pulls the root from the sandbox execution context, builds an EnterIsolatedWorkspaceRequest with caller identity, and delegates to sandbox.host.isolated_workspace_lifecycle.enter_isolated_workspace.

  1. The host lifecycle layer counts local background tasks for the same agent and asks the daemon for in-flight sandbox-bound requests.
  2. If any are active, enter returns ephemeral_jobs_in_flight instead of racing an isolated namespace against shared overlay work.
  3. With a sandbox id, the host calls daemon op api.isolated_workspace.enter; without one, tests can call the local pipeline registry directly.
  4. The isolated pipeline prepares a snapshot, allocates scratch upper/work state, starts or reuses the private namespace runtime, and records a handle keyed by agent id.
  5. The tool returns success plus manifest version/root hash, or a structured lifecycle error.

The resulting handle is persistent across tool calls. It represents a private scratch namespace over a pinned lower snapshot, not a publishable branch. The scratch tree is part of the isolated runtime state and is removed on exit.

Private Routing Override

Once the agent has an active isolated handle, workspace_tool_dispatch routes all typed workspace verbs through IsolatedPipeline.run_tool_call before attempting the default shared path. This applies to read_file, write_file, edit_file, shell, glob, and grep.

This routing makes isolated mode useful for low-confidence or experimental work: commands and file writes see a stable private workspace across calls, but peers in the default shared workspace continue collaborating through the normal layer-stack/OCC path.

Exit Tool

exit_isolated_workspace accepts grace_s, builds an ExitIsolatedWorkspaceRequest, and delegates to the host lifecycle layer. The host first asks the background manager to cancel and drain per-agent background work, then calls daemon op api.isolated_workspace.exit when running against a sandbox daemon.

The daemon-side exit path tears down the handle: namespace/runtime resources are closed, the snapshot lease is released, scratch upper/work data is removed, and lifecycle metrics such as evicted upperdir bytes and lifetime seconds are returned. Pre-flight tests enforce that exit and teardown do not call OCC publish-shaped primitives.

Discard boundary Exit is a discard boundary. Current isolated workspace changes are intentionally not merged into the shared workspace by this tool. A future promotion workflow would need a separate explicit design; this page documents the current no-publish lifecycle.

Plugin, LSP, and Background Gates

The lifecycle layer protects the transition into and out of isolated mode. Enter rejects active sandbox-bound background work for the same agent by combining the local background manager count with daemon in-flight count. Exit cancels and drains per-agent background tasks before tearing down the isolated handle.

The daemon dispatcher blocks plugin-family operations when the same agent has an open isolated workspace, returning forbidden_in_isolated_workspace. This includes the plugin/LSP path because those operations maintain their own projection and long-lived state that should not be mixed with the private scratch namespace.

Two reusable tool-boundary prehooks (backend/src/tools/_hooks) front-line the same invariants before the tool body runs. They are deliberately layered with — not a replacement for — the authoritative lifecycle-layer checks above; a prehook rejection is a hook_failure ToolResult, distinct from a LifecycleError.

  • RequireNoInflightBackgroundTasks is wired on enter_isolated_workspace, exit_isolated_workspace, and the nine main terminals (ordered before AdvisorApprovalPreHook, so the background rejection is the one surfaced). It rejects when max(local count_by_agent, daemon inflight_count) > 0 for the calling agent (running, sandbox-bound tasks only). The exit gate makes exit_isolated_workspace refuse while background work is in flight rather than silently draining it — the agent cancels via cancel_background_task, then retries; the lifecycle drain remains as defense-in-depth and closes the check→complete race. On a daemon error with a zero local count the gate fails safe (block) for success/handoff terminals but fails open for the bail-out set (submit_execution_blocker, submit_evaluation_failure, submit_verification_failure, submit_plan_defers_goal) so a flaky daemon can never trap the agent's bail path.
  • BlockInIsolatedMode is wired on ask_advisor. It reads isolated-state from the daemon (the same get_handle truth as the plugin gate) via the thin sandbox.api.isolated_active wrapper over the existing api.isolated_workspace.status op — no engine mirror flag. It fails open on a daemon error because ask_advisor is read-only. Consequence: because every main terminal requires a prior advisor approval and ask_advisor is blocked while isolated, terminal submission is impossible while isolated — the agent must exit_isolated_workspace first (private scratch; exit to publish/handoff).

Engine, Sandbox, and TaskCenter Workflow

From the engine perspective, the lifecycle tools are regular tools with Intent.LIFECYCLE. They change the workspace routing state that subsequent tool calls observe; they do not themselves complete a TaskCenter attempt. Two enforcement layers keep that ordering invariant from drifting under concurrent dispatch:

  • Engine batch policy. backend/src/engine/tool_call/dispatch.py rejects any non-lifecycle sibling co-batched with an Intent.LIFECYCLE call (_record_lifecycle_batch_rejection); the lifecycle call still dispatches solo so the agent's intent advances, and siblings come back with an actionable is_error=True result instructing retry. When more than one lifecycle call appears in the batch, every lifecycle call is rejected because the engine cannot pick an ordering.
  • Daemon per-agent quiesce. backend/src/sandbox/daemon/workspace_tool_dispatch.py and the matching plugin gate in backend/src/sandbox/daemon/rpc/dispatcher.py wrap every routing-observing daemon RPC in acquire_dispatch_slot: a short-held per-agent entry_lock checks exit_pending and increments an inflight counter before the RPC body runs unlocked. exit_isolated_workspace drains that counter (begin_exit_drain) before mutating the handle maps, and _teardown runs only after the drain succeeds — closing the lockless-probe race regardless of which entry path the dispatch arrived on.

The engine still relies on terminal tools and submission state to mark completion, handoff, evaluation, or failure.

TaskCenter context should treat isolated workspace as a temporary execution mode attached to the agent's sandbox work. The durable state transition is not a peer handoff and not a global orchestrator event. It is an explicit private workspace lifecycle around normal tool calls, with results and terminal submissions later reflected through the standard attempt/goal state model.

The practical workflow is: enter isolated workspace, run typed sandbox tools against private scratch, inspect results, optionally discard by exit, then use normal shared workspace tools outside isolated mode for changes that should publish through OCC. The current exit tool always discards isolated scratch.
