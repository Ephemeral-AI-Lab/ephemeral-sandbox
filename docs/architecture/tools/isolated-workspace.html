Isolated Workspace Tools

enter_isolated_workspace and exit_isolated_workspace are explicit lifecycle tools for one agent's private sandbox workspace. In Rust, the agent-facing tool path is protected by RequireNoBackgroundSessions: lifecycle calls are refused while the agent still has outstanding workflows, subagents, or command sessions. After that guard passes, enter_isolated_workspace pins a layer-stack snapshot and creates a persistent private namespace; exit_isolated_workspace tears down the namespace and scratch state and discards changes.

The Rust daemon now has matching public lifecycle op names backed by sandbox/crates/eos-daemon/src/services/isolated_workspace/mod.rs and sandbox/crates/eos-isolated-workspace/src/session.rs. The current Rust slice is session/control-plane plus Linux file/command routing with no OCC publish, and it now includes the holder/setns/network handoff: sandbox/crates/eos-ns-holder/src/lib.rs unshares and handshakes the namespace holder, applies best-effort rtnetlink loopback-up/default-IPv6-route hardening plus namespace-side veth programming, the daemon opens inherited namespace FDs and mounts through eosd ns-runner --mount-overlay, sandbox/crates/eos-runner/src/setns.rs implements runner-side setns entry and setns overlay mounting, sandbox/crates/eos-daemon/src/services/command_session/mod.rs selects the isolated command policy, and eos-command-session owns command-session runtime lifecycle. sandbox/crates/eos-isolated-workspace/src/network.rs creates/moves/attaches the bridge/veth pair and installs static nftables NAT/filter rules through netlink on Linux. Local amd64 Docker/dask proof passed in a target image without ip or nft; the latest live rerun also asserts the cleanup inspection fields under the real kernel path, including holder, mountinfo, cgroup, lease, scratch, veth, and command-session cleanup.

  • Isolated mode is opt-in per agent, not the default shared collaboration path.
  • While open, typed workspace verbs route through the isolated pipeline before the shared direct LayerStack/OCC file ops.
  • Writes are captured for audit but are not published to OCC.
  • Plugin and LSP operations are blocked for that agent while the isolated workspace is active.

Enter Tool

enter_isolated_workspace accepts an optional layer_stack_root. If the input is empty, the Rust tool pulls the root from the sandbox execution context, builds an EnterIsolatedWorkspaceRequest with caller identity, and reaches the daemon through the executor-captured SandboxToolService. The prehook runs before the tool body, so raw daemon lifecycle calls are not the agent-facing policy boundary.

  1. RequireNoBackgroundSessions checks the three background-session lanes for the calling agent: outstanding delegated workflows, in-flight subagents, and daemon-visible command sessions.
  2. If any are active, enter returns a hook_failure tagged ephemeral_jobs_in_flight instead of racing an isolated namespace against shared or background work.
  3. After the guard passes, the runtime calls daemon op api.isolated_workspace.enter with the sandbox id and layer-stack root.
  4. The isolated pipeline prepares a snapshot, allocates scratch upper/work state, starts or reuses the private namespace runtime, and records a handle keyed by agent id.
  5. The tool returns success plus manifest version/root hash, or a structured lifecycle error.

The resulting handle is persistent across tool calls. It represents a private scratch namespace over a pinned lower snapshot, not a publishable branch. The scratch tree is part of the isolated runtime state and is removed on exit.

Private Routing Override

Once the agent has an active isolated handle, workspace verbs route through the private handle before attempting the default shared path. Python does this in workspace_tool_dispatch by calling IsolatedPipeline.run_tool_call. Rust does this in sandbox/crates/eos-daemon/src/workspace_ops.rs by checking the daemon isolated-workspace service before the shared LayerStack/OCC handlers for model-facing read_file, write_file, and edit_file. sandbox/crates/eos-daemon/src/services/command_session/mod.rs applies the same active-handle routing to exec_command by selecting the isolated policy before calling eos-command-session.

Rust file writes and edits write to retained isolated upperdir state, mark results workspace=isolated, record changed paths for audit, and skip OCC publish. Raw daemon glob and grep are not part of the Rust isolated routing surface. The daemon provides the holder and inherited namespace FD handoff into the runner's Linux setns mode, and the Linux bridge/veth plus nftables setup path uses target-gated netlink dependencies instead of image-local ip/nft binaries. Local amd64 Docker/dask evidence covers finite command, command-session start/progress/write/natural exit/timeout/cancel, peer shared-publish isolation, same-port 3000 isolation across two agents, bridge/veth, nftables NAT/filter, holder network hardening, status/list closure, and host-veth teardown. Enter and exit also run daemon command-session cleanup as a bypass backstop before lifecycle state changes, while the model-facing guard remains RequireNoBackgroundSessions. Exit inspection coverage now checks handle/agent maps, lease release, active lease count, holder process cleanup, mountinfo references, cgroup removal, scratch removal, and audit JSONL payloads under the real kernel path.

This routing makes isolated mode useful for low-confidence or experimental work: commands and file writes see a stable private workspace across calls, but peers in the default shared workspace continue collaborating through the normal layer-stack/OCC path.

Exit Tool

exit_isolated_workspace accepts grace_s and builds an ExitIsolatedWorkspaceRequest. The Rust tool first runs RequireNoBackgroundSessions. That hook may settle in-process subagent records, but it still refuses exit while outstanding workflows or daemon-visible command sessions remain; the agent must collect or cancel those sessions, then retry. After the guard passes, the runtime calls daemon op api.isolated_workspace.exit.

The daemon-side exit path tears down the handle: namespace/runtime resources are closed, the snapshot lease is released, scratch upper/work data is removed, and lifecycle metrics such as evicted upperdir bytes and lifetime seconds are returned. Pre-flight tests enforce that exit and teardown do not call OCC publish-shaped primitives.

The Rust daemon still records active command sessions against the isolated agent handle for routing, TTL, and cleanup bookkeeping. That record is not an isolated-workspace exit guard: live command sessions are blocked earlier by RequireNoBackgroundSessions through api.v1.command_session_count. If a caller bypasses the hook and invokes the lifecycle RPC directly, enter and exit cancel/finalize that agent's command sessions before mutating isolated state so the daemon does not leave dead or orphaned command-session records.

Discard boundary Exit is a discard boundary. Current isolated workspace changes are intentionally not merged into the shared workspace by this tool. A future promotion workflow would need a separate explicit design; this page documents the current no-publish lifecycle.

Plugin, LSP, and Background Gates

The Rust agent-facing lifecycle policy lives at the eos-tools boundary. RequireNoBackgroundSessions protects the transition into and out of isolated mode by refusing lifecycle calls while the calling agent has outstanding workflows, subagents, or command sessions. The daemon lifecycle remains the state authority for isolated handles and teardown.

The daemon dispatcher blocks plugin-family operations when the same agent has an open isolated workspace, returning forbidden_in_isolated_workspace. This includes the plugin/LSP path because those operations maintain their own projection and long-lived state that should not be mixed with the private scratch namespace.

Reusable tool-boundary prehooks front-line the same invariants before the tool body runs. They are deliberately layered with — not a replacement for — daemon state checks; a prehook rejection is a hook_failure ToolResult, distinct from a LifecycleError.

  • RequireNoBackgroundSessions is wired on enter_isolated_workspace, exit_isolated_workspace, and the four main submission terminals (ordered before AdvisorApprovalPreHook, so the background rejection is the one surfaced). It rejects while any of the three background-session lanes is still open for the calling agent: outstanding workflows from WorkflowControlPort::find_outstanding, in-flight subagents from the background supervisor, or daemon-visible command sessions from api.v1.command_session_count. The hook makes exit_isolated_workspace refuse while sessions are open at the tool boundary; the agent cancels or collects active work, then retries. On a daemon-count error the gate fails safe (block) for non-bailout terminals but fails open for bailout payloads (submit_generator_outcome(status="failed", ...), submit_reducer_outcome(status="failed", ...), or submit_planner_outcome with nonblank deferred_goal_for_next_iteration) so a flaky daemon can never trap the agent's bail path.
  • BlockInIsolatedMode is wired on ask_advisor. It reads isolated-state from the daemon (the same get_handle truth as the plugin gate) via the thin sandbox.api.isolated_active wrapper over the existing api.isolated_workspace.status op — no engine mirror flag. It fails open on a daemon error because ask_advisor is read-only. Consequence: because every main terminal requires a prior advisor approval and ask_advisor is blocked while isolated, terminal submission is impossible while isolated — the agent must exit_isolated_workspace first (private scratch; exit to publish/handoff).

Engine, Sandbox, and Workflow

From the engine perspective, the lifecycle tools are regular tools with Intent.LIFECYCLE. They change the workspace routing state that subsequent tool calls observe; they do not themselves complete a Workflow attempt. Two enforcement layers keep that ordering invariant from drifting under concurrent dispatch:

  • Engine batch policy. agent-core/crates/eos-engine/src/tool_call/dispatch.rs rejects any non-lifecycle sibling co-batched with an Intent.LIFECYCLE call (_record_lifecycle_batch_rejection); the lifecycle call still dispatches solo so the agent's intent advances, and siblings come back with an actionable is_error=True result instructing retry. When more than one lifecycle call appears in the batch, every lifecycle call is rejected because the engine cannot pick an ordering.
  • Daemon per-agent quiesce. sandbox/crates/eos-daemon/src/dispatch/dispatcher.rs and the matching plugin gate in the same daemon dispatch layer wrap every routing-observing daemon RPC in acquire_dispatch_slot: a short-held per-agent entry_lock checks exit_pending and increments an inflight counter before the RPC body runs unlocked. exit_isolated_workspace drains that counter (begin_exit_drain) before mutating the handle maps, and _teardown runs only after the drain succeeds — closing the lockless-probe race regardless of which entry path the dispatch arrived on.

The engine still relies on terminal tools and submission state to mark completion, handoff, evaluation, or failure.

Workflow context should treat isolated workspace as a temporary execution mode attached to the agent's sandbox work. The durable state transition is not a peer handoff and not a global orchestrator event. It is an explicit private workspace lifecycle around normal tool calls, with results and terminal submissions later reflected through the standard attempt/workflow state model.

The practical workflow is: enter isolated workspace, run typed sandbox tools against private scratch, inspect results, optionally discard by exit, then use normal shared workspace tools outside isolated mode for changes that should publish through OCC. The current exit tool always discards isolated scratch.
