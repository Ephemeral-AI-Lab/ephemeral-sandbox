run_subagent is the supervised async-session launch surface for focused worker agents without creating peer-to-peer orchestration.

Registration #

agent-core/crates/eos-tools/src/tools/subagent owns the caller-aware launch tool run_subagent and the typed control tools check_subagent_progress and cancel_subagent. The framework factory uses make_subagent_tool_from_context(), which wraps the base launch tool in RestrictedRunSubagentTool and narrows agent_name to registered dispatchable subagents.

The launch tool is declared with intent=Intent.WRITE_ALLOWED and task_type=SUBAGENT_TASK_TYPE. Agent finalization and dispatch use engine.background.policy to route run_subagent to the internal supervisor and synthesize typed subagent controls instead of generic check_background_task_result / wait_background_tasks controls.

Launch Workflow #

Parent callrun_subagent(agent_name, prompt)
Supervisor dispatchersingle id subagent_session_id
Validation gatesknown target, subagent type, no recursive spawn
Ephemeral child runstatic guidance plus caller prompt
Progress providercheck_subagent_progress peeks child messages
Child terminalterminal output becomes parent session result
Parent resumeschecks result, continues, or submits Workflow terminal
  1. The parent calls run_subagent(agent_name, prompt). Engine dispatch validates the input, assigns subagent_session_id, and uses that same id as the supervisor record key before starting the tool body as a supervised asyncio task.
  2. The tool rejects missing runtime config, blank prompts, unknown targets, targets that are not agent_type="subagent", and recursive calls from a caller whose metadata says agent_type="subagent".
  3. The tool imports run_ephemeral_agent through engine.api, builds subagent metadata with agent_type="subagent" and role information, and forwards the active sandbox id.
  4. The child does not inherit the parent's Workflow AgentContext. It receives the parent's free-text prompt as its initial user message and a static subagent guidance prompt as the run prompt.
  5. persist_agent_run=False keeps the helper run ephemeral. The parent remains responsible for final Workflow submission through its own terminal tools.

Progress and Results #

When the child agent is spawned, run_subagent registers a supervisor progress provider under the same subagent_session_id. The provider snapshots the child agent's latest conversation messages and renders text, thinking, tool-use, and tool-result blocks for check_subagent_progress(subagent_session_id, last_n_messages).

The child must call one of its registered terminal tools. run_ephemeral_agent returns an EphemeralRunResult; run_subagent forwards terminal_result.output, is_error, and terminal metadata to the parent, adding subagent_terminal_called=true. The initial launch response stays status=running with the public session id; the terminal output is later surfaced by the typed completion notification and by check_subagent_progress. If the child crashes or exits without a terminal result, the metadata marks subagent_terminal_called=false, and the typed progress tool reports the session as failed with the latest peek instead of pretending the findings were delivered.

Cancellation is also subagent-aware. cancel_subagent(subagent_session_id) asks the supervisor to request an early stop for that session, cancel the child task, and allow the child path to salvage a partial result before it settles. If the parent run submits a terminal while a subagent is still active, the query loop terminates that subagent as non_cancellation_tool_request and emits one typed notification plus audit reason instead of leaving an anonymous abandoned worker.

Workflow Boundary #

run_subagent is a worker launch tool, not a global orchestrator. It lets the parent ask for focused exploration or implementation help while Workflow state still advances through persisted terminal submissions, rendered role context, lifecycle transitions, and attempt state machines.

The parent receives the subagent's terminal output through the typed subagent session surface. The parent then decides whether to continue, call another tool, ask a helper, or submit a Workflow terminal tool. This keeps task handoff and completion state in Workflow rather than creating direct agent-to-agent control flow.

Sandbox behavior follows the child run's ordinary tool execution. If the child uses sandbox tools, those calls still route through sandbox APIs, the daemon workspace dispatcher, the ephemeral workspace overlay pipeline, direct LayerStack/OCC fast paths, or isolated workspace lifecycle according to the active workspace mode.

Verification #

Key tests cover factory policy, background session service exposure, subagent_session_id launch/progress/cancel behavior, retry behavior through run_ephemeral_agent, terminal metadata projection, failed/no-terminal outcomes, mocked query-loop natural completion, explicit cancel, parent terminal abandonment, concurrent subagent calls, status rendering, and event-printer display. Refresh this page when subagent target registration, terminal metadata, progress-provider rendering, run-finalization cleanup, or cancellation semantics change.
