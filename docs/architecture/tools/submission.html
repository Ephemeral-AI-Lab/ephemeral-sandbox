Submission Tools

Contents

Submission tools are terminal adapters under backend/src/tools/submission. They validate model output, resolve Workflow context, and delegate lifecycle mutation to Workflow attempt, workflow, and run services. After the reducers redesign there are three roles that mutate attempt state — planner, generator (executor), and reducer — plus two terminals that only return metadata to their caller: the advisor (helper) and the explorer (subagent). The old verifier and evaluator submission tools are gone.

Registration Map #

make_submission_tools() returns the terminal submission tools registered by the framework factory: root outcome, planner outcome, generator outcome, reducer outcome, advisor feedback, and exploration result. delegate_workflow, check_workflow_status, and cancel_workflow are registered separately as non-terminal workflow tools; ask_advisor is the separate helper request tool used before main-role terminal submissions.

The reducer is a plan task that submits a verdict (success / failed) on the slice it gates, replacing both the per-task verifier and the per-attempt evaluator. The registration tests assert legacy submission aliases are absent. Submission tools are terminal; workflow tools are intentionally non-terminal.

Planner, Executor, Reducer #

submit_planner_outcome resolves an attempt-bound submission context, builds a validated PlannerSubmission from flat tasks (each PlanTaskInput carries id, agent_name, and generator-only needs), task_specs, and reducers (each ReducerInput carries id, one or more generator needs, and a required prompt), then calls AttemptOrchestrator.apply_plan_submission(). The optional deferred_goal_for_next_iteration field names current-iteration goal items intentionally left for the next iteration; omitted or null means the plan covers all current-iteration goal items and leaves no remaining items.

submit_generator_outcome(status, outcome) resolves GeneratorSubmissionContext, which wraps the generator task row, runtime dependencies, and attempt context. It calls apply_generator_submission() with status="success" or status="failed"; the terminal result still records generator_role="executor" because the current generator profile name is executor.

submit_reducer_outcome(status, outcome) resolves the strict resolve_attempt_submission_context() and calls AttemptOrchestrator.apply_reducer_submission() with a ReducerSubmission whose status is the literal "success" or "failed". The reducer is the attempt's exit gate: it completes <assigned_task> using <dependencies>. After either submission the orchestrator runs advance_ready_tasks() over the single RUN stage; the attempt passes once every plan task is DONE and fails (TASK_FAILED) if any task failed or is unreachable behind a failed dependency. There is no separate evaluate stage and no per-task verifier.

Delegated Workflow Tools #

  1. delegate_workflow(goal=...) validates a nonblank goal, resolves the current running Task from tool metadata, and calls WorkflowStarter.start(prompt=goal, parent_task_id=current_task_id).
  2. WorkflowStarter validates that the parent Task is RUNNING and has no open delegated workflow for that parent, then creates a Workflow, initial Iteration, and initial Attempt. The parent Task stays RUNNING; there is no legacy waiting-state transition and no legacy delegation link column.
  3. The workflow tool returns a workflow_task_id handle and workflow_id. Because the tool is non-terminal, the caller continues its own agent loop and later uses check_workflow_status or cancel_workflow.
  4. Closing a delegated workflow persists workflow outcomes only. It does not mutate the parent generator Task; the launching agent reads the handle result and synthesizes its own submit_generator_outcome.
  5. The no-inflight terminal hook counts outstanding workflow handles. A generator/root terminal is allowed once the handle is reported by check_workflow_status or explicitly cancelled by cancel_workflow.

This flow is the workflow-delegation boundary: agents do not communicate peer-to-peer, but they also do not terminate to delegate. The root request is a root Task finished by submit_root_outcome, not a workflow-wrapped request.

Helper and Explorer Results #

submit_advisor_feedback (the helper/advisor terminal) and submit_exploration_result (the subagent/explorer terminal) are terminal tools, but they do not resolve an attempt context or call workflow mutators. They return bounded output text plus metadata that the caller can interpret: advisor verdict and role, or explorer findings and references.

This distinction is intentional. Helper agents terminate their own helper run and provide structured metadata to the parent workflow, while workflow attempt state remains owned by the planner, executor, and reducer submission paths and the orchestrator.

Workflow Connections #

frameworkEvery submit_* function is a decorated BaseTool with is_terminal_tool=True; successful execution is stamped by execute_tool_once().
engineTerminal exclusivity and TOOL_STOP are enforced in engine.tool_call.dispatch and engine.query.loop, not in individual submission tools.
workflowAttempt state changes are centralized in AttemptOrchestrator (apply_plan_submission, apply_generator_submission, apply_reducer_submission). Delegation state starts through WorkflowStarter.start() and is observed through workflow handles; closing a workflow does not mutate the parent Task.
sandboxSubmission tools use Intent.READ_ONLY. They do not perform workspace file mutations directly; generator work that touched the sandbox should already have gone through sandbox tools before terminal submission.
promptingProfile terminal frontmatter and the shared terminal descriptor registry decide which terminal options appear in workflow task guidance and skills; nested planner deferral limits are enforced by submission prehooks and soft notifications.
  • Submission tools are adapters, not orchestrators.
  • Workflow delegation uses persisted workflow handles, not direct agent-to-agent calls.
  • Helper terminal outputs return metadata to their caller without mutating workflow attempt state.
