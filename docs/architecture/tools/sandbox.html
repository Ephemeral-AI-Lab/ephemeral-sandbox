agent-core/crates/eos-tools/src/tools/sandbox is the current Rust agent-facing tool folder for file, command, and isolated-workspace work inside a sandboxed workspace. Its registry returns read_file, write_file, edit_file, multi_edit, exec_command, and write_stdin, and it also includes the isolated workspace lifecycle tools in the same sandbox context so the engine can register one coherent sandbox tool surface. Legacy Python exposed glob and grep; Rust intentionally deletes them from both the model-facing tool registry and the raw daemon/protocol/runner surface.

  • Read/write/edit wrappers normalize tool input and call the public sandbox.api facade.
  • exec_command is write-capable and runs destructive command pre-hooks before daemon dispatch; model-facing background command work uses exec_command and typed command-session controls.
  • The public API facade sends api.v1.* daemon operations over the sandbox transport.
  • The daemon routes each workspace tool through workspace_tool_dispatch, which prefers an active isolated workspace, then direct layer-stack/OCC file paths, then ephemeral overlay execution.

Tool Surface

The registry is intentionally narrow: it returns the default workspace verbs plus command/session controls and enter_isolated_workspace / exit_isolated_workspace, then stamps every tool with SANDBOX_CONTEXT. That means all of these tools expect the engine/tool framework to provide the sandbox id, caller metadata, layer-stack root, repo root, invocation id, audit sink, and background manager in the execution context.

The read/write/edit wrappers construct typed request models and delegate to the API facade rather than touching provider or filesystem code directly. Write and edit route through a shared mutation normalization layer so status, conflicts, changed paths, mutation source, and timing fields stay stable for the engine and Workflow context consumers.

API and Daemon Hop

sandbox.api.tool is the public facade between tools and the in-sandbox daemon. Each verb builds a JSON-safe daemon payload from the typed request identity fields, calls the transport operation, parses the daemon response back into a typed result, and wraps the call in audited operation metadata.

[agent sandbox tool]
        |
        v
[engine tool dispatch]
        |
        v
[sandbox tool registry]
        |
        v
[daemon client call]
        |
        v
[workspace_tool_dispatch]
        |
        +--> [LayerStack/OCC fast path]
        +--> [overlay pipeline]
        +--> [isolated workspace pipeline]
  1. The engine dispatches a tool call through the tool framework with SANDBOX_CONTEXT.
  2. The tool wrapper builds a typed request and calls sandbox.api.
  3. The facade calls daemon op names such as api.v1.read_file, api.v1.write_file, api.v1.edit_file, api.v1.shell, api.v1.exec_command, api.v1.write_stdin, api.v1.command.cancel, and api.v1.command.collect_completed. Rust does not register raw daemon search ops.
  4. daemon.rpc.dispatcher validates the envelope, registers in-flight state, blocks plugin ops when this agent has an active isolated workspace, and invokes the registered handler.
  5. daemon.builtin_operations maps workspace ops to dispatch_workspace_tool_call with the static verb and intent.

The daemon operation table is deliberately generic: the handler does not know whether a request will use layer-stack/OCC or overlay. It only establishes the verb, intent, in-flight lifecycle, and error envelope. The workspace dispatcher owns routing.

The old Rust grep/glob model-tool and raw daemon schemas are intentionally gone. Search-like work now runs through explicit shell commands or a future dedicated tool rather than hidden daemon primitives.

Default Shared Workspace Routing

For the default shared workspace, workspace_tool_dispatch first checks whether the agent has an active isolated pipeline. If not, it tries a direct layer-stack/OCC path for read_file, write_file, and edit_file when the layer-stack root has a workspace binding. That direct path avoids a per-call overlay mount for single-file operations.

Fast readread_file resolves the requested path through the workspace binding and reads text from the active layer-stack manager.
Fast writewrite_file turns the final content into an API write change and applies it through the OCC service.
Fast editedit_file turns each search/replace edit into an OCC edit change and reports applied edit count from the changeset result.
FallbackUnbound file requests, finite commands, outside-workspace paths, and other overlay-shaped work use the ephemeral overlay pipeline.

The ephemeral pipeline acquires a fresh overlay handle for one call, executes the tool payload in a new namespace, captures upperdir changes for write-allowed calls, publishes accepted changes through OCC, attaches timing/resource metadata, and destroys the handle. Write-capable overlay results must pass through the OCC-gated publish boundary before becoming part of the shared layer-stack state.

Command Sessions and Policy

exec_command is declared WRITE_ALLOWED and is the model-facing replacement for the deleted public shell tool package. There is no public mode switch switch: completed commands return directly, while still-running commands return command_session_id and are controlled with write_stdin. Generic background=true and generic background check/wait/cancel tools are not exposed.

Before daemon dispatch, exec_command runs destructive git and destructive shell pre-hooks through the Tool Hooks pipeline. These hooks block command forms that would bypass the sandbox write/audit path or destructively mutate workspace/system state. That policy keeps commands useful for builds and tests while steering file edits toward edit_file and write_file.

Session connection The engine supervisor tracks long-running command sessions for completion notifications and isolated-workspace lifecycle gates, while eos-command-session owns runtime session state behind the daemon adapter. Generic background management tools are no longer exposed to the model.

Engine and Workflow Connections

The sandbox tools are ordinary framework tools from the engine's perspective. The engine loop and tool-call dispatch layer enforce tool execution, background wrapping, and terminal-tool completion rules; sandbox tools provide the workspace mutation/search/command substrate those tasks use while working toward terminal submission.

Workflow does not receive peer-to-peer messages from sandbox tools. It observes persisted workflow, iteration, attempt, task, terminal, and context state. The important contract is that sandbox tool results carry enough structured metadata for later role context: status, conflicts, changed paths, timings, and background task state. Terminal tools then mark completion and handoff state; sandbox tools only mutate or inspect the workspace that those terminal decisions refer to.

The isolated workspace lifecycle tools share this registry context, but their routing semantics differ: once an agent enters isolated mode, typed sandbox verbs for that agent are intercepted by the isolated pipeline before the shared layer-stack/OCC path. See Isolated Workspace for the private scratch workflow and exit behavior.
