backend/src/tools/sandbox is the agent-facing tool folder for file, search, and shell work inside a sandboxed workspace. Its registry returns read_file, write_file, edit_file, shell, glob, and grep, and it also includes the isolated workspace lifecycle tools in the same sandbox context so the engine can register one coherent sandbox tool surface.

  • Read/write/edit wrappers normalize tool input and call the public sandbox.api facade.
  • shell is write-capable, can be backgrounded by the tool framework, and runs destructive command pre-hooks before daemon dispatch.
  • The public API facade sends api.v1.* daemon operations over the sandbox transport.
  • The daemon routes each workspace tool through workspace_tool_dispatch, which prefers an active isolated workspace, then direct layer-stack/OCC file paths, then ephemeral overlay execution.

Tool Surface

The registry is intentionally narrow: it returns the six default workspace verbs plus enter_isolated_workspace and exit_isolated_workspace, then stamps every tool with SANDBOX_CONTEXT. That means all of these tools expect the engine/tool framework to provide the sandbox id, caller metadata, layer-stack root, repo root, invocation id, audit sink, and background manager in the execution context.

The read/write/edit wrappers construct typed request models and delegate to the API facade rather than touching provider or filesystem code directly. Write and edit route through a shared mutation normalization layer so status, conflicts, changed paths, mutation source, and timing fields stay stable for the engine and TaskCenter context consumers.

API and Daemon Hop

sandbox.api.tool is the public facade between tools and the in-sandbox daemon. Each verb builds a JSON-safe daemon payload from the typed request identity fields, calls the transport operation, parses the daemon response back into a typed result, and wraps the call in audited operation metadata.

[agent sandbox tool]
        |
        v
[engine tool dispatch]
        |
        v
[sandbox tool registry]
        |
        v
[daemon client call]
        |
        v
[workspace_tool_dispatch]
        |
        +--> [LayerStack/OCC fast path]
        +--> [overlay pipeline]
        +--> [isolated workspace pipeline]
  1. The engine dispatches a tool call through the tool framework with SANDBOX_CONTEXT.
  2. The tool wrapper builds a typed request and calls sandbox.api.
  3. The facade calls daemon op names such as api.v1.read_file, api.v1.write_file, api.v1.edit_file, api.v1.shell, api.v1.exec_command, api.v1.pty.write_stdin, api.v1.pty.progress, api.v1.pty.cancel, api.v1.glob, and api.v1.grep.
  4. daemon.rpc.dispatcher validates the envelope, registers in-flight state, blocks plugin ops when this agent has an active isolated workspace, and invokes the registered handler.
  5. daemon.builtin_operations maps workspace ops to dispatch_workspace_tool_call with the static verb and intent.

The daemon operation table is deliberately generic: the handler does not know whether a request will use layer-stack/OCC or overlay. It only establishes the verb, intent, in-flight lifecycle, and error envelope. The workspace dispatcher owns routing.

Default Shared Workspace Routing

For the default shared workspace, workspace_tool_dispatch first checks whether the agent has an active isolated pipeline. If not, it tries a direct layer-stack/OCC path for read_file, write_file, and edit_file when the layer-stack root has a workspace binding. That direct path avoids a per-call overlay mount for single-file operations.

Fast readread_file resolves the requested path through the workspace binding and reads text from the active layer-stack manager.
Fast writewrite_file turns the final content into an API write change and applies it through the OCC service.
Fast editedit_file turns each search/replace edit into an OCC edit change and reports applied edit count from the changeset result.
FallbackUnbound file requests, shell, glob, grep, outside-workspace paths, and other overlay-shaped work use the ephemeral overlay pipeline.

The ephemeral pipeline acquires a fresh overlay handle for one call, executes the tool payload in a new namespace, captures upperdir changes for write-allowed calls, publishes accepted changes through OCC, attaches timing/resource metadata, and destroys the handle. Read-only search work can use the overlay execution shape without publishing. Write-capable overlay results must pass through the OCC-gated publish boundary before becoming part of the shared layer-stack state.

Shell Background and Policy

shell is declared WRITE_ALLOWED and background="optional". The tool wrapper does not expose a separate background parameter in its input model; instead, it reads background_task_id from the tool execution context and sets ShellRequest.background. The API facade forwards that as background=true in the daemon payload, and the daemon in-flight registry tracks the invocation by agent id and background flag.

Before daemon dispatch, shell runs destructive git and destructive shell pre-hooks through the Tool Hooks pipeline. These hooks block command forms that would bypass the sandbox write/audit path or destructively mutate workspace/system state. That policy keeps shell useful for builds and tests while steering file edits toward edit_file and write_file.

Background connection Background management tools live outside backend/src/tools/sandbox, but the sandbox shell is one of their main producers. wait_background_tasks, check_background_task_result, and cancel_background_task operate on the background task manager state that the engine/tool framework attaches to the execution context.

Engine and TaskCenter Connections

The sandbox tools are ordinary framework tools from the engine's perspective. The engine loop and tool-call dispatch layer enforce tool execution, background wrapping, and terminal-tool completion rules; sandbox tools provide the workspace mutation/search/shell substrate those tasks use while working toward terminal submission.

TaskCenter does not receive peer-to-peer messages from sandbox tools. It observes persisted workflow, iteration, attempt, task, terminal, and context state. The important contract is that sandbox tool results carry enough structured metadata for later role context: status, conflicts, changed paths, timings, and background task state. Terminal tools then mark completion and handoff state; sandbox tools only mutate or inspect the workspace that those terminal decisions refer to.

The isolated workspace lifecycle tools share this registry context, but their routing semantics differ: once an agent enters isolated mode, typed sandbox verbs for that agent are intercepted by the isolated pipeline before the shared layer-stack/OCC path. See Isolated Workspace for the private scratch workflow and exit behavior.
