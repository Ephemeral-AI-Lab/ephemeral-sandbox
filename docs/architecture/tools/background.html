Engine-owned async task plumbing for hard-coded background surfaces: subagent launches and command sessions.

Dispatch Policy #

Background execution is an engine dispatch mode, not a separate tool runtime or BaseTool policy. engine.background.policy hard-codes the surfaces that need the background manager: run_subagent is deferred through dispatch_background_tool_call(), while command sessions attach to the same supervisor through typed command-session plumbing.

Engine policyrun_subagent / command sessions
Streaming deferralforeground executor skips background-bound calls
Final dispatchervalidates clean input and assigns typed ids
Supervisor recordtask, status, progress, sandbox binding
Running taskprogress callback or provider snapshot
Typed deliverynotifies completion or renders running status
Cancelasyncio cancel plus sandbox cancel when bound
Deliveredterminal state is surfaced once
Subagent launch
run_subagent is always deferred by the engine policy and launched through the supervisor with subagent_session_id as both record id and model-facing id.
command sessions
exec_command and typed command-session controls keep the manager available so interactive commands can be checked, written to, cancelled, and auto-notified by command_session_id.
Rejected tools
launch_background_tool() rejects any tool that is not allowed by engine.background.policy.is_engine_background_tool().
Executor boundary
The streaming executor only asks a predicate whether to defer. Background policy remains in engine dispatch, not in provider stream consumption or tool schema decoration.

Record Lifecycle #

BackgroundSessionRuntime owns the in-memory session managers for one agent run. Subagent sessions use subagent_session_id, workflow sessions use the delegated workflow handle, and command sessions use command_session_id. Each session stores the handle, status, session metadata, live progress, and final ToolResult where that background family produces one.

StatusMeaningDelivery rule
RUNNINGThe asyncio task is still active.Typed control tools return progress lines or a progress-provider snapshot.
COMPLETEDThe task produced a successful ToolResult.Undelivered until the matching typed delivery path marks it delivered.
FAILEDThe task raised or returned an error result through the done callback.Undelivered until the matching delivery path marks it delivered.
CANCELLEDThe supervisor marked the task cancelled.Undelivered until delivered; completion has higher terminal precedence than cancellation in races.
DELIVEREDThe terminal state has already been surfaced to the model.Not collected again.

The terminal-status latch orders terminal states as completed over failed over cancelled over running. That keeps a near-complete sandbox command from being overwritten by a late cancel acknowledgement. progress_provider is a pull callback for structured progress; run_subagent uses it to expose the child agent's latest rendered messages.

Management Tools #

The generic background management tools are retired from model-facing runtime and catalog exposure. The internal background session service remains, while command-session command tools and subagent controls expose typed ids. Subagent controls live in agent-core/crates/eos-tools/src/tools/subagent: check_subagent_progress and cancel_subagent.

Retired generic toolscheck_background_task_result, wait_background_tasks, and cancel_background_task are no longer synthesized for agents or catalog output.
Command-session controlswrite_stdin writes non-empty input and waits briefly for output; read_command_progress reads progress without stdin; daemon cancel/collect/count ops stay internal.
Subagent controlscheck_subagent_progress and cancel_subagent operate on subagent_session_id.
Workflow controlsdelegate_workflow returns a typed workflow handle inspected by check_workflow_status and stopped with cancel_workflow.

The model-facing surfaces are typed. Subagents, delegated workflows, and eos-command-session runtime sessions have completion bridges: SubagentSessionMonitor, WorkflowSessionMonitor, and CommandSessionMonitor poll their managers and emit one typed system notification through BackgroundNotificationEmitter. If a typed control tool already reports a terminal result, the owning manager marks that session delivered so the automatic bridge does not duplicate it.

Sandbox Dispatch #

The engine background launcher is now subagent-only. Command sessions are created by exec_command; the tool registers the returned command_session_id with CommandSessionManager for lifecycle gates, cancellation, completion notifications, and isolated-workspace blocking.

Daemon command-session controls call typed sandbox APIs: write_stdin sends non-empty input and waits up to yield_time_ms, read_command_progress returns a stateless tail snapshot, and internal api.v1.command.cancel stops a daemon-owned session. Empty stdin polling is not exposed through write_stdin. These controls do not pass through generic background check/wait/cancel tools.

This is separate from the workspace execution mode itself. exec_command still routes through the sandbox command execution path and may publish captured changes through OCC; command-session state only changes how the long-running command is supervised and controlled.

Engine Workflows #

  1. The agent factory enables background tasks when the registry includes run_subagent or command session tools. It synthesizes typed subagent controls only for the run_subagent surface.
  2. AgentRunControlFactory creates one BackgroundSessionService for the run; tool metadata receives the narrow background and command-session ports.
  3. Provider streaming defers engine-background policy calls so the final assistant batch can still enforce terminal-tool exclusivity.
  4. Deferred subagent calls are validated, given subagent_session_id, launched as asyncio tasks, and acknowledged with [SUBAGENT LAUNCHED]. exec_command returns command_session_id, and delegate_workflow returns a typed workflow handle. The old generic background management tools are no longer exposed to the model.
  5. The model continues the same run and may call typed subagent or command-session controls. Terminal tools remain Workflow-facing completion signals and must still be called through the normal terminal-tool path.
  6. On resource-limit exit or query-loop finalization, pending background tasks are cancelled so engine exits do not leave orphaned sandbox work.

Verification #

Unit coverage is concentrated in agent-core/crates/eos-engine/tests background-task and terminal-latch tests, and exec-command toolkit tests. Subagent-specific background behavior is also covered by agent-core/crates/eos-tools/tests/tools/subagent. Refresh this page when background schema policy, supervisor state transitions, typed command-session controls, typed workflow handles, or typed subagent controls change.
