6. Daemon and Background Tasks §

  • Sandbox operations need resident state, cached managers, cancellation, and plugin registration.
  • The daemon accepts JSON RPC envelopes, registers in-flight work, dispatches handlers, and exposes lifecycle control.
  • Provider exec/TCP transports reach the daemon; the background manager sends heartbeats and cancellations.
  • The daemon is the runtime control plane inside the sandbox.

Each daemon envelope is validated, registered in the in-flight registry with agent_id, op, and background flag, dispatched through OP_TABLE, then deregistered in finally backend/src/sandbox/daemon/rpc/dispatcher.py:55-113. The registry supports cancellation, heartbeat refresh, per-agent background counts, and TTL reaping for stale background invocations backend/src/sandbox/daemon/rpc/in_flight.py:20-154. Public daemon handlers expose cancel, heartbeat, and inflight_count backend/src/sandbox/daemon/operation_handlers.py:56-85, and the host API wraps them backend/src/sandbox/api/daemon_invocations.py:18-65.

The engine background path creates a background alias, detects whether the tool requires sandbox context, assigns a sandbox invocation id, and launches the coroutine through BackgroundTaskManager backend/src/engine/background/dispatch.py:59-156. The manager stores sandbox id, invocation id, agent id, and uses_sandbox on each tracked task backend/src/engine/background/manager.py:67-155. Running sandbox invocations are heartbeated periodically via sandbox_api.heartbeat() backend/src/engine/background/manager.py:405-454. Cancelling a sandbox-bound background task first wires daemon cancellation through sandbox_api.cancel() backend/src/engine/background/manager.py:281-311.

Isolated vs ephemeral background behavior Ephemeral background shell tasks are normal sandbox-bound daemon invocations with heartbeats and TTL cleanup. Isolated enter is blocked while those are running. Isolated exit cancels running sandbox-bound tasks for the same agent before closing the namespace backend/src/sandbox/host/iws_lifecycle.py:96-133.

6.1 Background Task Lifecycle §

  1. The tool execution layer decides to launch a tool in the background and calls the background dispatch helper.
  2. Dispatch detects sandbox context, creates a sandbox invocation id, and stores it on the tracked background task.
  3. When the sandbox request reaches the daemon, the dispatcher registers the invocation with background=true.
  4. The background manager heartbeat loop periodically batches sandbox invocation ids and calls daemon heartbeat.
  5. If the user cancels the background task, manager cancellation first calls daemon cancellation for the invocation id.
  6. If heartbeats stop, the daemon in-flight TTL reaper cancels stale background invocations.
  7. Result tools read the local manager state; they do not directly inspect daemon state unless cancellation/heartbeat is needed.
