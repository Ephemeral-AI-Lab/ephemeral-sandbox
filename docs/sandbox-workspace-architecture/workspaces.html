5. Workspace Modes §

  • Not every task needs the same workspace lifetime or privacy model.
  • The package defines persistent, per-operation, and explicit private-session workspace behavior.
  • Daemon dispatch chooses isolated if open, otherwise ephemeral; main workspace provides binding and base identity.
  • All modes start from LayerStack truth but choose different temporary writable lifetimes.
main_workspace ephemeral_workspace Fast-path Technical Workflows isolated_workspace Mode Selection Workspace Routing and State

main_workspace

Persistent identity: the bound repo plus LayerStack snapshots, written only through OCC.

ephemeral_workspace

Default per-call mode: each operation gets a fresh overlay or a direct file fast path.

isolated_workspace

Opt-in private mode: enter once, run many tools, exit and discard the upperdir.

5.2 main_workspace §

  • The sandbox needs one persistent repository identity to anchor every projection.
  • main_workspace is a thin package anchor around LayerStack and OCC based workspace truth.
  • Host bootstrap uploads runtime, ensures git, then ensures the workspace base.
  • Main workspace is the source identity; the active LayerStack manifest is the mutable truth.

The main_workspace package intentionally remains a thin facade: its README says implementation remains in sandbox/layer_stack and sandbox/occ so the three workspace concepts can have sibling package anchors backend/src/sandbox/main_workspace/README.md:1-6. The host lifecycle calls ensure_workspace_base after uploading the runtime and ensuring git backend/src/sandbox/host/bootstrap.py:300-312. Runtime binding fails closed if the binding or manifest is missing/empty backend/src/sandbox/main_workspace/workspace_binding.py:12-35.

5.3 ephemeral_workspace §

  • Most tool calls should not keep private workspace state after the operation ends.
  • ephemeral_workspace creates a fresh overlay for shell/search/plugin operations, or direct OCC fast paths for simple file APIs.
  • LayerStack leases lower layers, overlay captures upperdir, and OCC publishes accepted changes.
  • Every call gets a clean snapshot and releases its scratch space when done.

Daemon routing uses an active isolated pipeline only if the caller already has an isolated handle; otherwise it returns get_sandbox_overlay(..., start=False) backend/src/sandbox/daemon/workspace_tool_dispatch.py:30-50. The start=False path is important: it gives the daemon a cached EphemeralPipeline facade without starting a persistent workspace mount backend/src/sandbox/ephemeral_workspace/overlay_registry.py:33-70.

  2. It extracts exactly one tool path and converts it with WorkspaceBinding.workspace_relative_path().
  3. The router builds typed OCC changes directly from the tool arguments.

5.4 isolated_workspace §

  • Low-confidence or experimental tasks can request a private scratch workspace across many tool calls.
  • isolated_workspace opens one persistent overlay namespace on enter, runs tools inside it, and discards upperdir on exit.
  • Enter leases LayerStack paths; runtime wires namespace/network/cgroup; exit releases the lease and removes scratch.
  • Isolated workspace gives privacy and continuity without publishing changes to shared truth.

The tools module registers both lifecycle tools with the normal sandbox tool set backend/src/tools/sandbox/_lib/registry.py:17-31. enter_isolated_workspace calls host lifecycle enter with the current caller, layer stack root, background manager, and sandbox id backend/src/tools/isolated_workspace/enter_isolated_workspace/definition.py:24-47. exit_isolated_workspace calls host lifecycle exit and returns discarded upperdir bytes and phase timings backend/src/tools/isolated_workspace/exit_isolated_workspace/definition.py:24-46.

The daemon registers lifecycle operations api.isolated_workspace.enter and api.isolated_workspace.exit in the OP table backend/src/sandbox/daemon/rpc/dispatcher.py:258-389.

On enter, the isolated lifecycle checks capacity, prepares a LayerStack snapshot, creates upper and work under its scratch dir, keeps layer_paths, then wires a namespace holder, namespace fds, veth, overlay mount, DNS, and cgroup backend/src/sandbox/isolated_workspace/_control_plane/workspace_handle_lifecycle.py:21-155. The runtime spawns unshare --user --map-root-user --net --pid --mount --fork --kill-child --propagation private backend/src/sandbox/isolated_workspace/_control_plane/linux_runtime.py:64-104 and mounts the overlay inside that namespace with the LayerStack paths as lowerdirs backend/src/sandbox/isolated_workspace/_control_plane/linux_runtime.py:105-143.

Subsequent tool calls do not create/destroy overlays. IsolatedPipeline.run_tool_call() wraps the existing handle into an OverlayHandle, runs in the existing namespace, marks workspace=isolated, and captures changed paths for audit only. It explicitly does not commit through OCC; writes are discarded when the scratch dir is removed on exit backend/src/sandbox/isolated_workspace/pipeline.py:121-158. Exit kills the holder, tears down veth, closes fds, releases the snapshot lease, removes the cgroup, and deletes the scratch dir backend/src/sandbox/isolated_workspace/_control_plane/workspace_handle_lifecycle.py:175-258.

5.4.1 Enter, Run, Exit Details §

  1. The host lifecycle checks local background tasks and daemon in-flight background counts for the caller.
  2. The daemon registry lazily constructs or returns the isolated pipeline for the requested layer-stack root.
  3. The pipeline waits for startup orphan recovery, checks feature enablement, verifies agent id, enforces one handle per agent, and checks capacity.
  4. LayerStack prepares a snapshot and returns the lease plus existing layer_paths.
  5. The handle lifecycle creates scratch, upper, and work directories under the overlay writable root.
  6. The Linux runtime spawns the namespace holder, opens namespace fds, installs networking, mounts overlay, configures DNS, signals readiness, and creates cgroup state.
  7. The pipeline persists manager state and emits an audit event with phase timings.
