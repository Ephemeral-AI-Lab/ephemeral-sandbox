8. Plugin in ephemeral_workspace §

  • Plugins need current workspace state without bypassing LayerStack/OCC ownership.
  • The plugin path installs plugin runtime, registers daemon ops, and supplies projection and operation-overlay APIs.
  • EphemeralPipeline provides overlays and publish cycles; LSP builds a long-lived Pyright session on top.
  • Plugins are daemon extensions that consume the same snapshot/overlay/OCC pipeline as tools.

8.1 Host and Daemon Flow §

Host-side call_plugin is the single entrypoint: resolve sandbox/layer stack/caller, install the plugin bundle, call api.plugin.ensure, dispatch plugin.<name>.<op>, and wrap the response backend/src/sandbox/ephemeral_workspace/plugin/host_dispatch.py:1-15. The implementation ensures runtime only when the digest changes, then dispatches the plugin op with caller, workspace root, and intent metadata backend/src/sandbox/ephemeral_workspace/plugin/host_dispatch.py:70-186.

Inside the daemon, plugin_ensure imports plugins.catalog.<plugin>.runtime.server, flushes plugin registrations into the daemon dispatcher, warms the runtime, and records loaded state backend/src/sandbox/ephemeral_workspace/plugin/runtime_api.py:76-162. The context factory gives each plugin a WorkspaceProjection and an EphemeralPipeline overlay facade created with start=False so plugin calls do not create a long-lived daemon mount or lease backend/src/sandbox/ephemeral_workspace/plugin/runtime_api.py:266-350.

  1. Host code resolves sandbox id, layer-stack root, plugin name, operation name, caller identity, and plugin bundle digest.
  2. The host installs or refreshes the bundle in the sandbox runtime area.
  3. The host calls api.plugin.ensure; the daemon imports the plugin runtime and flushes registered operations into OP_TABLE.
  4. The host sends plugin.<name>.<op> through the same daemon transport as built-in tools.
  5. The daemon plugin context supplies projection APIs, an overlay facade, caller metadata, and workspace root information.

8.3 Projection Refresh and LSP Writes §

  1. The daemon runtime caches WorkspaceProjection by layer-stack root using the daemon-owned LayerStack manager.
  2. A read-only LSP op asks session_manager.get_session() for a session keyed by active manifest identity.
  3. If the key changed, the session manager acquires a fresh overlay/projection handle or remounts an existing Pyright namespace.
  4. The ephemeral pipeline emits workspace-change events after publishes, and the LSP session pump forwards them to Pyright.
  5. Pyright receives file-watch notifications and refreshes open-document text so diagnostics/hover/rename see the latest projection.

The workspace-change event bus is generic, not Pyright-specific. The ephemeral pipeline exposes subscribe/unsubscribe methods and emits a bounded event queue; if a subscriber queue overflows, it sends a full-resync marker instead of losing correctness silently backend/src/sandbox/ephemeral_workspace/pipeline.py:284-315, backend/src/sandbox/ephemeral_workspace/events.py:51-101. LSP uses that generic stream to drive Pyright refresh, but another plugin could consume the same projection and event model.

Daemon change for plugins Plugin ops are registered into the same daemon OP_TABLE, but the dispatcher blocks plugin access while the caller has an open isolated workspace backend/src/sandbox/daemon/rpc/dispatcher.py:216-235. That avoids a plugin reading/publishing through the shared ephemeral projection while the agent is intentionally inside private isolated scratch.
