3. Overlay §

  • Shells and plugins need a normal filesystem, not a layer-by-layer API.
  • Overlay mounts layer paths as read-only lowerdirs plus a private writable upperdir/workdir.
  • LayerStack supplies lower layers; OCC consumes captured upperdir changes.
  • Overlay turns a versioned layer log into a temporary working tree for one operation or one isolated session.

The overlay lifecycle primitive allocates a per-operation run dir, leases a snapshot, requires layer_paths, allocates upper and work, and returns an OverlayHandle backend/src/sandbox/overlay/lifecycle.py:20-79. Upper/work directories live under /eos-mount-scratch/eos-sandbox-runtime; there is intentionally no fallback because overlayfs requires a suitable writable filesystem backend/src/sandbox/overlay/writable_dirs.py:13-52.

The kernel mount helper calls Linux fsopen, fsconfig, fsmount, and move_mount. It adds each layer with lowerdir+ in manifest order, where the first path is the highest-priority lower backend/src/sandbox/overlay/kernel_mount.py:45-67. Inputs are validated with O_DIRECTORY | O_NOFOLLOW and passed as /proc/self/fd/... paths to pin the checked directories backend/src/sandbox/overlay/kernel_mount.py:139-198.

Does overlay replace the whole filesystem? No. A fresh ephemeral operation runs in a new mount namespace and overmounts only the bound workspace_root. Other inherited filesystem paths remain available subject to denylist policy. Isolated workspace does the same idea once inside a persistent namespace at /testbed. The code mount target is workspace_root, not / backend/src/sandbox/overlay/kernel_mount.py:45-67.

Fresh ephemeral calls write a JSON payload and launch unshare -Urm python -m sandbox.overlay.namespace_entrypoint backend/src/sandbox/overlay/namespace_runner.py:66-128. The entrypoint validates/mounts the overlay, runs the tool from workspace_root, unmounts, and writes timings backend/src/sandbox/overlay/namespace_entrypoint.py:85-175. For isolated workspaces, the runner passes mount_overlay=false and executes inside the already-mounted namespace backend/src/sandbox/overlay/namespace_runner.py:131-197.

  1. The operation finishes and the parent pipeline keeps the overlay handle alive.
  2. For write-intent calls, capture_changes(handle) walks only upperdir.
  3. Directory traversal is sorted per level and emits opaque-directory changes before children.
  4. Regular files become disk-backed write changes with content_path and hash.
  5. Whiteout entries become deletes; symlinks become symlink changes; opaque xattrs/markers become opaque-dir changes.
  6. The pipeline converts captured path changes to OCC changes and then destroys the overlay handle.
