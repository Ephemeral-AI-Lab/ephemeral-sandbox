4. OCC §

  • Multiple operations can edit overlapping files from different snapshots.
  • OCC routes, validates, stages, serializes, and publishes changes with stale-base rejection.
  • LayerStack provides live manifests; overlay capture converts filesystem changes into typed OCC mutations.
  • OCC is the gate that decides whether temporary workspace changes become durable truth.

4.2 Technical Workflow §

  1. OccService.apply_changeset() prepares changes before entering the commit queue.
  2. If the caller did not provide a snapshot, the service reads the current active manifest.
  3. Prepare normalizes paths, groups ordered changes by path/route, attaches gated base hashes, and returns PreparedChangeset.
  4. The commit queue serializes publish work, optionally batches disjoint non-atomic non-overlay groups, and retries manifest CAS races.
  5. The transaction reads the live manifest, validates every path group, writes accepted content to OCC staging, and builds layer deltas.
  6. If atomic semantics require all-or-nothing and any path fails, accepted paths are marked dropped and no layer is published.
  7. Accepted changes publish as one LayerStack layer and the result reports per-file statuses plus the published manifest version.

4.3 Routes and Statuses §

DROP
Used for paths under .git; the mutation is intentionally not published.
DIRECT
Used for gitignored paths; it skips base-hash checks and behaves like last-writer-wins for those paths.
GATED
Used for normal tracked/non-ignored paths; it compares expected base hashes against the live manifest before publishing.
REJECT
Used when path normalization or route validation fails before a publishable change exists.
