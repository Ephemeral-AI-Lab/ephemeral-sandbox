7. Provider §

  • The sandbox runtime can run on different backends without changing workspace logic.
  • The provider boundary defines provider-neutral CRUD, exec, preview/log, and context-preparation primitives.
  • Host lifecycle bootstraps runtime and workspace base through the selected adapter.
  • Providers own containers; sandbox workspace logic stays provider-agnostic.

7.2 Bootstrap Workflow §

  1. Provider bootstrap chooses and initializes a Docker or Daytona adapter, then the registry binds the adapter globally or per sandbox.
  2. Container create/start returns provider metadata, including the workspace root when the provider can report it.
  3. Host post-lifecycle setup uploads the runtime bundle, ensures git, finishes upload, runs runtime bootstrap, and calls ensure_workspace_base backend/src/sandbox/host/bootstrap.py:300-312.
  4. The daemon client later uses the same adapter for TCP endpoint discovery or AF_UNIX thin-client exec.
  5. Workspace code below the daemon boundary receives layer_stack_root and workspace_root; it does not care which provider created the container.
